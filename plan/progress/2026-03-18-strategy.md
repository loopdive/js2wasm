## Unique value proposition

**ts2wasm is the only compiler that turns TypeScript directly into tiny, sandboxed WebAssembly modules with no runtime, no GC, and no JS engine dependency.**

| | ts2wasm | AssemblyScript | Cheerp | Emscripten |
|---|---|---|---|---|
| Input language | TypeScript | AssemblyScript (TS-like) | C/C++ | C/C++ |
| Output size | 100B – 10KB | 10KB – 100KB | 100KB+ | 100KB+ |
| Runtime bundled | No (WasmGC) | Yes (linear memory GC) | Yes | Yes |
| JS engine needed | No | No | Yes (for JS interop) | Yes |
| Sandbox isolation | Full Wasm sandbox | Full Wasm sandbox | Partial | Partial |
| Existing TS code | Compatible subset | Requires rewrite | N/A | N/A |

### Three differentiators

1. **Security**: Run untrusted TypeScript in-process without V8 isolates. Each module is sandboxed — no access to filesystem, network, or globals unless explicitly imported.

2. **Size**: No runtime, no allocator, no stdlib. A fib function is ~200 bytes. A DOM app is ~1KB. This makes ESM-style micro-modules practical in Wasm.

3. **Portability**: Run TypeScript in any Wasm host — wasmtime, wasmer, wazero, browser, Cloudflare Workers — without a JS engine.

## Target markets

### 1. Edge computing platforms (Cloudflare, Deno, Vercel)
**Problem**: Running user TypeScript at the edge requires V8 isolates (heavy, slow cold start).
**ts2wasm solves**: Compile to tiny Wasm modules, instant cold start, full sandbox.
**Revenue**: Commercial license for embedding ts2wasm in edge runtimes.

### 2. Plugin/extension systems (SaaS, no-code, game modding)
**Problem**: Letting users write custom logic safely. Current options: sandboxed JS (QuickJS), Lua, or WASI.
**ts2wasm solves**: Users write TypeScript (familiar), it compiles to sandboxed Wasm (safe). No new language to learn.
**Revenue**: Commercial license, or hosted compile API.

### 3. Non-JS backends (Go, Rust, C++ services)
**Problem**: Want to embed TypeScript business logic without bundling a JS engine.
**ts2wasm solves**: Compile TS to Wasm, load in wasmtime/wasmer. No V8 dependency.
**Revenue**: Open source adoption → consulting/support contracts.

### 4. IoT / embedded
**Problem**: Run logic on devices with <1MB RAM, no JS engine.
**ts2wasm solves**: Tiny Wasm modules run on Wasm3, WAMR, or similar lightweight runtimes.
**Revenue**: Long-tail, mostly open source adoption.

## Monetization strategy

### Open core model
- **Open source**: compiler, CLI, playground — MIT license
- **Commercial**: embedding license for companies integrating ts2wasm into their products (edge runtimes, SaaS platforms, game engines)
- **Hosted service**: compile API (POST TypeScript, GET Wasm) — usage-based pricing

## Risks

1. **Diminishing returns** — each additional pass requires more work. The curve is flattening.
2. **WasmGC adoption** — still relatively new. Not all hosts support it yet.
3. **TypeScript evolution** — new TS features (decorators, import attributes) need ongoing support.
4. **Single developer** — bus factor of 1 (+ AI agents). Need to build community.
5. **Competitive moat** — if a well-funded team decides to build a similar compiler, they could catch up quickly. The moat is execution speed and early mover advantage.
