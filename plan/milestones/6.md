# Milestone 6: Advanced Optimization

**Goal**: Improve generated code quality through whole-program type analysis and monomorphization. Make compiled Wasm competitive with JIT-compiled JavaScript.

**Target**: Reduce externref boxing/unboxing overhead. Smaller binaries. Faster execution.

**Estimated impact**: Performance improvement rather than pass count. May fix ~500 tests where type mismatches cause wrong results.

## Issues (priority order)

| # | Title | Type | Impact | Complexity |
|---|-------|------|--------|------------|
| **743** | Whole-program type flow analysis | Perf | Eliminates unnecessary externref | XL |
| **760** | Monomorphize functions with call-site types | Perf | Native arithmetic everywhere | Hard |
| **744** | Function monomorphization for polymorphic sites | Perf | Specialized copies per type | Hard |

## Rationale

These are the "make it fast" issues. Once correctness is solid (Milestones 1–5), optimization becomes the differentiator. Whole-program type flow (#743) is the foundation — it tells the compiler what types actually flow through each function. Monomorphization (#760, #744) uses that information to generate specialized native-typed code.

Example impact:
```javascript
function add(a, b) { return a + b; }
add(1, 2);  // Currently: externref boxing → f64.add → externref unboxing
            // After: f64.add (zero overhead)
```

## Success criteria

- Functions called only with known types compile without externref parameters
- Polymorphic functions get specialized copies (up to 4 per function)
- Binary size does not increase more than 20% from monomorphization
- Binaryen wasm-opt can deduplicate identical specializations
