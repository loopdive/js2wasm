# Milestone 4: Dual-Mode Architecture (JS Host Optional)

**Goal**: Add Wasm-native implementations for features that currently require JS host imports, making JS host mode optional rather than mandatory. JS host imports remain available as a fast path when running in a browser/JS environment.

**Target**: All features work in both JS host mode and standalone/WASI mode. Enable standalone WASI execution for real programs.

**Estimated pass rate improvement**: Indirect — enables WASI deployment rather than test count improvement.

## Rationale

The project needs to work in two modes:

1. **JS host mode** (default for browsers): Uses JS host imports for performance and full spec compliance. Existing host imports are kept as-is.
2. **Standalone mode** (WASI, edge platforms): No JS runtime available. Needs Wasm-native implementations for generators, iterators, RegExp, etc.

This follows the same dual-mode pattern established by #679 (dual string backend: wasm:js-string vs native i16 arrays) and #682 (RegExp: host vs Wasm engine).

Generators and iterators are the two biggest remaining features without standalone alternatives. Adding Wasm-native implementations enables:

1. **WASI target works for real programs** — currently generators crash under WASI
2. **Component Model** — clean exports with no JS glue needed
3. **Edge deployment** — Fastly, Fermyon, Shopify Functions

#680 (generators) is XL but the phased approach makes it manageable:
- Phase 1 (simple yields) covers 60% of cases
- Phase 2 (yields in loops) covers 85%
- Phase 3 (yield*, return, throw) covers 95%

## Success criteria

- Generator-using programs run under both JS host mode and WASI
- Iterator protocol has Wasm-native path (struct-based) and JS host path
- JS host mode is not regressed — existing host imports still work
- Mode selection via `--target wasi` (standalone) vs default (JS host)
- Component Model WIT generation produces valid interfaces
- At least one edge platform deployment demonstrated (Fastly or Fermyon)
