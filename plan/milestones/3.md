# Milestone 3: Iterator & Async Correctness

**Goal**: Implement the Symbol.iterator protocol and fix async/generator edge cases. This unblocks a cascade of dependent features.

**Target**: Fix ~800 test failures and unblock 5+ dependent issues.

**Estimated pass rate improvement**: +600–800 tests (44% → 46%)

| # | Title | Type | Impact | Complexity |

| **735** | Async iteration correctness | FAIL | 329 FAIL | Medium |

## Rationale

Symbol.iterator is an architectural keystone. Without it, `for-of` only works on arrays, `Map`/`Set` iteration fails, custom iterables are broken, and spread on non-arrays crashes. Fixing it unblocks:

- Symbol.toPrimitive
- Symbol.species
- Symbol RegExp protocols (Symbol.match, Symbol.replace, Symbol.search)
- Symbol.toStringTag / Symbol.hasInstance
- User-defined Symbol property keys

#762 (generator .next() args) is small but important for coroutine patterns. #761 (rest/spread) fixes silent data loss in destructuring.

## Success criteria

- `for-of` works on Map, Set, custom iterables, and generators
- Spread operator works on any iterable, not just arrays
- Generator `.next(value)` passes values correctly
- Rest elements in destructuring capture remaining items

## Dependencies

- #762 depends on #680 (pure Wasm generators) for full integration, but can be partially implemented on current host-backed model
- #766 is independent
