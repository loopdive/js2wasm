# #700 — Superseded: reuse TypeScript host state across test262 compilations

## Status: superseded

This issue proposed an explicit `reuseHost` option on `compile()`/`compileSource()`
so pool workers could keep a warm `ts.CompilerHost` across many test262
compilations.

That exact API was never implemented. However, the underlying goal is largely
covered by the current incremental compiler worker path:

- [src/index.ts](/Users/thomas/Documents/Arbeit/Startup/Projekte/Mosaic/code/@loopdive/ts2wasm/src/index.ts) exposes `createIncrementalCompiler()`
- [scripts/compiler-worker.mjs](/Users/thomas/Documents/Arbeit/Startup/Projekte/Mosaic/code/@loopdive/ts2wasm/scripts/compiler-worker.mjs) keeps a persistent incremental compiler alive inside each worker
- the worker reuses parsed lib files and old-program state across many
  sequential compilations, then periodically recreates the compiler to bound
  memory growth

So the original optimization intent is no longer best represented as “add a
warm `CompilerHost` option to `compile()`”.

## Why this is not kept open

Keeping this issue open is misleading because:

1. the project already uses an incremental compiler path in the hot test262
   worker loop
2. the remaining performance work is better tracked in the newer pool /
   incremental compiler issues, especially [#699](../ready/699.md)
3. the specific `reuseHost` API sketched here is no longer the preferred design

## Follow-up direction

If there is still measurable compiler-host reuse work left, it should be filed
as a new issue against the current incremental compiler architecture rather than
reviving this older design sketch.

## Resolution

Superseded by the current incremental compiler / compiler-pool implementation.
