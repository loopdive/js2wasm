# #497 — Dynamic import() via host-side module loading (442 tests)

## Status: wont-fix

## Reason

Dynamic `import()` fundamentally requires loading, compiling, and linking separate
modules at runtime. This is inherently a host-side operation that violates the
project's core principle of "never delegate to JS host."

### Investigation findings

All 442 skipped tests are in `language/expressions/dynamic-import` and genuinely
test `import()` semantics. None of them test "other things that would pass
independently" -- `import()` is the centerpiece of every test.

**Breakdown of the 939 non-FIXTURE test files in the dynamic-import directory:**

| Category | Count | Status |
|----------|-------|--------|
| Negative parse (expect SyntaxError) | ~313 | Already handled before shouldSkip -- NOT in the 442 |
| Tagged dynamic-import but body has no `import(` | ~362 | Already pass through -- NOT in the 442 |
| Body has `import(` + references `_FIXTURE` files | ~397 | Would be skipped by FIXTURE filter anyway |
| Body has `import(` + no FIXTURE, syntax-only (0 asserts) | ~22 | Just validate import() is valid syntax |
| Body has `import(` + no FIXTURE, with assertions | ~55 | Test import() return value, error handling, Promise behavior |

**Why mocking does not help:**
- Tests check that `import()` returns a Promise, that the Promise resolves to a
  namespace object with specific exports, that specifier toString is called, etc.
- A mock that returns an empty object would fail the assertions
- A mock that returns a real module would require the host to do module loading

**Why further narrowing does not help:**
- The 22 syntax-only tests could theoretically pass if the compiler supported
  `import()` as an expression. But the compiler does not parse `import()` at all.
- No test uses `import()` incidentally while testing something else.

### Considered alternatives

1. **Mock `import()` in test runner** -- Would violate "never delegate to JS host"
   principle, and tests assert on the resolved module contents anyway.
2. **Compile `import()` as a stub returning rejected Promise** -- Requires async/Promise
   codegen support; most tests assert on successful resolution.
3. **Further narrow the skip filter** -- Already narrowed by #358. The 442 remaining
   genuinely use `import()` and cannot pass without module loading.

## Original approach (not pursued)

`import("./module.js")` would compile as a host import call:

1. Wasm module calls `__dynamic_import(specifier)` host import
2. Host resolves the specifier, loads the source, compiles to Wasm via ts2wasm
3. Host instantiates the new module, linking shared imports
4. Host returns the module's exports as an externref object
5. The calling code awaits the result (import() returns a Promise)

## Implementation Summary

No code changes. Investigation determined that all 442 skipped tests are genuinely
about dynamic import semantics and cannot be un-skipped without implementing full
module loading, which requires JS host delegation. The skip filter is already
optimally narrow (only skips tests whose body actually contains `import(`).
