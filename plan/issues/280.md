# Issue #280: Function expression compile errors -- name binding and hoisting

## Status: review

## Summary
~101 tests fail in the language/expressions/function category with compile errors. Named function expressions need their name bound in the function body scope. Function expressions with complex parameter patterns or multiple return types also cause errors.

## Category
Sprint 4 / Group A

## Complexity: M

## Scope
- Ensure named function expressions bind their name inside the function body
- Handle function expressions with rest parameters
- Support function expressions in computed property positions
- Update function expression compilation in `src/codegen/expressions.ts`

## Acceptance criteria
- Named function expressions can reference themselves by name
- Function expressions in various positions compile
- At least 30 compile errors resolved

## Implementation Notes

### Changes in `src/codegen/expressions.ts`

1. **Assignment-based closure map registration** (line ~1022): When a function expression is assigned via `=` (e.g., `f = function(...)...`), register the closure info in `closureMap` for the left-hand identifier. Previously only `var f = function(...)...` (VariableDeclaration) was handled.

2. **Property assignment closure map registration**: When a function expression is in a property assignment (e.g., `{ fn: function(...)... }`), register the closure info for the property name.

3. **Externref-to-struct cast in closure calls** (in `compileClosureCall`): When a local variable holding a closure ref has type `externref` (because the TS type resolver mapped the function type to externref), add `any.convert_extern` + `ref.cast` instructions before `struct.get` to properly access the closure struct.

4. **Fallback closure resolution by type index** (in call expression handling): When a function identifier isn't in `funcMap` or `closureMap`, check if the corresponding local variable holds a ref to a known closure struct type (via `closureInfoByTypeIdx`). This handles cases where closures are assigned through indirect paths.

### Tests added
- Named function expression recursive call (factorial)
- Named function expression self-reference without captures
- Function expression assigned via `=` then called
- Function expression hoisted var then called multiple times
