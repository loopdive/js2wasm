# Issue 88: Test262 coverage — language/expressions

## Summary

Add `language/expressions/*` subcategories to the test262 runner and fix compiler failures exposed by these tests. Expressions are the most fundamental building block of any program.

## Motivation

Our test262 suite currently only covers `built-ins/Math/*`. Real-world code uses arithmetic, comparison, assignment, typeof, ternary, comma, and other expression forms constantly. We need conformance validation for these core semantics.

## Subcategories to add

High-confidence (we already compile these):
- `addition`, `division`, `exponentiation`, `concatenation`
- `bitwise-and`, `bitwise-or`, `bitwise-xor`, `bitwise-not`
- `left-shift`, `right-shift-operator`, `unsigned-right-shift-operator`
- `equals`, `does-not-equals`, `greater-than`, `greater-than-or-equal`, `less-than`, `less-than-or-equal`
- `logical-and`, `logical-not`, `logical-or`
- `conditional` (ternary)
- `comma`
- `typeof`, `instanceof`, `in`
- `compound-assignment`, `logical-assignment`
- `grouping`
- `call`, `function`

Skip (unsupported features):
- `arrow-function`, `async-arrow-function`, `async-function`, `async-generator`
- `class`, `generators`, `dynamic-import`, `import.meta`
- `delete`, `coalesce` (may need runtime support)

## Approach

1. Add subcategory paths to `TEST_CATEGORIES` in `test262-runner.ts`
2. Run and triage: categorize as pass / compile_error / runtime fail
3. Fix compiler bugs exposed by failing tests (prioritize compile errors)
4. Update `UNSUPPORTED_FEATURES` filter if needed

## Complexity

L — many subcategories, likely exposes multiple compiler bugs

## Dependencies

None
