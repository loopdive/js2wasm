# Issue #261: ClassDeclaration + new expression for anonymous classes

## Status: review

## Summary
~179 tests fail with "Unsupported statement: ClassDeclaration; Unsupported new expression for class: C". These involve class declarations where the class name does not resolve in the new expression context, often because the class is in a nested scope or the name lookup fails.

## Category
Sprint 4 / Group B

## Complexity: S

## Scope
- Fix class name resolution for `new C()` when C is declared in current or parent scope
- Handle class declarations in expression positions that are immediately instantiated
- Update class lookup in `compileNewExpression`

## Acceptance criteria
- `new C()` resolves when C is declared in enclosing scope
- At least 30 compile errors resolved

## Implementation notes
- Added `compileNewClassExpression()` in `expressions.ts` to handle `new (class { ... })()` patterns
- The callee in `new (class {...})()` is a `ParenthesizedExpression` wrapping a `ClassExpression` -- must unwrap parens
- On-the-fly compilation: calls `collectClassDeclaration()` (Phase 1: struct types + function stubs) then `compileClassBodies()` (Phase 2: constructor/method bodies) with a synthetic name `__anon_class_N`
- Added early detection in `compileVariableStatement()` (statements.ts) to compile the `new ClassExpression` initializer first, so the local variable gets the correct `ref $struct` type instead of `externref`
- Without the variable type fix, the local would be typed as `externref` and subsequent `struct.get` operations would fail with a type mismatch
