# #212 -- Bug: Tagged template caching (identity check)

## Status: review

## Summary

6 tagged template tests fail. They test that the template object passed to a tag function is cached per call site -- calling the same tag from the same source location should yield the same template object.

## Failing Tests

- `cache-differing-expressions.js` -- different expressions, same site -> same object
- `cache-differing-raw-strings.js` -- different raw strings -> different objects
- `cache-differing-string-count.js` -- different string count -> different objects
- `cache-identical-source.js` -- same source -> same object
- `cache-same-site-top-level.js` -- same site -> same cached object
- `cache-same-site.js` -- same site in loop -> same cached object

## Root Cause

The tagged template implementation didn't cache template objects. Each call created a new template array, so identity checks (`===`) failed.

## Fix

Created a global cache of template objects keyed by call site. Each tagged template call site gets a unique module global variable (initialized to `ref.null`). On first call at a given site, the template array is created and stored in the global. On subsequent calls from the same site, the cached value is loaded from the global.

### Implementation Details

- Added `templateCacheCounter` field to `CodegenContext` for unique cache IDs
- Exported `nextModuleGlobalIdx` from `index.ts` for use in `expressions.ts`
- In `compileTaggedTemplateExpression`: allocate a `__tt_cache_N` global per call site, emit a `ref.is_null` check and structured `if` block to lazily create and cache the template array

## Complexity: S

## Tests

- 3 equivalence tests added:
  - Same call site returns same object across calls
  - Different sites produce different objects
  - Same site caches even with different expression values
