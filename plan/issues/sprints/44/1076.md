# #1076 — Split merge job so push-to-main always refreshes baseline

## Problem

`.github/workflows/test262-sharded.yml` has a single `merge` job that both
builds the merged test262 report AND enforces the "fail on regressions" gate.
`promote-baseline` depends on `needs.merge.result == 'success'` (line 198).

On push-to-main, if the merged report shows any regression vs. the currently-
committed baseline, the `Fail on regressions` step exits 1 → `merge` job
fails → `promote-baseline` is skipped → baseline stays frozen. Every push
repeats this, so the baseline never refreshes and drift accumulates silently.

Observed fallout 2026-04-11: 12 PRs merged over ~2 hours, actual main at
20,544/43,164 pass, committed baseline still at 21,750. Each PR's CI deltas
were self-consistent but the cumulative delta was wrong by ~4,000 tests.

## Fix sketch

Split `merge` into two jobs:

1. **`merge-report`** — download shard artifacts, merge the JSONL, build the
   merged report, upload as `test262-merged-report` artifact. Always succeeds
   (unless the shard run itself is broken, in which case there's nothing to
   promote).

2. **`regression-gate`** — `needs: merge-report`. Download the merged
   artifact, run `scripts/diff-test262.ts` against
   `benchmarks/results/test262-current.jsonl`, fail on regressions. This job
   gates PRs and surfaces regressions on push, but its failure does NOT
   prevent baseline promotion.

3. **`promote-baseline`** — change `needs: merge` → `needs: merge-report`.
   Drop the `needs.merge.result == 'success'` predicate. Guard remains
   `if: github.event_name == 'push' || github.event_name == 'workflow_dispatch'`.
   Push-to-main now unconditionally promotes the actual merged report to the
   committed baseline.

4. **Keep the sanity check** at line 274 (`PASS < 1000 || TOTAL < 40000`) —
   it's the only thing between us and publishing a corrupt run as baseline.

## Acceptance criteria

- [ ] A simulated push-to-main with known regressions vs. the previous
      baseline: `merge-report` succeeds, `regression-gate` fails (visible in
      the Actions UI), `promote-baseline` runs and commits the new baseline.
- [ ] PR CI unchanged in behavior for clean PRs: merge-report succeeds,
      regression-gate passes.
- [ ] PR CI with regressions: regression-gate fails, PR cannot merge (admin
      override still works via `gh pr merge --admin`).
- [ ] Subsequent PRs inherit the refreshed committed baseline and see
      accurate deltas.
- [ ] Sanity check still rejects corrupt reports (pass < 1000 or total < 40000).

## Risks

- **Push regressions become advisory on main**: the gate no longer blocks
  baseline promotion, so a regression lands and is immediately canonicalized.
  This is the desired behavior — main is then honest about its state — but
  paired visibility (#1079) is essential.
- **Concurrent push runs**: the `concurrency` block (line 33) already
  serializes per-ref, so two push-to-main runs won't race on the baseline
  commit. Verify this holds with the split.
- **Artifact retention**: if `merge-report` uploads and `promote-baseline`
  runs in a separate job, the artifact must survive until promote reads it.
  Default retention is 90 days — safe.

## Notes

- Minimal diff: rename the existing `merge` job to `merge-report`, move the
  `Compare against current main baseline` + `Fail on regressions` steps into
  a new `regression-gate` job that depends on `merge-report`, update
  `promote-baseline` to `needs: merge-report`.
- Keep `regression-gate` running on ALL events (push + PR + workflow_dispatch)
  for visibility; the `allow_regressions` workflow_dispatch input already
  skips it when explicitly requested.
- No runtime cost — two small serial jobs instead of one larger job.
