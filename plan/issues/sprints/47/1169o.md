# #1169o — IR Phase 4 Slice 12: dynamic element access + array literals

## Problem

`isPhase1Expr` in `src/ir/select.ts` rejects two very common patterns:

1. **Dynamic element access** — `arr[i]` where `i` is not a string literal.
   Currently only `obj["string-key"]` passes the element-access guard (line
   ~1085). Numeric index access (`arr[0]`, `arr[i]`) fails, disqualifying
   all array-indexing functions.

2. **Array literals** — `[1, 2, 3]`, `[]`, `[a, b, c]`. `ArrayLiteralExpression`
   has no branch in `isPhase1Expr`, so any function creating an array literal
   falls back to legacy.

## What this unlocks

Array indexing and array creation are fundamental patterns in test262 numeric
and algorithmic tests. This slice, combined with #1169n (missing operators),
covers the vast majority of remaining numeric/algorithmic test262 cases.

## Acceptance criteria

1. `isPhase1Expr` accepts `ElementAccessExpression` when the argument is a
   Phase-1 expression (not restricted to string literals)
2. `isPhase1Expr` accepts `ArrayLiteralExpression` when all elements are
   Phase-1 expressions (spread elements restricted — no unpacking arbitrary
   iterables)
3. IR lowering emits correct `array.get` / `array.set` / `array.new_fixed`
   instructions for the above
4. For typed arrays (`number[]`, `i32[]`), the existing `ArrayTypeAnnotation`
   path is reused / extended
5. Equivalence tests pass; test262 does not regress; net improvement expected

## Implementation notes

- Element access: check operand `IrType` — if `array<T>`, emit `array.get`
  with i32-coerced index; if object-shaped, fall back to legacy (or property
  lookup by dynamic key — deferred)
- Array literal: emit `array.new_fixed` with element count; type inferred from
  element types (all numeric → `array<f64>`, all i32-shaped → `array<mut i32>`,
  mixed → `array<externref>`)
- Spread in array literal (`[...a, b]`) — deferred; reject in selector if
  spread elements present

## Out of scope

- Dynamic property access on non-array objects (`obj[dynamic]` — shape
  inference required)
- Array prototype methods (`.map`, `.filter`, etc.) — see #1169p

## Implementation summary (slice 12a — dynamic element access only)

Implemented in branch `issue-1169o-ir-slice12`:

### Selector (`src/ir/select.ts`)
- `isPhase1Expr` ElementAccessExpression branch widened to accept any
  Phase-1 argument expression (was: only string literal). Both the
  receiver and the argument expression are recursively shape-checked.
- `ArrayLiteralExpression` is intentionally NOT accepted in this slice
  — the call-graph closure would otherwise widen the IR claim set to
  callers of vec-returning helpers like `function builder(): number[]
  { return [1,2,3]; }` (covered by issue-1169e bridge tests #1181),
  and the lowerer can't yet emit `vec.new_fixed`. Deferred to a
  follow-up slice that adds the IR primitive.

### IR nodes (`src/ir/nodes.ts`)
- `IrUnop` extended with `i32.trunc_sat_f64_s` — saturating f64 → i32
  truncation. Used by the index-conversion path in
  `lowerElementAccess`. Saturation handles NaN→0 and out-of-range
  values gracefully (no trap), matching what test262's typical
  `arr[i]` patterns expect.

### Constant fold (`src/ir/passes/constant-fold.ts`)
- `foldUnary` gains a case for `i32.trunc_sat_f64_s` that mirrors Wasm
  semantics: NaN → 0, +∞ → INT32_MAX, -∞ → INT32_MIN, otherwise
  `Math.trunc`.

### Lowering (`src/ir/from-ast.ts`)
- `lowerElementAccess` rewritten as a dispatcher:
  1. Lower the receiver, inspect its IrType.
  2. If string-literal arg + object receiver → existing slice-2 named
     field path (unchanged).
  3. If receiver is a vec-shaped ref (`(ref|ref_null) $vec_*`) →
     lower the index, sat-truncate to i32 if it's f64, emit
     `vec.get` with the resolved element ValType.
  4. Otherwise → throw clean fallback.
- `lowerExpr` ArrayLiteralExpression branch added with a clean-throw
  fallback message — defensive guard in case a future selector change
  accepts the shape but the IR still hasn't grown a primitive.

### Tests (`tests/issue-1169o.test.ts`)
- 9 tests, all passing locally.
- 6 dynamic-element-access cases covering: numeric literal index,
  identifier index, computed (`i + 1`), bitwise composition (`i & 3`),
  string-literal element access (slice-2 path unchanged), and
  multiple accesses in one expression.
- 1 selector-shape test confirming a self-recursive vec-indexing
  kernel passes the call-graph closure.
- 1 negative test for ArrayLiteralExpression confirming compilation
  succeeds via legacy fallback.
