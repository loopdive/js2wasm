# #1169p — IR Phase 4 Slice 13: String + Array prototype methods through IR

## Problem

Method calls on string and array values fall back to legacy even when the
receiver and arguments are Phase-1 claimable. The selector currently accepts
`<recv>.<method>(args)` only when `recv` resolves to a **local class or extern
class** instance (slice 4 / #1169d + slice 10 / #1169i). Built-in string/array
methods are not modelled as extern classes, so `str.slice(1)`, `arr.push(x)`,
`arr.length` (on array — `.length` on string IS already handled) fail the
selector.

High-value methods (by test262 occurrence):
- **String**: `.length` (done), `.slice()`, `.indexOf()`, `.includes()`,
  `.charAt()`, `.charCodeAt()`, `.split()`, `.trim()`, `.toUpperCase()`,
  `.toLowerCase()`, `.startsWith()`, `.endsWith()`, `.padStart()`, `.replace()`
- **Array**: `.push()`, `.pop()`, `.length`, `.join()`, `.indexOf()`,
  `.includes()`, `.slice()`, `.map()`, `.filter()`, `.reduce()`, `.forEach()`,
  `.find()`, `.findIndex()`, `.every()`, `.some()`

## What this unlocks

Prototype method dispatch on strings and arrays is the last major category of
expressions blocking IR claim on real-world JS. Combined with slices 11–12,
this slice should push `planIrCompilation`'s claim rate to 80%+.

## Acceptance criteria

1. `IrExternClassMeta` (or equivalent) entries registered for `String` and
   `Array` built-ins with their method signatures
2. `isPhase1Expr` accepts `.slice()`, `.indexOf()`, `.includes()`, `.push()`,
   `.pop()`, `.join()` and the other high-value methods listed above
3. IR lowering calls the appropriate host import or emits Wasm-native
   implementation for each method (reuse `src/codegen/` implementations where
   possible, forwarding to the IR runtime interface)
4. `nativeStrings` mode: string methods lower to WasmGC i16-array intrinsics
   where available; JS-host mode: lower to string host imports
5. Equivalence tests pass; test262 does not regress

## Implementation notes

- Model `String` and `Array` as pseudo extern-classes in the IR resolver.
  `getExternClassInfo("String")` and `getExternClassInfo("Array")` return
  method signatures. The selector already accepts `<extern-class>.<method>()`.
- For each method: either (a) emit a direct Wasm-native implementation or (b)
  call the existing host import that the legacy path uses. Option (b) is easier
  but doesn't work in standalone/WASI mode — prefer (a) for the high-value set.
- `arr.map(fn)` / `arr.filter(fn)` require callback handling — defer these if
  the closure argument interacts poorly with the IR's closure model.

## Implementation summary (slice 13a — `arr.length` proof of concept)

Implemented in branch `issue-1169p-string-array-methods`:

### Lowering (`src/ir/from-ast.ts`)
- `lowerPropertyAccess` extended with a vec-receiver branch: when the
  receiver's IrType resolves through `cx.resolver.resolveVec(recvVal)`
  (i.e. it's a `(ref|ref_null) $vec_*` shape), `.length` lowers via
  `cx.builder.emitVecLen(recv)` — a primitive that already exists
  (slice 6 part 2 / #1169e). Other property names on a vec receiver
  throw clean fallback so the function reverts to legacy.

### Selector (`src/ir/select.ts`)
- No change required — `isPhase1Expr` already accepts
  `PropertyAccessExpression` recursively, and `resolveParamType`
  accepts `ArrayTypeNode` (`T[]`) since slice 6 part 2.

### Tests (`tests/issue-1169p.test.ts`)
- 9 tests, all passing locally.
- 4 `arr.length` cases (single param, with arithmetic, in comparison,
  two vec params) using the #1181 bridge pattern: a legacy `builder()`
  constructs the vec, the IR-claimed function consumes it. Confirms
  the IR selector claims the consumer and the IR-emitted body returns
  the same value as the legacy path.
- 1 negative case: `arr[0]` (element access) still works via legacy
  fallback (#1169o handles dynamic element access in IR; this just
  confirms no regression).

### Out of scope for this PR (clean-fallback)

- Array prototype methods: `.push()`, `.pop()`, `.indexOf()`,
  `.includes()`, `.slice()`, `.join()`, `.find()`, `.findIndex()`,
  `.every()`, `.some()`, `.forEach()`, `.map()`, `.filter()`,
  `.reduce()`. The selector currently rejects these because
  `lowerMethodCall` only handles `extern` and `class` receivers — vec
  receivers throw "method call .X on val not in slice 4". Pattern for
  follow-up: add a vec-receiver branch in `lowerMethodCall` that
  resolves the vec layout via `cx.resolver.resolveVec`, looks up the
  method against a new pseudo-extern-class registry for `Array`, and
  emits `vec.<method>` IR primitives (or call to the legacy host
  helpers via `cx.builder.emitFuncCall`).
- All String prototype methods. Same pattern: `lowerMethodCall` would
  need a `IrType.string` branch dispatching to a pseudo-extern-class
  registry for `String`.

### Recommended next steps for full slice-13 completion

1. Add `String` and `Array` as pseudo-`ExternClassInfo` entries in
   `ctx.externClasses` (legacy already has these methods registered
   under `String_*` / `Array_*` host imports — extract their
   signatures into the registry).
2. Make `lowerMethodCall` dispatch to the extern path when
   `recvType.kind === "string"` (auto-widening to a synthetic
   `extern { className: "String" }` view).
3. Same for vec receivers (`asVal(recvType)` resolving to a vec via
   `resolveVec`) — auto-widen to `extern { className: "Array" }`.
4. For each method: either emit a host import call (existing
   `emitExternCall` works) or add a Wasm-native lowering for the
   standalone mode set listed in #1105.
5. Defer callback-taking methods (`.map`, `.filter`, `.reduce`,
   `.forEach`, `.find`, `.findIndex`, `.every`, `.some`) — these
   interact with the IR's closure model in ways slice 13 doesn't yet
   model. Track in #1169q (follow-up).
