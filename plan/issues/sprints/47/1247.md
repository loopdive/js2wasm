# #1247 — Typed `string[]` local with `String.prototype.split` initializer triggers struct-type mismatch

## Problem

The natural typed form of Hono's `splitPath` utility:

```ts
export function splitPath(path: string): string[] {
  const paths: string[] = path.split("/");
  if (paths[0] === "") {
    paths.shift();
  }
  return paths;
}
```

compiles successfully but **fails Wasm validation at instantiation**:

```
WebAssembly.instantiate(): Compiling function #6:"splitPath" failed:
struct.get[0] expected type (ref null 6), found local.get of type
(ref null 1) @+2547
```

Two struct types are involved:
- `(ref null 1)` — the type the codegen allocates for the local `paths`
- `(ref null 6)` — the type `String.prototype.split` returns

The two don't match, so `struct.get[0]` (reading the `length` field) fails validation.

Workaround: type both the local and the function return as `any`. That bypasses the
typed-array struct and uses a uniform externref path.

## Root cause sketch

`String.prototype.split` returns a different struct-type from what the codegen allocates
for a typed `string[]` local. Either the local allocation needs to use the same struct
type as `__string_split` returns, or a coercion needs to be inserted at the assignment.

## Acceptance criteria

1. Minimal repro (`splitPath` with typed signature) instantiates and runs correctly.
2. `tests/issue-1247.test.ts` exercises the typed-`string[]` form of `path.split("/")`.
3. No regression in `tests/string-methods.test.ts` or `tests/equivalence/`.
4. `tests/stress/hono-tier1.test.ts` Tier 1a can use the natural typed signature.
