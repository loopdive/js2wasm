# #1079 — Baseline age stamp + SHA visibility

## Problem

The 2026-04-11 baseline drift incident was invisible until a human manually
ran test262 against main and compared to the committed numbers. There was no
signal anywhere — landing page, CI dashboard, or PR view — that said "the
baseline you are comparing against is N hours old and was generated by
commit X". Drift could have continued indefinitely.

Visibility is the only thing that catches a self-inflicted regression between
two structural-fix runs. If #1076 lands but some new code path reintroduces
a gate that blocks baseline promotion, #1079 is what exposes the bug within
hours instead of days.

## Fix sketch

Three cooperating pieces:

1. **Stamp the committed baseline** with `baseline_generated_at` (ISO-8601
   timestamp) and `baseline_sha` (commit SHA of the run that produced it) at
   promotion time. Today `scripts/build-test262-report.mjs` emits a JSON
   report; add two fields to its summary section, populated from
   `github.sha` and `new Date().toISOString()` in the workflow.

2. **Surface age on the landing page**. `benchmarks/results/report.html`
   reads the current report to draw the trend graph. Add a small badge near
   the top-line pass count: "Baseline age: 2h 14m (commit abc1234)". When
   age > 6 hours, color it orange; > 24 hours, color it red.

3. **PR comment annotation (optional, lowest priority)**: if the PR CI diff
   finds that the committed baseline age is > 6 hours, append a line to the
   regression report: "⚠️ baseline is 8h old, consider requesting a refresh
   via #1078 before trusting these numbers".

## Acceptance criteria

- [ ] `benchmarks/results/test262-current.json` contains `baseline_generated_at`
      and `baseline_sha` fields populated at workflow promotion time.
- [ ] Landing page (`benchmarks/results/report.html`) shows the age and SHA
      near the pass-count header. Age updates on every page reload.
- [ ] Age > 6 hours = visible orange warning; > 24 hours = red alarm.
- [ ] `runs/index.json` trend data is unchanged (backward-compatible).

## Dependencies

- None hard. Benefits from #1076 but is independently valuable.

## Risks

- **Backward compatibility**: existing entries in `runs/index.json` and
  consumers of `test262-current.json` must tolerate the new fields. Add as
  optional fields; readers that don't know about them should not break.
- **Clock skew**: trust the GitHub Actions runner's clock. Minor skew is
  acceptable for an age badge measured in hours.

## Notes

- The entire fix is ~30 lines of yaml + 20 lines of html/js + 5 lines in
  `scripts/build-test262-report.mjs`. Should ship in a single PR.
- This is the operational-visibility rescue: when the next
  structural-gate bug happens (because it will), this is how we notice.
