# #862 -- Empty error message failures: iterator/destructuring step-err tests (212 FAIL)

## Problem

212 tests fail with an empty error message. The test executes and returns a non-zero exit code, but no error text is captured. Most of these are destructuring tests involving iterator step errors (elision-step-err, rest-id-iter-step-err, rest-id-elision-next-err patterns) and some BigInt non-primitive tests.

## Sample files

- `test/language/expressions/arrow-function/dstr/ary-ptrn-elision-step-err.js`
- `test/language/expressions/arrow-function/dstr/ary-ptrn-rest-id-iter-step-err.js`
- `test/language/expressions/arrow-function/dstr/dflt-ary-ptrn-elision-step-err.js`
- `test/language/expressions/async-generator/dstr/dflt-ary-ptrn-rest-id-elision-next-err.js`
- `test/language/expressions/bitwise-not/bigint-non-primitive.js`

## ECMAScript spec reference

- [§7.4.2 IteratorNext](https://tc39.es/ecma262/#sec-iteratornext) — if .next() throws, the error propagates immediately
- [§7.4.7 IteratorClose](https://tc39.es/ecma262/#sec-iteratorclose) — step 6: if .return() throws, that error replaces the original completion
- [§13.15.5.3 Runtime Semantics: DestructuringAssignmentEvaluation](https://tc39.es/ecma262/#sec-runtime-semantics-destructuringassignmentevaluation) — errors during iteration steps must propagate with correct error type and message


## Root cause hypothesis

These tests define iterators that throw on `.next()` or during elision steps. The compiled Wasm likely traps (unreachable or null deref) without producing a catchable JS error, so the error message is empty. This is related to #851 (iterator close protocol) and #852 (destructuring null_deref/illegal_cast).

## Acceptance criteria

- All 212 tests either pass or fail with a meaningful error message
- If the root cause is iterator protocol, this may be fixed as a side effect of #851/#852

## Sprint-31 Regression Warning

**APPROACH THAT FAILED**: Wrapping generator body in try/catch_all to defer exceptions to .next(). This caught SyntaxErrors that should propagate as compile errors, causing 880 test regressions.

**DO NOT**: Wrap generator bodies in try/catch_all. SyntaxErrors are compile-time and must not be caught.

**BETTER APPROACH**: Only catch runtime exceptions (not compile-time). Consider checking if the generator body compiled successfully before adding error deferral, or use a more targeted exception type filter.

**MUST**: Run full test262 on the integrated branch before merging. The equiv tests did NOT catch this regression.
