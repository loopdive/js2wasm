# #1127 — Nested rest patterns fail to decode — `[...[x,y,z]]`, `[...{length}]`, `[...[,]]`, `[...[...x]]`, `[...{0:v,...}]`

## Problem

Array destructuring patterns where the rest element itself contains a nested binding pattern (array, object, indexed object, further rest, or elision) fail to decode the captured slice. Symptoms:

- `[...[x, y, z]] = [3, 4, 5]` → `x`, `y`, `z` are all `NaN` (should be 3, 4, 5)
- `[...{ length }] = [1, 2, 3]` → `length` is wrong (should be 3)
- `[...[,]]` → CompileError in emitted Wasm
- `[...[...x]]` → `x` is wrong length / contents
- `[...{ 0: v, 1: w, ..., length: z }]` → all slots wrong

**Critical detail**: these fail even when the caller passes explicit arguments. This is NOT the #1135 default-value path — the bug is in how the destructure decodes a rest element whose binding is a nested pattern (not a bare identifier).

## Surfaced by

PR #222 (`fix(dstr): externref + default for nested fn decls — #1135 follow-up`) fixed 1 of the 6 residual `dflt-ary-ptrn-rest-*` test262 failures. The remaining 6 all use nested rest patterns. Probes confirmed they fail independently of the default-value path (call with explicit args also fails).

## Evidence / probes

From `.tmp/probe-nested.mjs` and `.tmp/probe-nested2.mjs`:

```js
// Explicit args — default path NOT taken. All still fail:
function f([...[x, y, z]]) { return x + y + z; }
f([3, 4, 5])    // → NaN (should be 12)

function f([...[x, y, z]]) { return typeof x; }
f([3, 4, 5])    // → 'number' (typeof NaN === 'number')  but value is NaN

function f([...{ length }]) { return length; }
f([1, 2, 3])    // → 0 (should be 3)
```

Baseline (flat rest) works correctly:

```js
function f([...r]) { return r.length; }
f([3, 4, 5])    // → 3  ✓
function f([...r]) { return r[0]; }
f([3, 4, 5])    // → 3  ✓
```

So the rest element itself is being captured — the bug is specifically in recursing the captured vec/array through a nested binding pattern.

## Spec

[ECMA-262 13.3.3.6 Runtime Semantics: IteratorBindingInitialization](https://tc39.es/ecma262/#sec-runtime-semantics-iteratorbindinginitialization)

```
BindingRestElement : ... BindingPattern
  1. Let A be ! ArrayCreate(0).
  2. Let n be 0.
  3. Repeat,
     a. If iteratorRecord.[[Done]] is false, then
        i-iii. Call IteratorStep, handle abrupt completion, set iteratorRecord.[[Done]] if needed.
        iv. Else, append next value to A, increment n.
     b. Else, return the result of performing
        BindingInitialization for BindingPattern passing A and environment.
```

The spec requires: (1) capture remaining iterator values into a fresh Array, then (2) recursively apply `BindingInitialization` on that array using the **nested pattern**. Our codegen appears to handle (1) correctly (the flat `[...r]` case works) but mishandles (2) — likely passing a tuple/non-vec to the nested decoder, or applying the wrong base offset.

## Suspected root cause

`src/codegen/destructuring-params.ts` has paths for `BindingRestElement` where the rest binding is a nested pattern. Two hypotheses:

1. The emitted rest value is a vec/array, but the nested decoder expects a tuple struct (or vice versa) — the same vec-vs-tuple representation mismatch that bit #1135 and PR #222, just in a different location.
2. The nested decoder uses a base offset of 0 on the **original** array, not on the captured rest slice — so `[...[x, y, z]]` on `[3, 4, 5]` reads from the outer source at index 0 (wrong) instead of the rest array at index 0 (right).

Architect should confirm by reading `destructureParamArray` rest-with-nested-pattern path and `destructureParamObject` rest path.
