# #1078 — Emergency baseline refresh workflow_dispatch

## Problem

When the committed baseline drifts from reality (the situation that triggered
#1080), we need a one-click "rebuild the baseline to match current main RIGHT
NOW" operation. Today's `.github/workflows/test262-sharded.yml` has an
`allow_regressions` workflow_dispatch input, but:

1. It's a boolean on the regular sharded workflow, not a dedicated emergency
   path. Easy to miss, easy to misuse.
2. The "promote-baseline" job still depends on `needs.merge.result == 'success'`,
   so `allow_regressions=true` on its own doesn't guarantee promotion if the
   step-level `if:` gating isn't exactly right.
3. There's no audit record of "human explicitly asked to overwrite baseline"
   distinct from a normal push-to-main run.

With #1076 landed, promote-baseline becomes self-correcting and this issue
is less urgent, but we still want an explicit recovery lever for disaster
scenarios (corrupt shard, partial run, infrastructure breakage).

## Fix sketch

Two small changes:

1. **Clarify `allow_regressions` semantics** in the existing
   `test262-sharded.yml` workflow_dispatch input — rename to
   `force_baseline_refresh` (or add as a second input), with a description
   that makes the intent obvious in the Actions UI: "Promote the merged
   report to the committed baseline regardless of regressions. Requires
   explicit confirmation."

2. **Add a confirmation input** for the emergency path:
   `confirm_force: "YES"` — must be typed exactly. Prevents accidental
   one-click runs that silently rewrite baseline.

3. **Add a dedicated audit step** that runs when `force_baseline_refresh`
   is used, commits a human-readable note alongside the baseline commit:
   `chore(test262): FORCED baseline refresh by <actor> — pass=X/Y`.

## Acceptance criteria

- [ ] Running `test262-sharded.yml` via workflow_dispatch with
      `force_baseline_refresh=true` + `confirm_force=YES` always promotes the
      merged report to the committed baseline, regardless of regression gate
      outcome.
- [ ] Running with only `force_baseline_refresh=true` (no confirmation) fails
      loudly with a message explaining the confirmation requirement.
- [ ] The audit commit message includes the actor name and the pass/total
      numbers, distinct from the normal `refresh sharded baseline` message.
- [ ] Normal PR and push runs are unaffected.

## Dependencies

- Builds on #1076 — once the normal path is self-correcting, #1078 is a
  disaster-recovery lever, not a weekly operation.

## Risks

- **Misuse hides regressions**: the confirmation input and audit commit are
  the mitigation. Anyone using the emergency path leaves a trace.
- **Sanity check bypass**: the `PASS < 1000 || TOTAL < 40000` check at
  line 274 must still run — corrupt reports should never be promoted even
  with the force flag.
