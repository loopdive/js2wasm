# #1327 — Feature support section: live test counts, report page, playground deep-links

## What the user sees today

The landing page shows 80 language features with `full` / `partial` / `none` badges and a
code snippet. There is no connection to the 48k-entry test262 JSONL — no pass counts, no
way to drill down to individual test failures, no way to open a failing test in the playground.

## Goal

Three connected improvements:

### A — Pass/total count badge per feature (landing page)

```
✅ Closures           743 / 791
⚡ Async / Await      12 / 184
```

Next to each feature name: `pass / total` from the live JSONL, updated at every CI run.

### B — Feature report page

A new page `/benchmarks/feature-report.html` (or new route) showing the full test breakdown
per feature:

- Feature header with badge + description
- Collapsible test list: one row per test
- Each row: status icon (✅ pass / ❌ fail / ⚠️ compile_error / ⏭ skip), short filename,
  truncated error message, and the specific code line that fails (from `error` field in JSONL)
- Failing rows have "▶ Open in playground" link (Slice D)

URL format: `/benchmarks/feature-report.html?feature=built-ins%2FArray`
The landing page "View test results →" link in each expanded feature section routes here.

### C — Playground deep-link (load test file by URL)

Add `?t262=test/built-ins/Array/from/iter-cstm-ctor.js` query param support to
`playground/index.html`:

1. On load, if `?t262=<path>` is present, fetch the file from the test262 submodule
   (served as static asset from `/test262/<path>` or via a pre-built JSON bundle).
2. Load the file content into the left Monaco editor.
3. Trigger compilation + run immediately.
4. Overlay the error message from the JSONL (passed as `?error=<encoded>`) as an inline
   Monaco decoration at the cited line number.

This closes the loop: from the report page, one click lands in the playground with the
exact failing code and error highlighted.

## Data pipeline

### 1. Enrich `feature-examples.json` source

The source for `feature-examples.json` is generated by `dashboard/build-data.js`.
Each feature object currently has no `testCategories` field.

Add a `testCategories: string[]` field to each feature definition in the source
(the list of `TEST_CATEGORIES` prefixes that belong to this feature). Example:
```json
{
  "name": "Closures",
  "testCategories": ["language/statements/function", "built-ins/Function"],
  ...
}
```

### 2. Compute per-feature stats in `dashboard/build-data.js`

Add a `buildFeatureStats(jsonlPath, features)` function that:
1. Streams `benchmarks/results/test262-current.jsonl`
2. Buckets each entry into the first matching feature by `category` prefix
3. Per feature: computes `passCount`, `totalCount`
4. Stores the full test list (capped at first 500 per feature for JSON size):
   `tests: [{file, status, error, error_category}]`
5. Writes augmented features back into `public/feature-examples.json`

Run this as part of `npm run build:pages` / CI after the JSONL is refreshed.

### 3. Landing page display

In the feature accordion (`dashboard/index.html` or the site framework that renders it):
- Add `<span class="feature-test-count">743 / 791</span>` next to the feature name
- Style: green for ≥90% pass, yellow for 50–90%, red for <50%
- In the expanded body, below the code example and description, add:
  `<a href="/benchmarks/feature-report.html?feature=...">View all test results →</a>`
  (hidden if `totalCount === 0` — feature not covered by test262)

### 4. Feature report page

New file: `public/benchmarks/feature-report.html`

Pure static HTML+JS, reads feature name from `?feature=` query param, fetches
`/feature-examples.json`, finds the matching feature, renders:

```
[feature name] — 743 pass / 791 total (93%)
[description]

Tests:
✅ test/built-ins/Array/from/iter-cstm-ctor.js
❌ test/built-ins/Array/from/iter-set-length.js
   TypeError: Cannot read property 'length'   [▶ playground]
   at L14: assert.sameValue(result.length, 3);
⚠️ test/built-ins/Array/from/...
   [object WebAssembly.Exception]   [▶ playground]
```

Sort order: failures first, then compile_error, then pass.
Filter controls: show all / only failures / only passes.

### 5. Playground URL loading (`?t262=`)

In `playground/index.html`, in the JS init block:

```js
const params = new URLSearchParams(location.search);
const t262File = params.get('t262');
if (t262File) {
  const src = await fetch(`/test262/${t262File}`).then(r => r.text());
  editor.setValue(src);
  compile(); // trigger compilation
  const err = params.get('error');
  if (err) showInlineError(decodeURIComponent(err));
}
```

The test262 files are served as static assets from the `test262/` submodule. If not
already served, add a build step that copies the test262 directory (or just the
`test/` subset) into `dist/test262/`.

## Acceptance criteria

1. Landing page feature list shows `X / Y` count next to each feature name, styled by ratio
2. Expanding a feature with `totalCount > 0` shows a "View test results →" link
3. The feature report page loads at `/benchmarks/feature-report.html?feature=built-ins%2FArray`,
   shows all tests for that feature, failures first, with truncated error messages
4. Clicking "▶ playground" on a failing test opens playground with the test code loaded and
   the error message visible
5. `npm run build:pages` regenerates `feature-examples.json` with test stats from latest JSONL

## Files

- `dashboard/build-data.js` — add `buildFeatureStats()` + `testCategories` field population
- `public/feature-examples.json` — add `testCategories`, `passCount`, `totalCount`, `tests[]`
- `dashboard/index.html` — add count badges + report links

## Notes

- The `tests[]` array in `feature-examples.json` should be capped (e.g. 500 entries) to
  keep the JSON file reasonable. The report page should note "showing first 500 of N tests."
- The `error` field in JSONL is already truncated at ~200 chars. Pass it through as-is.
- If `?t262=` fetch fails (file not in static build), show an editor placeholder with a
  "test file not available" message and the raw source from the JSONL `file` field.
- The playground `?t262=` feature is low-risk: it's purely additive and gracefully degrades
  (no param = current behavior unchanged).
