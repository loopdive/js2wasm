# #1127 — export default of HOF variable not exported — add.js blocked

## Problem

`lodash-es/add.js` uses a higher-order function (HOF) pattern:

```js
var add = createMathOperation(function(augend, addend) {
  return augend + addend;
}, 0);

export default add;
```

The `export default add` is an `ExportAssignment` node where `add` resolves
to a module-level variable (not a function declaration). The #1074 fix only
looks up identifiers in `ctx.funcMap` — variables assigned from HOF calls
are not in `funcMap`, so the export is silently dropped.

Result: `compileProject('lodash-es/add.js')` produces a 19KB binary with
**zero function exports**.

## Impact

Blocks `add(3, 4) === 7` in lodash-es Tier 1 E2E tests.

## Acceptance Criteria

- `compileProject('lodash-es/add.js')` emits at least one function export
- `exports.default(3, 4) === 7` round-trips correctly
- Existing function-declaration default exports (#1074) unaffected

## Possible Approaches

1. **Global-variable export**: If `add` is a module global holding a funcref,
   emit a Wasm exported function that calls the funcref indirectly via
   `call_ref`. Requires the variable's type to be recognized as callable.

2. **HOF pattern recognition**: Detect `var x = createFn(...)` where the
   result is a closure, and register the resulting function in `funcMap`
   so the existing ExportAssignment handler picks it up.

3. **Export wrapper**: For any `export default <ident>` where the identifier
   is a module global (not in funcMap), emit a thin wrapper function that
   reads the global and does `call_ref`.
