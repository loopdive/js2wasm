# #1147 — Fix _start export regressions

## Problem

PR #177 ("replace `__init_done` guards with `_start` export", commit `cb5d43fa`) introduced 837 test262 regressions when it changed module initialization from a lazy guard pattern to a `_start` export. The change was net +496 pass overall, but the 837 regressions were never fixed.

Current state (2026-04-20): 21,324 / 43,172 pass (49.4%) vs April 13 peak of 22,450 (52.0%).
PR #237 recovered +198 of the regressions. ~639 remain.

## Root cause hypothesis

The `_start` export function runs module-level initialization. The test262 harness (`tests/test262-runner.ts`) may not be calling `_start` before invoking test exports, causing module-level assignments (prototype setups, const initializations) to be skipped at test runtime → `assertion_fail`.

## Regression profile (from PR #177 CI)

- Total regressions: 837
- `assertion_fail`: 227 (in PR #177 run; current remaining ~639 after PR #237)
- Pattern: tests that rely on module-level initialization (prototype assignments, globals, class setups)

## Acceptance criteria

- All 639 remaining `_start`-related regressions fixed
- `npm test -- tests/equivalence.test.ts` passes
- Net test262 improvement ≥ +500 (restoring toward 22,450 baseline)
- No new regressions in other categories

## Key files

- `src/codegen/index.ts` — `_start` / `__init_done` module init emission
- `src/codegen/declarations.ts` — module-level statement handling
- `tests/test262-runner.ts` — test harness execution (check if `_start` is called)
- `src/runtime.ts` — runtime host setup
