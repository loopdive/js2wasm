# #1147 — Function.length regression on library methods

## Original premise (wrong)

Tracked as "639 `_start` export regressions from PR #177". Investigation showed that
PR #177's `_start` change did not cause these regressions — test262 tests always have
an exported `test` function, so they take the unchanged `__init_done` guard path and
never exercise the new `_start` path (`src/codegen/declarations.ts:2770-2862`).
`tests/test262-runner.ts:2537` calls `instance.exports.test` directly; `_start` is
never invoked, which has been true since before PR #177.

Bisect of `Array.prototype.toSorted.length` codegen:
- `ad819fe3` (April 13 peak): correct (`__extern_get` path) → returns 1
- `d6d9a550` (before PR #192 toSorted impl): correct → returns 1
- `bd8060e4` (pre-PR #177): **already broken** — emits `f64.const 0; return`
- `cb5d43fa` (PR #177 merge): same broken output

The bug predates PR #177. Its numbers overlapped with PR #177's regression bucket.

## Actual root cause

`compilePropertyAccess` in `src/codegen/property-access.ts` has a fast path for
`Function.length` (lines 1177-1196) that computes the value statically from the
TypeScript signature:

```ts
const paramCount = sig.parameters.filter((p) => {
  const decl = p.valueDeclaration;
  if (!decl || !ts.isParameter(decl)) return true;
  return !decl.dotDotDotToken && !decl.questionToken && !decl.initializer;
}).length;
```

This is wrong for library methods declared in `.d.ts` files. TypeScript's lib types
use `?` liberally for any parameter the runtime treats as optional, but the ES spec
pins `Function.length` for many methods to a fixed integer that disagrees with
TypeScript's required-param count. Example:

- `Array.prototype.toSorted(compareFn?)` → TS-derived length = 0
- ES spec: "The length property of this method is 1." (§23.1.3.34)
- test262 check: `verifyProperty(Array.prototype.toSorted, "length", {value: 1})`
- Before fix: emits `f64.const 0` → assertion fails

The pattern appears across test262: slice, splice, unshift, join, toSorted,
toReversed, with, toSpliced, setFullYear, setUTCHours, toPrecision, fromCodePoint,
Atomics.notify, etc. A quick scan shows ~70 `*/length.js` tests failing with
`assertion_fail` on this alone, plus many other tests that inline
`verifyProperty(Builtin.method, "length", ...)`.

## Fix

In the Function.length fast path, check whether any call signature's declaration
lives in a `.d.ts` file. If yes, fall through to the generic externref
`__extern_get` path below, which reads the runtime `.length` from the JS host —
the authoritative value:

```ts
const isLibrarySig = lengthSigs.some((s) => {
  const decl = s.getDeclaration?.();
  return decl?.getSourceFile().isDeclarationFile === true;
});
if (!isLibrarySig) {
  // ...existing static computation...
  fctx.body.push({ op: "f64.const", value: paramCount });
  return { kind: "f64" };
}
// library sig → fall through to externref path
```

User-defined functions keep the static path because their runtime representation is
a Wasm funcref wrapped as externref — `__extern_get(f, "length")` on that would
return undefined rather than the param count.

## Verification

- Targeted repro (`.tmp/verify-1147.mts`): 12/12 representative property chains
  return the correct runtime `.length` value (toSorted=1, with=2, abs=1, pow=2,
  etc.). User-function `double.length` stays at 1 via the static path.
- Sampled 14 failing test262 `length.js` files: 12 now pass. The 2 remaining
  failures are separate (AggregateError not in BUILTIN_CTOR_NAMES; Iterator
  unavailable in CI Node).
- `tests/equivalence/` full suite: **105 fail / 1183 pass on both baseline and
  with fix** — zero equivalence regressions introduced.
- `Math.*.length` continues to use the earlier `mathMethodArity` table (no
  change in codegen for Math methods).
