# #1147 — Function.length regression on library methods

## Original premise (wrong)

Tracked as "639 `_start` export regressions from PR #177". Investigation showed that
PR #177's `_start` change did not cause these regressions — test262 tests always have
an exported `test` function, so they take the unchanged `__init_done` guard path and
never exercise the new `_start` path (`src/codegen/declarations.ts:2770-2862`).
`tests/test262-runner.ts:2537` calls `instance.exports.test` directly; `_start` is
never invoked, which has been true since before PR #177.

Bisect of `Array.prototype.toSorted.length` codegen:
- `ad819fe3` (April 13 peak): correct (`__extern_get` path) → returns 1
- `d6d9a550` (before PR #192 toSorted impl): correct → returns 1
- `bd8060e4` (pre-PR #177): **already broken** — emits `f64.const 0; return`
- `cb5d43fa` (PR #177 merge): same broken output

The bug predates PR #177. Its numbers overlapped with PR #177's regression bucket.

## Actual root cause

`compilePropertyAccess` in `src/codegen/property-access.ts` has a fast path for
`Function.length` (lines 1177-1196) that computes the value statically from the
TypeScript signature:

```ts
const paramCount = sig.parameters.filter((p) => {
  const decl = p.valueDeclaration;
  if (!decl || !ts.isParameter(decl)) return true;
  return !decl.dotDotDotToken && !decl.questionToken && !decl.initializer;
}).length;
```

This is wrong for library methods declared in `.d.ts` files. TypeScript's lib types
use `?` liberally for any parameter the runtime treats as optional, but the ES spec
pins `Function.length` for many methods to a fixed integer that disagrees with
TypeScript's required-param count. Example:

- `Array.prototype.toSorted(compareFn?)` → TS-derived length = 0
- ES spec: "The length property of this method is 1." (§23.1.3.34)
- test262 check: `verifyProperty(Array.prototype.toSorted, "length", {value: 1})`
- Before fix: emits `f64.const 0` → assertion fails

The pattern appears across test262: slice, splice, unshift, join, toSorted,
toReversed, with, toSpliced, setFullYear, setUTCHours, toPrecision, fromCodePoint,
Atomics.notify, etc. A quick scan shows ~70 `*/length.js` tests failing with
`assertion_fail` on this alone, plus many other tests that inline
`verifyProperty(Builtin.method, "length", ...)`.

## Fix

In the Function.length fast path, defer to `__extern_get("length")` ONLY when
both conditions hold:

1. At least one call/construct signature is declared in a `.d.ts` file
   (`isLibrarySig`).
2. The root of the property-access chain is an Identifier that is reachable at
   runtime — an entry in `BUILTIN_GLOBAL_ROOTS` (Object, Array, Function, Math,
   Date, RegExp, TypedArray ctors, errors, iterators, Symbol, BigInt, JSON,
   Reflect, Atomics, Promise, Map/Set/WeakMap/WeakSet, etc.) or `globalThis`.

If either condition fails, we keep the static paramCount emission.

Rationale for condition 2: bare library identifiers like `encodeURIComponent`,
`DisposableStack`, or `unescape` are declared in `.d.ts` but have no runtime
externref binding in the host-imports path — compiling them as expressions
produces `ref.null.extern`, and `__extern_get(null, "length")` throws TypeError
at runtime. By gating on a reachable root we keep the static path for bare lib
globals (where TS paramCount happens to match the spec value) while still
correcting the spec-vs-TS mismatch for methods reachable through
BUILTIN_CTOR_NAMES prototypes (toSorted, slice, splice, with, etc.).

User-defined functions still take the static path because their call signature
declaration is the TS AST node, not a `.d.ts` — their runtime representation is
a Wasm funcref wrapped as externref where `__extern_get(f, "length")` would
return undefined.

Chain walk: `while (isPropertyAccess(n) || isElementAccess(n)) n = n.expression;`
— stops at the innermost expression (Identifier, ThisExpression, or computed).
`fctx.localMap.has(rootText)` / `boxedCaptures.has(rootText)` checks guard
against shadowed globals (e.g. `const Array = ...; Array.prototype.toSorted`).

## Verification

- Targeted repro (`.tmp/verify-1147-v2.mts`): 16 defer-path cases return correct
  runtime `.length` (toSorted=1, toReversed=0, with=2, toSpliced=2, slice=2,
  splice=2, unshift=1, join=1, abs=1, pow=2, setFullYear=3, fromCodePoint=1);
  4 static-path cases match Node (encodeURI=1, decodeURI=1, encodeURIComponent=1,
  decodeURIComponent=1); user `double.length=1` stays on static path.
- Bare-global probe (`.tmp/probe-regressions.mts`): DisposableStack=0,
  AsyncDisposableStack=0, unescape=1, escape=1, parseInt=1, parseFloat=1,
  isNaN=1, isFinite=1 — all match ES spec via the static paramCount path.
- `tests/equivalence/` full suite: **105 fail / 1183 pass** — identical to
  pre-fix baseline. Zero equivalence regressions.
- CI revision 1 (initial fix, too aggressive): +21 net / 18 type_error
  regressions (DisposableStack ×12, new Function().call ×1, encodeURI* ×4,
  unescape ×1).
- Refined fix gates on `rootIsReachableBuiltin` so the 18 bare-lib cases stay
  on the static path.
- `Math.*.length` continues to use the earlier `mathMethodArity` table (no
  change in codegen for Math methods).
