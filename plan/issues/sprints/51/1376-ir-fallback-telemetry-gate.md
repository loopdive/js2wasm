# #1376 — IR fallback telemetry gate

## Problem

The `IrFallbackReason` type exists (`src/ir/select.ts:68`) and `trackFallbacks` flag is wired,
but the telemetry is never surfaced to CI. The IR retirement goal — "once the count of
unintended fallbacks is zero against the test262 corpus, the legacy emitters can be retired" —
has no automated enforcement. Without a gate, the fallback counts can silently grow as new
code is added.

The `planIrCompilation` call in `src/codegen/index.ts:782` passes `trackFallbacks` but the
`IrIntegrationReport.errors` are only logged locally; no count is exported or checked.

## Implementation plan

### Step 1 — Export fallback counts

In `src/ir/integration.ts`, extend `IrIntegrationReport`:
```typescript
export interface IrIntegrationReport {
  compiled: string[];
  errors: { func: string; message: string }[];
  fallbacks: Map<IrFallbackReason, string[]>;  // reason → list of function names
}
```

Populate from `planIrCompilation`'s `fallbackReasons` map. Distinguish:
- **Unintended**: `body-shape-rejected`, `external-call`, `call-graph-closure`,
  `param-shape-rejected` — these should decrease over time.
- **Deferred**: `deferred-feature`, `async-generator`, `async-function` — these are expected
  until their corresponding slices land; don't count toward the failure threshold.
- **Class-method**: once #1370 lands, methods not yet claimed go here.

### Step 2 — Snapshot baseline

Add `scripts/ir-fallback-baseline.json` — a committed JSON file recording current counts per
`IrFallbackReason`, generated by running `compileIrPathFunctions` over the equivalence test
suite inputs with `trackFallbacks: true`:

```json
{
  "generated": "2026-05-08",
  "unintended": {
    "body-shape-rejected": 142,
    "external-call": 87,
    "call-graph-closure": 63,
    "param-shape-rejected": 38
  },
  "deferred": { ... }
}
```

### Step 3 — CI gate script

`scripts/check-ir-fallbacks.ts`:
- Compiles each `.ts` file in `tests/ir/` (or the equivalence fixtures) with
  `trackFallbacks: true`.
- Reads `ir-fallback-baseline.json`.
- Fails if any **unintended** fallback count has increased vs. baseline.
- Prints a diff table when counts change (up or down).
- Succeeds and updates the baseline snapshot automatically when all counts decrease or are equal.

Add to CI quality job (alongside `tsc --noEmit` and `biome`).

### Step 4 — Living documentation

Add `## IR Fallback Budget` section to `CLAUDE.md` listing each unintended fallback reason,
its current count, and the issue that will reduce it.

## Acceptance criteria

1. `pnpm run check:ir-fallbacks` exits 0 on main.
2. CI quality job runs `check:ir-fallbacks` and fails when a PR increases unintended fallbacks.
3. `ir-fallback-baseline.json` is committed and up to date.
4. PRs for #1370/#1371/#1372 that reduce fallbacks show a green diff in the gate output.

## Files

- `src/ir/integration.ts` — extend `IrIntegrationReport` with `fallbacks`
- `scripts/ir-fallback-baseline.json` — new committed baseline
- `scripts/check-ir-fallbacks.ts` — new gate script
- `.github/workflows/quality.yml` — add gate step
- `package.json` — add `"check:ir-fallbacks"` script

## Notes

This is the retirement gate. It costs ~1 day to build but pays for itself by making IR
expansion PRs self-verifying. Do this before the #1370–#1375 series lands so they each show
a measurable improvement in the gate output.
