# #996 -- Array.prototype.toSorted comparefn-not-a-function hits 30s compiler timeout

## Problem

The last full recheck (`benchmarks/results/test262-results-20260407-005506.jsonl`)
contains a **30s compile timeout** for:

- `test/built-ins/Array/prototype/toSorted/comparefn-not-a-function.js`

This is another singleton timeout that likely belongs to array callback or
sorting helper lowering, not the other timeout clusters.

## ECMAScript spec reference

- [§23.1.3.34 Array.prototype.toSorted](https://tc39.es/ecma262/#sec-array.prototype.tosorted) — step 2: if comparefn is not undefined and not callable, throw TypeError


## Suggested fix

1. Reproduce this file in isolation
2. Compare with passing `toSorted` tests and existing array callback issues
3. Fix the slow path so the test fails or passes quickly instead of timing out

## Acceptance criteria

## Test Results

After fix (commit pending):
- `Array/prototype/toSorted/comparefn-not-a-function.js` — compiles in 525 ms with `skipSemanticDiagnostics` (the test262 worker uses this option). Runs to completion (returns 2 = assertion fail in `toSorted` not throwing TypeError as expected, but no longer hangs).

The 30-second compile_timeout was a runtime hang in `for (var i = 0; i < N; i++) { fn(function() { ... i ... }); }`. The closure correctly captured outer `i` as mutable; boxing was emitted lazily AT the closure-creation site (mid-loop body). The for-loop condition's `local.get i` was emitted earlier and read the unboxed local — never seeing `i++`'s ref-cell update.

Fix: `preBoxClosureCaptures` runs at function entry, before any statement compilation. It scans every nested closure for outer-scope variables captured-as-mutable and boxes them up front so reads/writes always go through the same ref cell. The fix in `compileVariableStatement` and the for-loop initializer detects when the slot is already a ref cell and writes through it via struct.set.
