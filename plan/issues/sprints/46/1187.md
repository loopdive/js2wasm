## Implementation Notes (senior-developer pickup)

Picked up from dev-2's suspended state on 2026-04-30. Resolved both
the late-import shift bug AND a pre-existing `__str_to_extern`
validation bug. All 7 round-trip tests pass; 0 equivalence regressions.

### Root cause analysis

Two distinct bugs were in play, only one of which dev-2 had identified:

**Bug 1 — hook position**: dev-2's pre-register hook ran AFTER
`collectAllSourceImports` (line 649), but that collector invokes
`ensureNativeStringHelpers` (via `finalizeUnifiedCollector` line 821)
when the source has any string literal — including the implicit
`"identity"` function-name literal that every named function carries.
So the helpers were emitted with `numImports=0` BEFORE dev-2's hook
added the 3 bridge imports. After the imports were added, the helpers'
actual binary positions shifted by 3 but their already-emitted funcIdx
cross-references stayed stale. `__str_flatten`'s `call $__str_copy_tree`
ended up pointing at `__str_from_mem`, producing the
`call[0] expected type i32 …` validation error.

**Bug 2 — `__str_to_extern` body** (pre-existing, latent): the function
signature declares the param as `(ref $AnyString)` but the body issued
`struct.get $NativeString fieldIdx 1` directly on the param.
`$AnyString` only has field 0 (`len`); fields 1 (`off`) and 2 (`data`)
live only on the `$NativeString` subtype. Wasm validation rejects with
`struct.get[0] expected type (ref null 2), found local.get of type
(ref null 1)`. The bug never surfaced before because:
- `String.fromCharCode + WASI` (the only normal end-to-end caller)
  was already blocked by an unrelated late-import shift bug.
- The other internal caller emits `emitFlatten()` before the call, but
  the validator checks the callee body in isolation against the
  declared param type, not against any specific call-site flow.

### Fix shape

**For bug 1**: split testRuntime hooks in `generateModule` —
pre-register the 3 bridge imports as REGULAR imports BEFORE
`collectAllSourceImports`, then force-emit
`ensureNativeStringExternBridge` AFTER. By then `ensureLateImport`
short-circuits on the pre-registered names, so no shift queue is
ever opened and the bridge bodies emit with stable funcIdx values.

**For bug 2**: prepend a flatten-and-cache preamble to
`__str_to_extern`'s body — call `__str_flatten` on the param, store the
result in a new `flat` local of type `ref_null $NativeString`, route
subsequent `struct.get $NativeString` reads through `local.get $flat
+ ref.as_non_null`. Body now validates whether the call site
flattened or not.

### Remaining

The forced bridge emission produces a structurally broken Wasm binary. With `testRuntime: true` + `nativeStrings: true` and a tiny source like
`export function identity(s: string): string { return s; }`,
`WebAssembly.instantiate` rejects function `__str_flatten` with
`call[0] expected type i32, found local.get of type (ref null 0) @+652`.

Diagnosis so far:

- `ensureNativeStringExternBridge` calls `ensureNativeStringHelpers` first, which emits `__str_copy_tree`, `__str_flatten`, `__str_concat`, … (~20 functions) and captures their `funcIdx` values.
- `ensureNativeStringExternBridge` then calls `ensureLateImport` for `__str_from_mem`, `__str_to_mem`, `__str_extern_len`. These imports go into `ctx.pendingLateImportShift`.
- `ensureLateImport` requires `flushLateImportShifts(ctx, fctx)` to be called against a `FunctionContext` to actually shift the captured `funcIdx` values inside helper bodies.
- We are at module top level here — there is no `fctx` to flush against. The shift never happens, so `__str_flatten`'s `call $__str_copy_tree` stays at the pre-shift index, which now points to an import (`__str_from_mem`) instead of the defined `__str_copy_tree` function.

Pre-registering the three bridge imports as regular imports BEFORE calling `ensureNativeStringExternBridge` was attempted in the current branch state — `ensureLateImport` correctly short-circuits when `funcMap` already has the name, but the failure persists, suggesting another late-import path inside `ensureNativeStringHelpers` is also queueing shifts. Worth tracing every `ensureLateImport` reachable from `ensureNativeStringHelpers` and pre-registering each one, OR adding a `flushModuleLevelLateImportShifts` helper that walks `ctx.mod.functions` directly without a `FunctionContext`.

Also worth considering the simpler "host-import" alternative noted in the original design (Notes section): expose `__test_str_from_externref` as a HOST IMPORT (JS-side), not a Wasm-side helper. JS can't call `struct.new` directly, but `buildImports` could provide a JS function that invokes the existing `__str_from_extern` Wasm export internally. That sidesteps the late-import shift problem entirely.

# #1187 — Test runtime: JS-string ↔ native-string coercion helper

## Problem

In `nativeStrings: true` mode, a Wasm export with a `string` param has
the Wasm signature `(ref $AnyString) -> ...`. JS callers cannot pass a
JS string into that param — `instance.exports.fn("hello")` throws
"type incompatibility when transforming from/to JS" because JS strings
auto-coerce to `externref` but the param wants a `(ref $AnyString)`
struct.

This blocks proper dual-run equivalence testing for any IR-claimable
function whose signature touches strings:

  - `function fn(s: string): number` — broken (param coercion fails)
  - `function fn(): string` — broken (return value comes back as
    `(ref $AnyString)`, JS sees it as an opaque struct)

`tests/issue-1183.test.ts` worked around this by switching to inline
string literals (`const s = "hello"; for (...)`) so no string ever
crosses the JS↔Wasm boundary as a param. But this is a band-aid:
future tests for native-strings features (e.g. `String.prototype.X`
through IR) need to feed real strings in and read real strings out.

## Goal

Provide a small test-only host helper that converts between JS
strings and `(ref $AnyString)` structs. Mount it from the test
runtime (`src/runtime.ts`) so the same WebAssembly module exports
become callable with JS-string args during testing.

## Design

### Helper exports (test-runtime side)

Add two exports to `buildImports`:

```ts
// src/runtime.ts
export function buildImports(
  imports: ...,
  envStub: ...,
  stringPool: ...
): { env: Record<string, unknown>; string_constants: ...; testHelpers: TestStringHelpers }
```

where `TestStringHelpers` exposes:

```ts
interface TestStringHelpers {
  /** JS string → ref $AnyString. Returns undefined when nativeStrings mode is off. */
  toNative?: (s: string) => unknown;
  /** ref $AnyString → JS string. Returns undefined when nativeStrings mode is off. */
  fromNative?: (s: unknown) => string;
}
```

### Wasm-side support

The compiler exposes a small pair of helpers when `nativeStrings:
true`:

  - `__test_str_from_externref(externref) -> ref $AnyString` —
    coerces a JS-side externref string to a NativeString struct by
    walking code units.
  - `__test_str_to_externref(ref $AnyString) -> externref` — reverse.

These are EXPORTS of the module (gated behind a new `--test-runtime`
or `testRuntime: true` compile flag) so test code can call them
directly. Production builds drop them.

### Test usage

```ts
const r = compile(source, { experimentalIR: true, nativeStrings: true, testRuntime: true });
const built = buildImports(r.imports, ENV_STUB, r.stringPool);
const { instance } = await WebAssembly.instantiate(r.binary, { env: built.env, ... });
const fn = instance.exports.fn as (s: unknown) => unknown;
const toNative = instance.exports.__test_str_from_externref as (s: string) => unknown;
const fromNative = instance.exports.__test_str_to_externref as (s: unknown) => string;

const result = fn(toNative("hello"));     // pass a string
const asJs = fromNative(result);           // read back as JS string
```

A thin convenience wrapper in test code:

```ts
const callWithStrings = <R>(fn: (...) => R, ...args: unknown[]): R => {
  const coerced = args.map(a => typeof a === "string" ? toNative(a) : a);
  const r = fn(...coerced);
  return typeof r === "object" && r !== null ? fromNative(r) : r;
};
```

## Acceptance criteria

1. New `testRuntime: true` compile option.
2. When `testRuntime && nativeStrings`, the module exports
   `__test_str_from_externref` and `__test_str_to_externref` helpers.
3. New tests in `tests/native-strings-roundtrip.test.ts` exercise
   the round-trip: `fromNative(toNative("hello")) === "hello"`,
   including BMP unicode and the empty string.
4. Re-enable the dual-run cases in `tests/issue-1183.test.ts` (the
   native-mode tests) using the new helper, AFTER #1186 lands the
   legacy fix. (This is documentation of intent; actual re-enable
   PR is separate.)
5. Production builds (`testRuntime` unset) emit zero overhead — the
   helpers are absent from the module entirely.

## Out of scope

- Changing the production string ABI. JS↔Wasm string passing for
  real applications already has solutions (`extern.convert_any` for
  externref, separate string ABI for native).
- Performance — these are test-only helpers; correctness over
  speed.

## Notes

- A simpler alternative is to just expose a host import like
  `__from_string(externref) -> ref $AnyString` that lives on the
  JS side and walks the JS string char-by-char, calling the
  existing native-string struct constructor. That avoids generating
  Wasm bytecode for the coercion entirely.
- The legacy `nativeStringLiteralInstrs` (in
  `src/codegen/native-strings.ts:24`) shows the inline pattern for
  materializing a literal — the JS-side helper would do the same
  thing dynamically.
