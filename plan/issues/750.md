# #750 -- Insert missing extern.convert_any coercions at call sites

## Status: done

## Problem

1,299 test262 compile errors with pattern:
`call[N] expected type externref, found local.get of type (ref null M)`

When a struct ref (WasmGC object) is passed to a function expecting externref, the compiler
must insert `extern.convert_any` to convert the GC ref to externref. This coercion was missing
at many call sites.

## Complexity: M

## Implementation Summary

### What was done

Two fixes applied:

1. **typeof runtime fallback coercion** (`src/codegen/typeof-delete.ts`):
   When `typeof` falls through to the `__typeof` host helper (for union/unknown types),
   the operand must be externref. The coercion already handled f64 and i32, but was missing
   ref/ref_null. Added `extern.convert_any` for ref/ref_null operands.

2. **Call argument fixup pass** (`src/codegen/stack-balance.ts`):
   Added `fixCallArgTypesInBody` that runs as a post-compilation pass on each function body.
   For every `call` and `return_call` instruction, it walks backward through the instruction
   stream to find argument-producing instructions. When a simple instruction (local.get,
   global.get, struct.new, etc.) produces a ref/ref_null but the callee expects externref,
   it inserts `extern.convert_any` between the value producer and the call.

   Key design decisions:
   - Only inserts `extern.convert_any` (ref->externref), the safest lossless coercion
   - Only handles simple value producers (local.get, struct.new, etc.), not complex control flow
   - Stops at control flow boundaries (if, block, loop, try) to avoid stack balance issues
   - Accounts for pass-through transformers (delta=0 instructions like ref.cast) between the
     value producer and the call to avoid double-coercing
   - Runs before existing stack-balance fixups so they can clean up any remaining issues

### What worked
- Conservative approach: only fix ref/ref_null -> externref (the most common pattern by far)
- Stop at control flow boundaries to avoid breaking stack balance
- Track pass-through transformers to prevent double-coercing when codegen already inserted coercion

### What didn't work
- First attempt included lossy coercions (drop + default for f64->externref) which broke 93 tests
- First attempt was too aggressive with backward stack tracing, incorrectly modifying values
  used by if-conditions and other non-call consumers

### Files changed
- `src/codegen/typeof-delete.ts`: Added ref/ref_null -> externref coercion in typeof fallback
- `src/codegen/stack-balance.ts`: Added fixCallArgTypesInBody pass, updated stackBalance to run it

### Tests
- 0 regressions in equivalence tests (211 pre-existing failures, 3787 passing)
- New test `/tmp/test-broad.ts` validates all common patterns
