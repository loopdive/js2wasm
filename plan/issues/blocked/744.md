# #744 — Function monomorphization for polymorphic call sites

## Status: open

## Problem

When whole-program type flow (#743) detects that a function is called with different type signatures at different call sites, it must widen to `externref`. This loses type information even though each individual call site has known types.

```javascript
function identity(x) { return x; }
identity(42);      // wants f64
identity("hello"); // wants externref (string)
// Without monomorphization: x: externref everywhere
```

JIT engines solve this with inline caches. We can do better: generate specialized copies at compile time.

## Approach

### When to monomorphize
After whole-program analysis (#743), identify functions where:
1. Parameter types conflict across call sites (would widen to externref)
2. Each individual call site has a concrete type signature
3. The function body is small enough that duplication is worthwhile (heuristic: < 100 instructions)

### How
For each distinct type signature observed at call sites:
1. Clone the function body
2. Compile with concrete parameter types
3. Name it `$funcName$mono_f64_f64` (or similar mangled name)
4. At each call site, emit a call to the matching specialization

### Example
```javascript
function square(x) { return x * x; }
square(3);     // → call $square$mono_f64  (f64.mul)
square(3n);    // → call $square$mono_i64  (i64.mul)
```

Two function copies, each using native Wasm arithmetic. Zero externref overhead.

### Limits
- Don't monomorphize for externref call sites (no benefit)
- Cap at N specializations per function (default: 4) to avoid code bloat
- Recursive functions: monomorphize the entry point, recursive calls use same specialization

### Code size trade-off
Monomorphization increases binary size. Mitigate with:
- Only monomorphize small functions (< 100 Wasm instructions)
- Binaryen `wasm-opt` deduplication pass removes identical function bodies
- For large functions, prefer #745 (tagged unions) over monomorphization
