# #487 -- User Symbol as property key via tagged struct variant (~60 tests)

## Status: blocked

Blocked on: #483 (Symbol constructor / narrowed skip filter) must land first to
determine the true test count. Also depends on #481 (well-known Symbol support) for
the compile-time well-known symbol field resolution that this builds on.

### Only ~17 tests genuinely need user Symbol as a property key
The actual pattern `obj[sym] = value` where `sym` is a user-created Symbol appears
in only ~17 tests. Examples:
- `test/language/expressions/call/spread-obj-symbol-property.js`
- `test/language/expressions/new/spread-obj-symbol-property.js`
- `test/language/expressions/array/spread-obj-symbol-property.js`

These tests create a Symbol, use it to set a property on a plain object, then spread
that object and verify the symbol-keyed property is preserved.

### The core problem: WasmGC structs have static fields
WasmGC struct types have fields defined at compile time. User-created Symbol keys
are runtime values -- there is no way to add a new field to a struct at runtime.

The proposed approach (adding a `Map<i32, externref>` field to structs) is viable but:
1. Requires Map/Set host imports (#458, done) as foundation
2. Every struct that might receive a symbol-keyed property needs the extra field
3. Type inference must detect which objects could receive symbol keys
4. `in` operator, `delete`, and spread must handle the symbol map

### Feasibility reassessment: hard (was medium)
The original estimate of M complexity was too low. This is genuinely hard because:
- Compile-time detection of "which objects might receive symbol keys" is imprecise
- Spread with symbol properties requires iterating the symbol map
- `Object.getOwnPropertySymbols` would also be needed for full conformance
- Only ~17 tests benefit directly

### Recommendation
Move to blocked. The 60-test count in the original issue was inflated -- most of those
are well-known symbols (handled by #481) or Symbol identity tests (handled by #483).
Only ~17 tests genuinely need this feature, making the ROI very low.

## Approach: Tagged struct variant

At compile time, detect objects that use symbol-keyed properties and add an extra field:

```
struct Object {
  field $name (ref string)
  field $value f64
  field $__symbol_props (ref null $SymbolMap)  // only on tagged variants
}
```

Where `$SymbolMap` is `Map<i32, externref>` (symbol id -> value).

### Compile-time detection
1. Scan the AST for `obj[expr]` where `expr` is typed as `symbol` or is a `Symbol()` call
2. If found, add `__symbol_props` field to that object's struct definition
3. Property access `obj[sym]` compiles to: check if sym is a known well-known symbol (use field index), otherwise look up in `__symbol_props` map

### Runtime behavior
- `obj[sym] = value` -> `map.set(sym_id, box(value))`
- `obj[sym]` -> `map.get(sym_id)` with unbox
- `sym in obj` -> `map.has(sym_id)`
- `delete obj[sym]` -> `map.delete(sym_id)`

## Complexity: L (revised from M)

## Acceptance criteria
- [ ] `const s = Symbol(); obj[s] = 42; obj[s] === 42` works
- [ ] Objects without symbol usage have no extra field (no regression)
- [ ] Well-known symbols still resolve to direct struct fields
- [ ] `s in obj` and `delete obj[s]` work for symbol keys
