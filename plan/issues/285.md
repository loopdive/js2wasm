# Issue #285: For-loop compile errors -- complex heads and function declarations

## Status: review

## Summary
~113 tests fail in language/statements/for with compile errors. In addition to the 15 runtime failures, many tests have complex for-loop heads (multiple declarations, comma expressions) or function declarations inside for-loop bodies that fail to compile.

## Category
Sprint 4 / Group D

## Complexity: S

## Scope
- Support multiple variable declarations in for-loop init (`for (var a = 1, b = 2; ...)`)
- Handle comma expressions in for-loop update (`for (...; ...; a++, b++)`)
- Support function declarations inside for-loop bodies
- Update for-loop compilation in `src/codegen/statements.ts`

## Acceptance criteria
- Complex for-loop heads compile
- Function declarations in loop bodies compile
- At least 20 compile errors resolved

## Implementation notes

### Changes made
- **`src/codegen/statements.ts`**: Enhanced `compileForStatement` initializer handling to support:
  - **Destructuring patterns**: Object (`{ a, b }`) and array (`[x, y, z]`) binding patterns in for-loop init now route to `compileObjectDestructuring` / `compileArrayDestructuring`
  - **Arrow/function expressions**: `for (var fn = (x) => x * 2, i = 0; ...)` now correctly infers closure struct ref types
  - **Class expressions**: `for (var C = class { ... }; ...)` now handled (skipped like in compileVariableStatement)
  - **Pre-hoisted var slots**: Reuses existing local slots for `var` declarations that were pre-hoisted by `walkStmtForVars`
  - **Error reporting**: Non-identifier binding patterns that can't be handled now get proper error messages

### What was already working
- Multiple simple variable declarations (`for (var a = 0, b = 10; ...)`)
- Comma expressions in update (`for (...; i++, j--)`) -- handled by expression compiler
- Function declarations in for-loop bodies -- handled by `hoistFunctionDeclarations` + `compileStatement`
- Empty for-loop heads (`for (;;)`)

### Pre-existing limitations not addressed
- Array destructuring with tuple-typed literals fails ("vec data is not array") -- limitation in `compileArrayDestructuring`
- Object destructuring with anonymous types fails ("not a known struct type: __object") -- limitation in `compileObjectDestructuring`
- Closure captures over `let` loop variables (per-iteration environments) -- runtime issue, not compile error

### Tests added
6 new equivalence tests covering arrow functions in init, multiple declarations, comma expressions in update, function declarations in body, empty head parts, and multiple declarations with different inits.
