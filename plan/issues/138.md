# #138 — Bug: valueOf/toString coercion on comparison operators

## Status: open

## Problem
Comparison operators (`>`, `>=`, `<`, `<=`, `==`, `!=`, `===`, `!==`) do not call `valueOf()` or `toString()` on objects before comparing. Test262 tests create objects with custom `valueOf` methods and compare them, expecting coercion.

Example failing test:
```javascript
var obj = { valueOf: function() { return 3; } };
assert.sameValue(obj > 2, true);  // fails: comparing struct ref, not number
```

## Root cause
The codegen emits direct `f64.gt` / `i32.gt_s` etc. on operands without checking if they are objects that need coercion via `valueOf`.

## Tests blocked
~20 test262 failures across `expr/greater-than` (7), `expr/less-than` (3), `expr/less-than-or-equal` (6), `expr/greater-than-or-equal` (2), `expr/strict-equals` (2), `expr/strict-does-not-equals` (2)

## Fix
In comparison codegen, when an operand is a struct type that has a `valueOf` method, emit a call to that method before the comparison instruction.
