# #679 — Dual string backend: js-host mode vs standalone mode

## Status: in-progress

25+ string operations currently use host imports (__str_concat, __str_slice, __str_includes, etc). With #599 nativeStrings, we have WasmGC i16 array strings. Need to cleanly support both.

### Two modes

**js-host mode** (default for browser/V8/SpiderMonkey):
- Use `wasm:js-string` builtins for all string ops (fastest on V8 — engine inlines these)
- String type: externref (opaque JS string)
- Zero overhead for JS interop (strings pass directly)

**standalone mode** (for wasmtime/wasmer/WASI/non-JS):
- All string ops compiled as pure Wasm functions operating on `(array (mut i16))` 
- No host imports for strings at all
- Implement: concat (alloc + copy), slice (alloc + copy), indexOf (loop), charCodeAt (array.get), comparison (element-wise loop), toString (identity), length (array.len)

### Mode selection
- `--target wasi` or `--nativeStrings` → standalone mode (already partially implemented)
- Default / `--target gc` → js-host mode
- `--target js` (new) → explicitly js-host mode

### Implementation
1. Each string operation has two implementations: `compileStringConcat_jshost()` and `compileStringConcat_native()`
2. A `ctx.stringMode: "js-host" | "native"` flag dispatches at compile time
3. The 25+ `__str_*` host imports are only registered in js-host mode
4. In standalone mode, emit inline Wasm functions for each string operation

### What exists today
- #599 decoupled nativeStrings from fast mode
- Native string type registration exists
- Some native string ops exist (str_eq_simd, etc)
- Missing: native concat, slice, indexOf, replace, split, trim, padStart, padEnd

## Complexity: L

## Implementation Progress

### Phase 1: Guard string_compare and String_fromCharCode (done)

**Problem**: In `nativeStrings=true` mode, `string_compare` and `String_fromCharCode` were still being registered as host imports, even though native Wasm implementations exist.

**Changes**:

1. **`src/codegen/index.ts`** (two sites each):
   - `string_compare` import now guarded with `if (ctx.nativeStrings) { ensureNativeStringHelpers(ctx); } else { addImport(...) }`.
   - `String_fromCharCode` import now guarded similarly -- in native mode, calls `ensureNativeStringHelpers` instead of adding the host import.

2. **`src/codegen/index.ts`** (new `__str_fromCharCode` helper):
   - Added pure Wasm implementation: creates a 1-element i16 array via `array.new_fixed`, wraps in NativeString struct (len=1, off=0, data=arr).

3. **`src/codegen/expressions.ts`**:
   - Updated `String.fromCharCode(code)` handler: in nativeStrings mode, compiles arg to i32, calls `__str_fromCharCode`; in host mode, unchanged (calls host import).

4. **`tests/native-strings-standalone.test.ts`**: Added 5 tests verifying:
   - `<`, `<=`, `>=` compile without `string_compare` import in native mode
   - `String.fromCharCode` compiles without host import in native mode
   - Host mode still correctly registers both imports

### Remaining work (future phases)
- `__str_from_mem`, `__str_to_mem`, `__str_extern_len` host imports still exist in `ensureNativeStringHelpers` for string marshaling to/from linear memory (needed for console.log interop). True standalone mode would need to remove these too.
- `number_toString` and other non-string host imports also need native implementations for full standalone.
