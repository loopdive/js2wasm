# #496 — eval() and new Function() via host-side dynamic compilation (533 tests)

## Status: open

533 tests use `eval()` or `new Function()` for dynamic code execution. Previously considered impossible in AOT Wasm, but achievable with a two-stage compilation approach.

## Approach

`eval(source)` compiles as a host import call:

1. Wasm module calls `__eval(sourceString, scopeSnapshot)` host import
2. Host runs ts2wasm on the source string → produces a new Wasm module
3. Host instantiates the new module, linking it with the caller's exports
4. Host returns the result as externref

### Scope sharing
The hard part: `eval` sees local variables from the calling scope.

**Simple approach**: serialize accessible locals into a scope object, pass to the eval host. The dynamically compiled module reads from this object. Mutations write back.

**Implementation**:
1. At `eval(...)` call sites, the compiler collects all in-scope local variables
2. Emits code to pack them into a scope struct and pass to the host
3. Host compiles the eval string with the scope struct as an import
4. After eval returns, unpack any mutations back to locals

### new Function()
Simpler — no scope sharing needed. `new Function("a", "b", "return a + b")` compiles the body with the named parameters. Returns a funcref.

## Complexity: L

## Acceptance criteria
- [ ] `eval("1 + 2")` returns 3
- [ ] `eval` can read variables from enclosing scope
- [ ] `new Function("a", "return a * 2")(5)` returns 10
- [ ] Host import is optional — if not provided, eval throws a runtime error
