# RegExp exec() result unpacking

## Problem

RegExp tests fail because `RegExp.exec()` returns an externref result whose properties (`.index`, `.input`, `[0]`, `[1]`, etc.) cannot be accessed. The property access fallback in `compilePropertyAccess` emits constant defaults (e.g., `f64.const 0`) instead of using `__extern_get` to dynamically read properties from the externref result.

## Root cause

1. `RegExp.exec()` returns `RegExpExecArray | null` which maps to externref in Wasm
2. `RegExpExecArray` is NOT an extern class (only `RegExp` is), so `isExternalDeclaredClass` returns false
3. The property access fallback emits constant defaults instead of using `__extern_get`
4. Test helper `buildImports` was missing `RegExp_new`, `RegExp_exec`, `RegExp_test` host imports

## Implementation Summary

### What was done

1. **`src/codegen/expressions.ts`**: In `compilePropertyAccess`, added an externref-aware path before the constant-default fallback. When the object is known to be externref (via `resolveWasmType` or local type check), emits `__extern_get(obj, propName)` with `__unbox_number` for numeric properties. Uses `ensureLateImport` + `flushLateImportShifts` to register imports on demand.

2. **`tests/equivalence/helpers.ts`**: Added RegExp host imports (`RegExp_new`, `RegExp_exec`, `RegExp_test`, property getters/setters) to the test helper's `buildImports` so equivalence tests can instantiate modules using RegExp.

3. **`tests/regexp-exec.test.ts`**: 8 equivalence tests covering `test()`, `exec()` null/match, `result.index`, `result.input`, `result[0]`, `result[1]`, global flag.

### Files changed
- `src/codegen/expressions.ts` — externref property access via `__extern_get`
- `tests/equivalence/helpers.ts` — RegExp host imports
- `tests/regexp-exec.test.ts` — new test file

### Tests now passing
- 8 new RegExp exec() equivalence tests
