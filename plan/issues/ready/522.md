# #522 — Object.keys() requires struct type argument (43 CE)

## Status: review

43 tests fail because `Object.keys(obj)` rejects non-struct arguments. The type check is too strict — it should accept `any`/`externref` typed objects and fall back to runtime enumeration or return empty array.

## Complexity: XS

## Implementation Summary

### What was done
When `compileObjectKeysOrValues` receives an argument whose type cannot be resolved to a struct name (e.g. `any`, `externref`), instead of emitting a compile error, the function now:
1. Compiles and drops the argument expression (to consume side effects)
2. Creates an empty backing array (`array.new_fixed` with length 0)
3. Wraps it in a vec struct with length 0
4. Returns the empty array as `ref_null` to the vec type

This applies to all three methods: `Object.keys()`, `Object.values()`, and `Object.entries()`.

### Files changed
- `src/codegen/expressions.ts` — Modified `compileObjectKeysOrValues` to handle non-struct arguments gracefully
- `tests/equivalence/object-keys.test.ts` — New equivalence test verifying Object.keys with any-typed argument

### What worked
- Simple fallback to empty array is correct for compile-time semantics since we cannot enumerate fields of an unknown type at compile time
- The fix is minimal and self-contained (replacing an error path with a graceful fallback)

### What didn't work
- The existing struct-typed Object.keys test fails at link time due to a pre-existing `__str_0` import issue (string literal thunks not provided by the test harness), unrelated to this change
