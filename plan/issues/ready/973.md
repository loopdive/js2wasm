# #973 — Incremental compiler state leak in CompilerPool

## Problem

~400 tests produce "invalid Wasm binary" in the test262 runner but validate fine when compiled standalone. The results are 100% reproducible (0 differences between identical runs).

## Root Cause (identified, not fixed)

The fork worker (`scripts/compiler-fork-worker.mjs`) uses `createIncrementalCompiler()` which shares a persistent `IncrementalLanguageService` (TypeScript checker) across compilations. Every 200 tests it recreates (`RECREATE_INTERVAL = 200`), but within each window, the TS checker accumulates type info from previous tests.

When test N introduces types (e.g. Promise, Error subclasses), the checker remembers them for test N+1. The codegen then picks up stale types and registers imports that shouldn't exist, corrupting the Wasm type index space.

Standalone compilation uses plain `compile()` which creates a fresh context each time — no leak.

## Key Files

- `scripts/compiler-fork-worker.mjs` — `createIncrementalCompiler()`, `RECREATE_INTERVAL`
- `src/index.ts:257` — `createIncrementalCompiler()` / `IncrementalLanguageService`
- `src/checker/index.ts` — `IncrementalLanguageService` implementation
- `src/codegen/index.ts` — module-level mutable state (funcTypeCache, struct registries)

## Prior Fixes (workarounds, not root cause)

- #963 phase 1: Added "Promise" to BUILTIN_SKIP (~295 CEs recovered)
- #963 phase 2: Fixed stack-balance sub-expression coercion bug (~263 CEs recovered)
- #966: Fixed collectExternDeclarations pre-registration

## Fix Approach

Either:
1. Reset all codegen-level mutable state at the start of each `compileSource()` — funcTypeCache, externClasses, import tables. The TS checker sharing is fine; the CODEGEN state leaks.
2. Reduce `RECREATE_INTERVAL` to 1 (fresh compiler per test) — correct but slower.
3. Move all mutable state from module scope into `CodegenContext` so it's per-compilation.

Option 3 is the proper fix.

## Acceptance Criteria

- Standalone validation matches runner results (within ±5 tests)
- No false CEs from fork contamination
- Runner performance not significantly degraded
