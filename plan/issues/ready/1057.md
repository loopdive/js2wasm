# #1057 — String.prototype.split result constructor !== Array

## Problem

`String.prototype.split` returns an array-like whose `.constructor` is not `Array`. Tests assert `__split.constructor === Array` and fail.

## Evidence from harvest

- **Test count:** 68 tests currently failing with this pattern
- **Top path buckets:**
  - `68 test/built-ins/String/prototype/split/*`
- **Top error messages:**
  - 24× `returned 2 — assert #1 at L18: assert.sameValue(__split.constructor, Array,`
- **Sample test files:**
  - `test/built-ins/String/prototype/split/argument-is-regexp-d-and-instance-is-string-dfe23iu-34-65.js`
  - `test/built-ins/String/prototype/split/arguments-are-new-reg-exp-and-void-0-and-instance-is-string-hello.js`
  - `test/built-ins/String/prototype/split/call-split-l-3-instance-is-string-hello.js`

## Root cause hypothesis

Our split implementation returns a WasmGC string[] ref instead of a proper JS Array. The prototype chain on the returned value does not point at `Array.prototype`.

## Fix

Ensure `split` materializes a true JS Array (goes through the same Array allocation path as `[]`/`Array.of`) so that `.constructor` resolves to `Array`. Alternatively, install `constructor === Array` on the string-array result's prototype surface.

## Expected impact

~68 FAIL, plus likely unblocks any downstream test that iterates `split` output via Array-specific methods.

## Key files

- src/codegen/expressions.ts (String.split host import / lowering)
- runtime split builtin
