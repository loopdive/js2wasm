# #773 -- Monomorphize functions: compile with call-site types, not generic externref

## Problem

Functions are compiled with externref parameters when the compiler doesn't know the argument type. But at many call sites, the exact struct type IS known. This forces an unnecessary round-trip: extern.convert_any at call → any.convert_extern + ref.cast inside the function.

## Approach

**Default to specific types, fall back to generic only when necessary:**

1. **Pre-pass**: Before compiling function bodies, scan all call sites to collect the concrete argument types each function is called with.

2. **Specialize**: For each function, if all call sites pass the same struct type for a parameter, type that parameter as the struct ref directly. No conversion needed.

3. **Monomorphize**: If a function is called with 2-3 different types, compile one variant per type combination. Use name mangling: `getX$Point`, `getX$Circle`.

4. **Generic fallback**: Only compile with externref parameters when:
   - The function is exported (unknown external callers)
   - The function is stored in a variable / passed as callback
   - The function is called with more than N different type combinations
   - The parameter type is genuinely `any` or a large union

5. **Call-site dispatch**: At each call site, call the specialized variant directly if one exists for the argument types.

## Benefits

- Eliminates extern.convert_any / any.convert_extern round-trips
- Direct struct.get instead of ref.test + ref.cast chains
- Smaller code (no multi-struct dispatch in specialized functions)
- Better V8 optimization (monomorphic call sites)

## Acceptance criteria

- Functions called with known struct types use struct ref parameters
- Only exported / polymorphic functions use externref parameters
- Net improvement in test262 pass count
- No regressions in equivalence tests
