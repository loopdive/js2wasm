# #779 -- Assert failures: tests compile and run but produce wrong values (8,700 tests)

## Problem

8,700 tests fail with "returned 2" (first assertion failed) or other non-1 return values. The code compiles, instantiates, and runs without crashing — but produces incorrect results.

## Root causes (estimated breakdown)

- ~3,000: Property access returns wrong value (type coercion, wrong struct field)
- ~2,000: Missing built-in method behavior (Array, String, Object methods)
- ~1,500: Control flow edge cases (scope, hoisting, TDZ, closures)
- ~1,200: Type coercion errors (toString, valueOf, implicit conversions)
- ~1,000: Other (prototype chain, Symbol, iterator protocol)

## Task: Error analysis and sub-issue creation

This is an **analysis-only** task — no src/ changes. The agent should:

1. Run the test suite (use `TEST262_WORKERS=2 pnpm run test:262`) or read latest results from `benchmarks/results/test262-results.jsonl`
2. Filter for tests with status "fail" and error messages containing "returned" or "assert"
3. Classify the top error patterns by:
   - Error message prefix (e.g., "assert.sameValue", "assert.throws", "returned N")
   - Category (language/expressions, built-ins/Array, etc.)
   - Root cause pattern (sample 10-20 tests per bucket, read the test source to understand what's being tested)
4. Create 5-8 sub-issues in `plan/issues/ready/` numbered #780-#787, each with:
   - Specific error pattern and test count
   - Sample test files
   - Suggested fix approach
   - Files to modify
   - Priority based on test count

## Acceptance criteria

- 5-8 sub-issues created with specific, actionable patterns
- Each sub-issue has ≥5 sample test files
- Total test count across sub-issues accounts for ≥80% of the 8,700
