# #1077 — PR CI fetches fresh baseline at runtime

## Problem

In `.github/workflows/test262-sharded.yml`, the regression diff step reads:

```bash
BASELINE="benchmarks/results/test262-current.jsonl"
```

This is whatever file is committed at the PR branch's tip — which is a copy
of the baseline as it was when the dev last ran `git merge main` locally. If
main's committed baseline has refreshed since then (even by minutes), the PR
compares against an outdated world:

- Tests that flipped on main between "dev merged main" and "PR CI runs" show
  up as "already in baseline" and hide real PR-caused regressions.
- Tests that flipped positive on main show up as PR improvements that aren't
  the PR's fault.

This is less catastrophic than the #1076 self-healing bug, but it is a
persistent source of PR-level noise and misattributed credit/blame.

## Fix sketch

Add a step early in the `regression-gate` (or current `merge`) job, only for
`pull_request` events, that fetches the latest baseline from `origin/main`
and writes it over the checked-out copy before running the diff:

```yaml
- name: Fetch fresh baseline from origin/main
  if: github.event_name == 'pull_request'
  run: |
    git fetch origin main --depth=1
    git show origin/main:benchmarks/results/test262-current.jsonl \
      > benchmarks/results/test262-current.jsonl
```

For `push` events the existing `BASELINE="benchmarks/results/test262-current.jsonl"`
path already uses the branch tip, which on main IS the canonical baseline;
no change needed there.

## Acceptance criteria

- [ ] PR CI runs always diff against `origin/main`'s current committed
      baseline, regardless of when the dev branched or last merged main in.
- [ ] The fetch step adds < 5s to PR CI wall time.
- [ ] Push-to-main runs are unaffected (they already use the checked-out
      baseline which is the tip on main).
- [ ] If `origin/main` lacks the baseline file for any reason, the step fails
      loudly rather than silently using the stale branch copy.

## Dependencies

- Depends on **#1076** so that `origin/main`'s baseline is actually kept
  fresh. Fixing #1077 without #1076 would just make PRs diff against a
  still-stale file.

## Risks

- **`git fetch` in Actions**: uses the default token, safe on public and
  private repos with the current workflow permissions (`contents: write`).
- **Race: baseline refresh lands while PR CI is mid-diff**: acceptable —
  diff runs against whatever snapshot it fetched at step start. The PR can
  be re-run if the result is ambiguous.
- **Concurrent PRs against different baselines**: each PR fetches its own
  snapshot and is consistent within its own run. Diffs across PRs may use
  slightly different baselines but that's already true today.

## Notes

- One-line fix in practice. File separately from #1076 only because it's an
  independent conceptual concern and can ship in a different PR.
- No test plan beyond the acceptance criteria — the change is observable in
  the workflow logs.
