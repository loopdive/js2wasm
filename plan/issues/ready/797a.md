# #797a -- Property descriptor static analysis (compile-time)

## Revised approach: compile away, don't emulate

Property descriptors should be resolved at **compile time** — no runtime flags, no per-instance metadata.

### How it works

1. **Per-shape descriptor table** (compile-time only):
   - The compiler already knows every struct type and its fields
   - Build a Map<structTypeIdx, Map<fieldName, {writable, enumerable, configurable}>>
   - Default: all true. Updated by Object.defineProperty/freeze/seal calls

2. **Object.freeze(obj)**:
   - Compiler marks the variable's type as frozen (flow-sensitive)
   - All subsequent `obj.x = y` emit `throw TypeError` instead of `struct.set`
   - No runtime data structure needed

3. **Object.defineProperty(obj, 'x', {writable: false})**:
   - Compiler marks field 'x' as non-writable in the type's descriptor table
   - Subsequent writes to that field emit TypeError

4. **Object.getOwnPropertyDescriptor(obj, 'x')**:
   - Compiler reads from the static table
   - Constructs descriptor struct with constant values inline

### Fallback for dynamic cases
When the compiler can't determine the type statically (externref, dynamic property names), delegate to host via `__getOwnPropertyDescriptor(obj, name)` / `__defineProperty(obj, name, desc)`.

### Files
- `src/codegen/index.ts` — build per-shape descriptor table during struct registration
- `src/codegen/expressions.ts` — compile Object.freeze/defineProperty/getOwnPropertyDescriptor

### Acceptance criteria
- Object.getOwnPropertyDescriptor returns correct descriptors for known types
- Object.freeze prevents writes (TypeError at compile time)
- Zero runtime overhead for the common case
