# #1127 — `new C()` with `var C = class {...}` resolves to inner class ctor when outer has nested class field

## Problem

When an outer class is declared via `var C = class {...}` (class-expression-bound-to-var) **and** has a nested class as a field (e.g. `B = class {...}`), `new C()` at the call site is compiled to the **inner** class's constructor, not the outer one. At runtime `c` is an instance of the inner class (with no outer fields), so any code that reads outer-class fields through `c` — including nested-class methods that access private or public fields of an outer-class receiver — gets null / wrong values.

The bug is *not* specific to private names; it affects public fields equally. It only manifests when the outer class-expression is bound to a `var`/`let`/`const` identifier (not a `class C {...}` declaration) and contains at least one nested class expression as a field.

## Minimal repro

```ts
var C = class {
  outer: string = 'test262';    // works identically with #outer
  B = class {
    method(o) { return o.outer; }
  }
}
let c: any = new C();
let innerB: any = new c.B();
innerB.method(c);                // returns null, expected 'test262'
```

**Diff matrix:**
- Replace `var C = class {...}` with `class C {...}` → works (returns `'test262'`)
- Remove the `B = class {...}` field → works
- Annotate `method(o: C)` instead of untyped `o` → works (struct dispatch uses compile-time type)
- Same failure with `#outer` private field and with public `outer` field

## WAT inspection summary

Compiling the repro produces three structs and three constructors:

```
(type $C              (struct (field $__tag i32) (field $outer (mut externref)) (field $B (mut (ref null 2)))))
(type $__anonClass_0  (struct (field $__tag i32) (field $outer (mut externref)) (field $B (mut (ref null 4)))))
(type $__anonClass_1  (struct (field $__tag i32)))

(func $C_new             ...)   ; func 2 — creates outer C
(func $__anonClass_0_new ...)   ; func 3 — creates a duplicate outer
(func $__anonClass_1_new ...)   ; func 4 — creates inner B
```

`__module_init` then emits:

```
call 4                 ; ← calls __anonClass_1_new (inner B)
extern.convert_any
global.set 11          ; __mod_c = (externref of inner-B instance)
```

So `c = new C()` is wired to the **inner class** constructor (func 4), not `C_new` (func 2). Consequently:
- `c` is actually an inner-B instance at runtime
- `innerB = new c.B()` compiles to `null` (the B field lookup on an inner-B instance fails/returns null)
- `innerB.method(c)` calls the method with a null `this` and a wrong-type `o`, and `o.outer` returns `null` via the `__extern_get` fallback

## Root-cause hypothesis (needs architect to confirm and pick fix)

`var C = class {...}` causes the class expression to be registered **twice** during the collection phase:

1. `src/codegen/declarations.ts:1590` — `collectClassDeclaration(ctx, decl.initializer, "C")` registers struct `C`.
2. Later in the same statement walk, `collectAnonymousClassesInNewExpr(stmt)` (called at `declarations.ts:1643`) recurses into the AST, encounters the same `ClassExpression` node, and calls `registerClassExpression` → `collectClassDeclaration(..., "__anonClass_0")` — registering a duplicate struct `__anonClass_0` with the same fields.

The inner `B = class {...}` class expression is registered as `__anonClass_1`.

At the `new C()` call site in `src/codegen/expressions/new-super.ts`:

- Line ~1847: `type = ctx.checker.getTypeAtLocation(expr); className = symbol?.name`
- Line ~1854: `if (className && !ctx.classSet.has(className))` — fallback checks `classExprNameMap.get(className)`
- Line ~1860: if still not resolved, check `ctx.classSet.has(expr.expression.text)` or `classExprNameMap.get(idName)`
- Line ~2099: `if (ctx.classSet.has(className)) { ctorName = \`${className}_new\`; ... }`

In this scenario the identifier `C` does resolve into `classSet`, but the resolution produces the wrong entry — either via the `classExprNameMap` (which maps `C` to one of the synthetic names) or because the symbol-derived `className` is a fallback synthetic that wasn't meant to be chosen. The net effect is that `funcMap.get(ctorName)` returns the index of `__anonClass_1_new` (func 4), i.e. the *inner* class's constructor.

**Architect should confirm**:
- Which lookup path is actually firing (symbol-name vs. identifier vs. `classExprNameMap`) — add a `console.log` at each branch in `compileNewExpression` and re-run the repro
- Whether the fix is
  - (a) **dedupe registration** — don't double-register the `var C = class {...}` expression; skip the `collectAnonymousClassesInNewExpr` recurse when the parent has already named it, or
  - (b) **fix the lookup priority** — when the identifier matches both a `classSet` entry and a `classExprNameMap` entry, prefer `classSet[idName]` (the outer) over the synthetic entry (which may point at an inner class), or
  - (c) **both** — dedupe is the clean fix; priority is the cheap patch if dedupe has ordering side effects

Given that `classExprNameMap` currently maps `C → __anonClass_0` (the *outer* duplicate, not the inner), the actual wrong-map pointer may come from a different path — possibly the inner class's `B = class {...}` initializer pollutes `classExprNameMap` because the walk order registers `__anonClass_1` under an alias that later gets queried for `C`. This is exactly what needs architect reading of `declarations.ts:1575-1578` + `new-super.ts:1847-1872` to pin down.
