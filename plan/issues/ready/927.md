# #927 -- Missing early/parse error detection (840 FAIL)

## Problem

840 tests expect a parse or early error (SyntaxError, ReferenceError) but the compiler compiles and instantiates them successfully. The test harness reports: `expected parse/early error but compiled and instantiated successfully`.

These are **negative tests** in the test262 suite — they contain intentionally invalid JavaScript that must be rejected at parse/compile time. Our compiler accepts them.

## Error pattern

```
expected parse/early error but compiled and instantiated successfully
```

## Sample test files

- `test/language/expressions/arrow-function/static-init-await-reference.js` — `await` used as identifier in static init
- `test/language/expressions/async-arrow-function/await-as-identifier-reference-escaped.js` — escaped `await` as identifier
- Tests span: arrow functions, async functions, class elements, destructuring, labels, identifiers

## Categories of missing checks (estimated)

| Category | Est. count | Description |
|----------|-----------|-------------|
| Reserved word violations | ~300 | `await`, `yield`, `eval`, `arguments` used as binding in strict/module context |
| Duplicate parameter names | ~150 | Non-simple params with duplicates |
| Invalid destructuring targets | ~100 | Assignment to non-assignable patterns |
| Class element restrictions | ~100 | Static init blocks with `await`/`yield`/`arguments` |
| Label violations | ~50 | Duplicate labels, break/continue to non-existent labels |
| Other | ~140 | Various early error conditions |

## Root cause

The TypeScript parser is lenient — it often accepts syntax that is invalid JavaScript (e.g., `await` as identifier in async context). The compiler needs an additional validation pass that checks JS-specific early error conditions from the ECMAScript specification (sections 12-15 "Static Semantics: Early Errors").

## Implementation Summary

### Approach

Extended the existing `detectEarlyErrors()` function in `src/compiler.ts` with ~15 new categories of ES-spec early error checks. All checks run after TS parsing but before codegen, producing compile errors that cause negative test262 tests to correctly fail compilation.

### New checks added (4 commits)

**Commit 1 (`de44eb47`)**: Core checks
- `arguments` in class field initializers (ContainsArguments)
- `await` with empty operand in async functions (identifier reference detection)
- `await` as label in async functions (colon-after-await pattern)
- `yield` with empty operand in generators
- `yield` as label in generators
- `return` outside function body
- `import`/`export` in non-top-level positions
- `import()` with spread arguments
- Duplicate export names (source-file level)

**Commit 2 (`099c1afa`)**: Escaped keywords and refinements
- Escaped keyword detection (`\u0061wait`, `\u0079ield`, `\u006Cet`, `im\u0070ort`)
- Escaped `async` modifier detection
- Refined import/export position checks for exported declarations
- `await` in class static initializer blocks
- `dynamic import()` as assignment target

**Commit 3 (`46ea59df`)**: Additional patterns
- Nullish coalescing (`??`) mixed with `||`/`&&` without parentheses
- Optional chaining with tagged template literal (`a?.`hello``)
- `void yield` / `typeof yield` / `+yield` patterns (empty operand detection)
- `super()` in constructor of class without `extends`
- Class fields named "constructor" (non-static)
- `for-of` var declarations with initializers (extended from lexical-only)
- `new.target` outside functions
- HTML close comment (`-->`) in module code
- Duplicate class constructors
- Fix: yield-as-binding check now includes generator methods

### Test Results

**Batch test (matching vitest test262 runner behavior):**
- 372/769 currently-failing negative tests now correctly rejected (48.4%)
- Up from 0/769 baseline (all 372 are new passes)
- 20 regression tests pass (valid code still compiles correctly)
- 8 additional safety tests confirm no false positives

**Below the 600/840 target because:**
- ~160 yield tests: `renameYieldOutsideGenerators` in wrapTest renames `yield` to `_yield`, making detection impossible at the compiler level
- ~48 escaped keyword tests: `resolveUnicodeEscapes` in wrapTest resolves escapes before compilation
- ~24 `using` syntax tests: TS doesn't parse `using` as a keyword in most contexts
- ~25 private field destructuring tests: TS parser handles `{ #x: x }` differently
- ~15 global-level tests: wrapTest puts code inside `function test()`, making `return`/`new.target` at global level uncatchable
- ~23 escaped async tests: resolved by wrapTest before compiler sees them

### Architectural Notes

The key insight is that the test262 vitest runner uses `wrapTest()` which transforms the source before compilation:
1. `resolveUnicodeEscapes()` — resolves `\uNNNN` sequences, destroying escaped keyword evidence
2. `renameYieldOutsideGenerators()` — renames `yield` identifiers, hiding spec violations
3. Function wrapping — puts everything inside `export function test()`, making global-level checks impossible
4. `skipSemanticDiagnostics: true` — the worker skips TS semantic diagnostics, so downgraded warnings don't help

To catch the remaining ~400 tests, changes would need to be made to wrapTest or the test262 runner flow (running early error detection before wrapTest transforms), not to the compiler's `detectEarlyErrors`.
