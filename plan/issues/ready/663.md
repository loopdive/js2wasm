# #663 — Null pointer dereferences: missing property access returns null instead of undefined (2,050 FAIL)

## Status: open

2,050 tests fail with "RuntimeError: dereferencing a null pointer". The dominant pattern:

1. **Computed property names** (`object[1]`, `object[expr]`) — the key isn't a struct field, so access returns null. The caller then does struct.get on null → trap.
2. **Missing properties** — accessing a property that doesn't exist on the struct. Returns null ref instead of externref undefined.
3. **Chained access** — `obj.a.b` where `obj.a` is null because `a` isn't a real field.

### Root cause
When property access fails to find a struct field, it falls through to externref dispatch which returns `ref.null.extern`. But the LOCAL that stores the result may be typed as `ref $SomeStruct` (non-nullable), so `ref.as_non_null` on the null traps.

### Fix (two approaches)

**Approach A — Widen locals to nullable:**
When a property access result is stored in a local, ensure the local is `ref_null` not `ref`. Then guard subsequent struct.get with ref.is_null checks.

**Approach B — Return undefined externref instead of null:**
When property access can't find a field, return a proper JS undefined externref (via host import) instead of ref.null. This lets downstream code handle it as a value rather than crashing.

**Approach C — Safety net in struct.get:**
Before every struct.get emission, check if the source is nullable. If so, wrap in `ref.is_null` → return default value.

Approach C is the most impactful (catches all cases) but adds overhead. A+B combined is zero overhead for the happy path.
