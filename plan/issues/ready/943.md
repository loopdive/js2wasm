# #943 — Test262 runner instability — 1,400+ pass variance between identical runs

## Problem

The test262 runner produces wildly different results on identical code. Same commit, same compiler, different runs:

- Run 205234: 17,717 pass
- Run 214050: 16,239 pass (-1,478)
- tester-855 rerun: 16,239 pass (same as above)
- tester-831 run: 16,268 pass

The 1,400+ pass variance makes it impossible to reliably detect regressions or improvements. A -65 regression could be real or noise.

## Suspected causes

1. **Worker timeouts** — 9 workers processing 48K tests in ~10 min. Under memory pressure, workers slow down and hit the 30s timeout, marking tests as `compile_timeout` instead of running them.
2. **Memory pressure from concurrent agents** — test262 runs alongside dev agents and the tech lead. Available RAM varies between 4GB and 14GB depending on what else is running.
3. **Stale disk cache** — cache keyed on compiler hash, but hash doesn't change for doc-only commits. Tests get stale cached results from prior runs.
4. **Fork worker GC pressure** — workers have 512MB max heap. Under system memory pressure, V8 GC becomes aggressive, slowing compilation past the timeout.

## Acceptance criteria

- Two consecutive test262 runs on identical code produce pass counts within ±50 (not ±1,400)
- Runner reports worker health metrics (timeout count, peak RSS, GC time)
- CI runs are reproducible
