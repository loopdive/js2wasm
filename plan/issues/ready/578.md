# #578 — npm/Node.js compatibility for WASI targets

## Status: open

Enable Node.js programs to compile to Wasm and run on WASI runtimes (wasmtime, wasmer, Fastly Compute, Cloudflare Workers). This is the key to the serverless use case.

## What Node.js programs need

### Tier 1 — Core runtime (enables basic programs)
- `console.log/error/warn` → WASI `fd_write` to stdout/stderr
- `process.exit(code)` → WASI `proc_exit`
- `process.env` → WASI `environ_get` / `environ_sizes_get`
- `Buffer` (basic) → TypedArray backed by linear memory
- `JSON.parse/stringify` → already supported

### Tier 2 — Module system (enables npm packages)
- `require()` → static module resolution at compile time (already have `compileProject`)
- `module.exports` / `exports` → Wasm exports
- `__dirname` / `__filename` → string constants from compile context
- `import`/`export` → already supported

### Tier 3 — I/O (enables server programs)
- `fs.readFileSync/writeFileSync` → WASI `path_open` + `fd_read` + `fd_write`
- `setTimeout/setInterval` → WASI `poll_oneoff` or host import
- Basic `http` module → requires socket WASI extensions (WASIX)
- `crypto` → pure Wasm implementations (SHA256, HMAC — see #508)

### Tier 4 — Full compatibility (enables most npm packages)
- `EventEmitter` → pure TypeScript, should compile today
- `Stream` → pure TypeScript with EventEmitter
- `path` → pure string manipulation, should compile today
- `url` → pure parsing, should compile today
- `util.promisify` → function wrapper, should compile today

## WASI imports needed

```wat
;; Tier 1
(import "wasi_snapshot_preview1" "fd_write" (func $fd_write (param i32 i32 i32 i32) (result i32)))
(import "wasi_snapshot_preview1" "fd_read" (func $fd_read (param i32 i32 i32 i32) (result i32)))
(import "wasi_snapshot_preview1" "proc_exit" (func $proc_exit (param i32)))
(import "wasi_snapshot_preview1" "environ_get" (func $environ_get (param i32 i32) (result i32)))
(import "wasi_snapshot_preview1" "environ_sizes_get" (func $environ_sizes_get (param i32 i32) (result i32)))
(import "wasi_snapshot_preview1" "clock_time_get" (func $clock_time_get (param i32 i64 i32) (result i32)))

;; Tier 3
(import "wasi_snapshot_preview1" "path_open" (func ...))
```

## Implementation approach

1. Add a `--target wasi` compiler flag
2. When targeting WASI, generate WASI imports instead of JS host imports
3. `console.log(x)` → serialize to string → write to fd 1 via `fd_write`
4. `process.env.KEY` → call `environ_get`, parse key=value pairs
5. `require("path")` → bundle the `path` module source and compile it inline

## What compiles TODAY for WASI

Many npm packages are pure JavaScript with no I/O:
- `lodash` functions (map, filter, reduce, etc.)
- `validator` (string validation)
- `uuid` (random generation with seed)
- Data structures (`immutable`, `rbtree`)
- Math/crypto libraries

The JWT showcase (#508) is an example — pure computation, no I/O needed.

## Complexity: L (tiered implementation)

## Acceptance criteria
- [ ] Tier 1: `console.log("hello")` compiles and runs on wasmtime
- [ ] Tier 1: `process.exit(0)` works
- [ ] Tier 2: `require("./lib")` resolves and bundles at compile time
- [ ] Tier 3: `fs.readFileSync("file.txt")` works via WASI
- [ ] A simple Express-like handler compiles and runs on Fastly Compute
