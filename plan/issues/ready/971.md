# #971 — Mixed assertion failures (~180 tests)

## Problem

~180 tests fail with various assertion errors that don't fit the major regression buckets. These are the long tail after fixing Array methods (#967), block scope (#968), and misc methods (#969).

## Approach

Re-analyze after #967-#969 are fixed — many of these may resolve as side effects. The remaining ones need individual investigation via error analysis.

## Acceptance Criteria

- Analyze remaining failures after #967-#969
- Fix any that are regressions from sprint 38 merges
- Document any that are pre-existing or correctness improvements

## Implementation Summary

### Root Cause

Array rest destructuring (`[x, ...y] = [1, 2, 3]`) silently truncated data. In `compileArrayLiteral` (literals.ts), the TS contextual type from the destructuring pattern gave a tuple type with fewer slots than the literal's elements. `compileTupleLiteral` only emitted values for the tuple slots, dropping excess elements. The rest element handler then saw a tuple (not a vec struct) and skipped.

### Fixes (3 files)

1. **literals.ts** — When contextual tuple type has fewer slots than the literal has elements, skip tuple path → use vec struct instead.
2. **expressions.ts** — When rest local was pre-allocated as externref, allocate fresh local with correct vec type.
3. **index.ts** — Same fix for function parameter destructuring rest elements.

### Pre-existing Failures (not regressions)

destructuring-extended, for-of-array-destructuring, default-parameters, computed-property-names, async/generator/arrow-call-apply tests all fail identically on main.
