# #888 -- Performance benchmark: js2wasm vs StarlingMonkey vs Javy

## Problem

We have no performance data comparing js2wasm against competing approaches. The STF application needs to show where js2wasm sits in the performance landscape. We claim "smallest possible Wasm output" and "no runtime bundled" — but we haven't proven the speed/size tradeoffs.

## Approach

Add StarlingMonkey, Javy, and Wasmtime as git submodules. Create a benchmark suite that compiles and runs the same JS programs through each toolchain, measuring:

### Metrics

1. **Output size**: .wasm file size for the same input program
2. **Compilation time**: time to compile JS → Wasm
3. **Startup time**: time from `instantiate()` to first instruction
4. **Execution time**: time to complete a benchmark program
5. **Memory usage**: peak RSS during execution

### Runtimes

| Toolchain | Compilation | Execution | Notes |
|-----------|------------|-----------|-------|
| **js2wasm + V8** | js2wasm AOT → .wasm | Node.js WebAssembly.instantiate | Our primary target |
| **js2wasm + Wasmtime** | js2wasm AOT → .wasm (WASI mode) | wasmtime CLI | Standalone, no JS host |
| **Javy** | javy compile → .wasm | Node.js / wasmtime | QuickJS bundled in Wasm |
| **StarlingMonkey** | componentize-js → .wasm | wasmtime (Component Model) | SpiderMonkey in Wasm |
| **V8 baseline** | N/A (JIT) | Node.js direct execution | The "1x" reference |

### Benchmark programs

Start simple, add complexity:
1. **fibonacci(40)** — pure compute, no runtime features
2. **JSON parse + stringify** — string handling, object creation
3. **Array sort (10K elements)** — array methods, callbacks
4. **RegExp matching** — regex engine
5. **Promise chain (1K deep)** — async runtime
6. **HTTP request handler** — real-world pattern (parse request, route, format response)

### Setup

```
benchmarks/

```

## Implementation

1. **Add submodules**:
   ```bash
   git submodule add https://github.com/bytecodealliance/StarlingMonkey StarlingMonkey
   git submodule add https://github.com/bytecodealliance/javy Javy
   ```
   Actually — submodules might be overkill for benchmarking. Better to install pre-built binaries:
   - `javy`: download release binary
   - `starlingmonkey`: use `componentize-js` npm package
   - `wasmtime`: download release binary

2. **Benchmark runner** (`benchmarks/run-benchmarks.sh`):
   - For each program × each toolchain:
     - Compile (measure time + output size)
     - Execute 10 times (measure avg time + peak memory)
   - Output structured JSON results

3. **Results page**: add to dashboard or generate standalone HTML comparison chart

## Acceptance criteria

- At least 3 benchmark programs running on all 4 toolchains
- Results include: output size, compile time, execution time, memory
- Comparison chart or table generated from results
- Results reproducible (scripts checked in, versions pinned)
- Wasmtime execution works for js2wasm (WASI mode)
