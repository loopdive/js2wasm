# #569 -- Wasm validation: call type mismatch f64 vs externref

## Status: in-progress

451 tests fail because function calls have f64 arguments where externref is expected, or vice versa.

### Root cause

In `compileStringBinaryOp`, when arithmetic/bitwise operators are used on mixed string/non-string operands (e.g. `"1" - new Number(1)`), the code unconditionally calls `parseFloat` on BOTH operands after compiling them. But if an operand already compiles to f64 (e.g. `new Number(1)` which returns a primitive f64), calling `parseFloat` (which expects externref) causes a Wasm validation error.

Similarly, several call argument compilation paths compile expressions without passing the expected type hint, causing type mismatches when the compiled result doesn't match the function's parameter type.

### Fix

1. In `compileStringBinaryOp` arithmetic path: check the actual result type of each operand before applying numeric conversion. Only call `parseFloat`/`__unbox_number` when the result is externref; skip conversion when already f64; apply `f64.convert_i32_s` for i32; use `coerceType` for ref/ref_null.

2. In element-access string method calls and optional-chain string method calls: pass `expectedType` to `compileExpression` instead of doing manual coercion after the fact.

3. In `number.toFixed()` argument compilation: pass `{ kind: "f64" }` as expected type.

4. In string concat (`+`) path: handle ref/ref_null operands by coercing to externref.

### Files to modify
- `src/codegen/expressions.ts` -- `compileStringBinaryOp`, string method call paths, `toFixed` paths
