# #569 -- Wasm validation: call type mismatch f64 vs externref

## Status: review

451 tests fail because function calls have f64 arguments where externref is expected, or vice versa.

### Root cause

In `compileStringBinaryOp`, when arithmetic/bitwise operators are used on mixed string/non-string operands (e.g. `"1" - new Number(1)`), the code unconditionally calls `parseFloat` on BOTH operands after compiling them. But if an operand already compiles to f64 (e.g. `new Number(1)` which returns a primitive f64), calling `parseFloat` (which expects externref) causes a Wasm validation error.

Additionally, several call argument compilation paths compile expressions without passing the expected type hint, causing type mismatches when the compiled result doesn't match the function's parameter type.

### Fix

1. In `compileStringBinaryOp` arithmetic path (both fast and non-fast modes): check the actual result type of each operand before applying numeric conversion. Only call `parseFloat`/`__unbox_number` when the result is externref; skip conversion when already f64; apply `f64.convert_i32_s` for i32; use `coerceType` for ref/ref_null.

2. In element-access string method calls and optional-chain string method calls: pass `expectedType` to `compileExpression` instead of doing manual coercion after the fact. Also handle VOID_RESULT by pushing default values.

3. In `number.toFixed()` argument compilation (3 locations): pass `{ kind: "f64" }` as expected type.

4. In string concat (`+`) path: handle ref/ref_null operands by coercing to externref.

### Files to modify
- `src/codegen/expressions.ts` -- `compileStringBinaryOp`, string method call paths, `toFixed` paths

## Complexity: M

## Implementation Summary

### What was done
- Fixed the core bug in `compileStringBinaryOp` where `parseFloat` was unconditionally called on both operands of string arithmetic, even when one operand was already f64 (e.g., `new Number(1)`)
- Applied the same fix to both fast mode (native string) and non-fast mode paths
- Fixed 3 `toFixed` argument compilation sites to pass `{ kind: "f64" }` expected type
- Fixed 2 string method call argument compilation sites (element access and optional chain) to pass expected type instead of manual post-hoc coercion
- Added ref/ref_null handling in string concat path to coerce to externref

### What worked
- The pattern `"1" - new Number(1)` now correctly compiles: `parseFloat` is only called for the string operand, while the `new Number(1)` operand (already f64) passes through without conversion
- All 117 existing tests pass with no regressions

### Files changed
- `src/codegen/expressions.ts` -- `compileStringBinaryOp` (both fast and non-fast arithmetic paths), `compileCallExpression` (toFixed, element-access string methods), `compileOptionalCallExpression` (toFixed, string methods)
