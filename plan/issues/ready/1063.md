# #1063 — createMathOperation closure ref (lodash chain, tier-1 blocker)

## Problem

Lodash `_createMathOperation.js` is a factory that returns a closure capturing a
function parameter `operator` and invoking it. Compiling the pattern fails with
low-level Wasm validation errors and/or returns the wrong result.

Minimal lodash source:

```js
function createMathOperation(operator, defaultValue) {
  return function(value, other) {
    // ... coercion ...
    result = operator(value, other);
    return result;
  };
}
```

## Two independent bugs hit the same pattern

### A) Inliner shared-instr bug (FIXED in this PR)

When an outer function (like `make`) is short enough to become an inlinable
candidate AND its body contains `ref.func` / `struct.new`, the inliner pushed
the **same `Instr` object** into both the original function body and each
inline site.  Later, `dead-elimination.ts` walks every function body and
remaps `funcIdx` / `typeIdx` in place — revisiting the shared instr twice,
remapping an already-remapped index, and corrupting the resulting Wasm.

Symptom on a synthetic minimal repro:

```ts
function make(op: any): (v: number) => any {
  return function (v: number): any { return op(v); };
}
function double(x: number): number { return x * 2; }
const f = make(double);
export function test(): number { return f(21); }
```

```
RT: WebAssembly.instantiate(): Compiling function #2:"make"
    failed: undeclared reference to function #0 @+562
```

The `ref.func 22` emitted inside `make`'s body for the inner `__closure_2`
was first remapped to `ref.func 6` (correct new index after dead-elim
removed 16 unused imports). Then, because `__module_init` inlined `make` and
shared the same instr object, the remap walker visited the instr a second
time and further remapped `6 → 0` (which happens to be the new index of an
unrelated import that used to live at old index 6).

**Fix (this PR):** shallow-clone each instr at the inline site in
`src/codegen/expressions/calls.ts` so each inlined copy owns its own
`Instr` object and is visited exactly once by later remap passes.

### B) Externref closure-captured callee body is empty (REMAINING)

Independent of (A), when the captured `operator` parameter is typed `any`
(which is what JS-parsed lodash sees after `--allowJs`-less compilation), the
inner closure body compiles without calling `op` at all. The generated
`__closure_N` body extracts `op` from the closure struct to a local and then
unconditionally returns `ref.null extern`, silently dropping the call:

```wat
(func $__closure_0 (type 6)
  (local $__self_cast (ref null 7))
  (local $op externref)
  local.get 0
  ref.cast (ref 7)
  local.set 2
  local.get 2
  struct.get 7 1
  local.set 3
  ref.null extern      ;; <-- op(v) silently replaced with null
  return
)
```

This is a codegen gap for externref-typed callees that originate from a
closure struct field. The typed variant (where `operator` has a
`(a: number, b: number) => number` contextual type) compiles and runs
correctly — proving the closure ABI and call_ref plumbing is fine; only
the untyped/`any` path is broken.

## Fix status

- Part (A) — inliner shared-instr — **FIXED** in this PR via
  `src/codegen/expressions/calls.ts` shallow-clone at inline site.
- Part (B) — externref closure-captured callee emission — **NOT FIXED** in
  this PR. Needs a follow-up spec to route externref callees through a
  runtime `__call_fn_*` export or unbox the externref to funcref and use
  `call_ref`. Filed as a deferred sub-issue: the typed-path workaround
  (adding a proper function type to the parameter in user code) is
  sufficient to unblock lodash math ops that use explicit types.

## Acceptance

- Minimal inlined-factory-returning-closure test passes (part A)
- Typed `createMathOperation` returns correct results for `add(6, 7)` (part A)
- Untyped `any` parameter case remains a known gap (part B), tracked here

## Test Results

- Part A minimal repro: compile + run → `f(21) = 42` ✓ (was "undeclared
  reference to function #0")
- Typed `createMathOperation(addOp, 0)` → `add(6, 7) = 13` ✓
- Untyped `operator: any` → closure body still silently returns null (known,
  deferred)
