# #1063 — Codegen bug in lodash-es/add.js path: `createMathOperation` closure emits undeclared function ref

## Problem

Compiling `node_modules/lodash-es/add.js` via `compileProject({ allowJs: true })` succeeds at the TS/codegen level but produces a Wasm binary that fails validation:

```
WebAssembly.compile(): Compiling function #17:"createMathOperation" failed:
  undeclared reference to function #11 @+3685
```

lodash-es `add.js` is:

```js
var add = createMathOperation(function(augend, addend) {
  return augend + addend;
}, 0);
export default add;
```

`createMathOperation` is a higher-order helper that captures the passed `operator` callback and returns a closure over it. The Wasm emitted references a function slot (#11) that was never actually declared in the module — implying the closure-capture lowering for a callback parameter passed from another file isn't allocating / registering the function slot correctly.

Surfaced by #1031.

## Notes

- Candidate area: `addUnionImports` and/or function-table allocation in `src/codegen/index.ts`. The fact that the referenced slot is #11 while the failing function is #17 suggests the callback's function-table index was computed before some shift/insert operation.
- Interacts with #1061 (multi-file `.js` support). This bug only becomes observable once #1060 + #1061 actually load the lodash-es bodies — but it's already triggered by the current partial path.
