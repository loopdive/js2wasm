# #1088 — Assertion location diagnostic misses `verifyProperty` / `verifyEqualTo`

## Summary

273 test262 tests report `returned 2 — assert #1 (found 0 asserts in source)`. The tests themselves execute correctly — they genuinely fail because the runtime doesn't support property descriptor attribute checks (writable, enumerable, configurable) or built-in `.length` properties. The bug is in the **diagnostic reporter**: when a test fails, it tries to locate the failing assertion in source to produce a human-readable error, but the source-scanning regex only recognizes `assert`, `assert.*`, and `$DONOTEVALUATE` — it misses `verifyProperty`, `verifyEqualTo`, `verifyWritable`, and other `propertyHelper.js` helpers.

## Affected tests

273 tests. Overwhelmingly `length.js` tests (60) checking `.length` on built-in functions, plus `Object.defineProperty`/`defineProperties` descriptor tests.

Example files:
- `test/built-ins/DataView/prototype/setFloat64/length.js`
- `test/language/arguments-object/mapped/nonwritable-nonenumerable-nonconfigurable-descriptors-set-by-param.js`
- `test/built-ins/Object/defineProperty/15.2.3.6-4-209.js`
- `test/built-ins/Number/prototype/toLocaleString/length.js`
- `test/language/expressions/class/elements/init-value-incremental.js`

## Root cause

`tests/test262-shared.ts:231` scans test source for assertion locations:

```ts
if (/^\s*(assert\b|assert\.\w+|\$DONOTEVALUATE)/.test(lines[i])) {
```

This regex catches `assert(...)`, `assert.sameValue(...)`, `assert.throws(...)`, etc. — but `propertyHelper.js` includes like `verifyProperty(...)`, `verifyEqualTo(...)`, and `verifyWritable(...)` don't start with `assert`. When `transformVerifyPropertyCalls` in `test262-runner.ts:1025` transforms `verifyProperty(obj, name, {value: X})` into `assert_sameValue(obj[name], X)`, the transformation happens on the compiled source — but the diagnostic scanner runs against the ORIGINAL source where the pattern is still `verifyProperty`.

The tests fail because:
1. `verifyProperty` calls WITHOUT a `value:` key get stripped (runner can't check writable/enumerable/configurable in Wasm)
2. `verifyProperty` calls WITH a `value:` key get transformed to `assert_sameValue` — these may fail because built-in `.length` properties aren't exposed, or because the runtime value differs from spec

The diagnostic issue masks the actual failure location, making triage harder.

## Proposed solution

Extend the regex at `tests/test262-shared.ts:231` to also match `propertyHelper.js` assertion functions:

```ts
if (/^\s*(assert\b|assert\.\w+|\$DONOTEVALUATE|verify\w+)/.test(lines[i])) {
```

This adds `verifyProperty`, `verifyEqualTo`, `verifyWritable`, `verifyEnumerable`, `verifyConfigurable`, `verifyNotWritable`, `verifyNotEnumerable`, `verifyNotConfigurable`, `verifyCallableProperty`, and `verifyPrimordialProperty` to the scan.

The same fix should be applied to `tests/test262-vitest.test.ts:492` (the vitest runner's copy of the same diagnostic).

This does NOT fix the underlying 273 test failures (which need runtime property descriptor support), but it makes the failure messages actionable: `assert #1 at L26: verifyProperty(DataView.prototype.setFloat64, "length", {...})` instead of `assert #1 (found 0 asserts in source)`.

## Effort estimate

**S** — single regex change in two files, no behavioral impact beyond diagnostic output.
