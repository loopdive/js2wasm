# #990 -- Remaining early-error gaps after detectEarlyErrors(): reserved words, module grammar, using, ASI

## Problem

The latest full recheck (`benchmarks/results/test262-results-20260407-111308.jsonl`)
still has **327 failures** with:

```text
expected parse/early error but compiled and instantiated successfully
```

This is no longer a generic “we need early errors” statement. The repo already
has a large `detectEarlyErrors()` pass in `src/compiler.ts`; the remaining gap is
the set of specific ES static-semantics families that still pass through it.

## Important clarification

The active early-error pipeline is **not Acorn-driven** here. The effective path
is:

- TypeScript parser + syntax diagnostics
- custom `detectEarlyErrors()` in `src/compiler.ts`

So the remaining failures are not “Acorn misses”; they are cases our current
validator still does not encode.

## Implemented in Sprint 39

The current partial implementation already closed several visible families:

- preserved script-vs-module goal for negative tests in the unified runner
- warning-only negative parse cases are now treated as successful negatives
- optional-chaining assignment is rejected as an early error
- top-level `using` in script goal is rejected
- `using` directly in `case` / `default` is rejected
- HTML close comments (`-->`) in module goal are rejected

Verified passing examples from the full run:

- `test/language/comments/single-line-html-close-without-lt.js`
- `test/language/expressions/optional-chaining/static-semantics-simple-assignment.js`

## ECMAScript spec reference

- Each grammar production specifies **Static Semantics: Early Errors** — e.g.:
  - [§13.1.1 IdentifierReference](https://tc39.es/ecma262/#sec-identifiers-static-semantics-early-errors) — reserved words
  - [§16.2.3 ExportDeclaration](https://tc39.es/ecma262/#sec-exports-static-semantics-early-errors) — module-only grammar
  - [§14.7 IterationStatements](https://tc39.es/ecma262/#sec-iteration-statements) — `using` in for-of


## Root cause

`detectEarlyErrors()` already covers many ES checks, but several families still
need dedicated logic because TypeScript either:

1. parses them into a permissive AST shape
2. downgrades them into diagnostics we tolerate
3. normalizes tokens enough that the current AST walk misses the specific ES rule

## Suggested fix

1. Split the remaining `327`-test bucket by family and add focused tests per family
2. Extend `detectEarlyErrors()` for the residual families:
   - reserved words in nested async/generator/private-method contexts
   - class-element early errors around private/static constructor names
   - remaining `using` / `await using` grammar restrictions
   - remaining module import/export placement rules
   - ASI restricted productions and comment/punctuator edge cases
3. Only reach for a second parser if a family is fundamentally invisible in the
   current TS AST/diagnostic surface

## Acceptance criteria

- reduce the residual early-error failure bucket well below the current `327`
- close the remaining `using` grammar misses visible in the April 7 full run
- each major family above has at least one dedicated regression test

## Test Results (slice 1 — module-item position)

This PR targets the largest clean sub-family: **ImportDeclaration /
ExportDeclaration / ExportAssignment nested inside any statement or
class-element context** (ES static semantics — ModuleItems must appear
at the top level of a Module).

- dedicated sweep over the 86 `parse-err-decl-pos-*` tests in
  `test/language/module-code/`: **86/86 now reject** (were 0/86).
- full sweep over all 263 `"expected parse/early SyntaxError but
  compiled and instantiated successfully"` failures on current main:
  **169/263 now reject** (+169 passes for this slice).
- added `tests/issue-990.test.ts` with 7 cases (5 negatives + 2
  positive no-regress): all pass via `.tmp/run-tests.mts`.

Remaining residual families (yield-as-ident, reserved words, class
early errors, using / await-using, module-code non-decl-pos, ASI) are
intentionally out of scope for this PR and will be separate narrow
slices to follow.

## Test Results (slice 2 — yield / await reserved identifier)

Adds a reserved-identifier check in `detectEarlyErrors()`:

- `yield` as Identifier / BindingIdentifier in strict-mode code, in
  module code, or inside any enclosing generator function is rejected.
- `await` as Identifier / BindingIdentifier in module code or inside
  any enclosing async function is rejected.
- Property-name / import-name / export-name positions are skipped.
- The wrapTest sentinel suppresses the check, matching slice 1.

Combined slice 1 + slice 2 result on the full early-error bucket:

- **217/263 now reject** (was 0/263 on current main; slice 1 alone:
  169/263).
- Slice 2 added **+48** additional rejections on top of slice 1.
- 200/200 passing test262 sample (broad random sample) stays clean —
  no false positives from this check.
