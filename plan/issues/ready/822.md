# #822 -- f64.add on externref — missing unbox coercion (57 CE)

## Problem

57 tests fail with Wasm compile errors because the codegen emits `f64.add` (or other f64 arithmetic) on an externref operand without first unboxing it to f64. The coercion from externref → f64 (via `__unbox_number` or `any.convert_extern` + appropriate cast) is missing in certain expression paths.

## Investigation (2026-03-28)

Actually 106 errors, not 57. Two main patterns:

1. **array.set in async generator closures** (~80 tests): `array.set[2] expected type f64, found array.get of type externref`. Happens in `__closure_N` functions for yield-star sync/async iteration. The generator machinery reads from an externref array and writes to an f64 array without unboxing.

2. **struct.new with if-expression** (~10 tests): `struct.new[0] expected type f64, found if of type externref`. Nested array destructuring defaults like `[[x,y,z] = [4,5,6]]` produce an if-expression that returns externref where struct.new expects f64.

3. **call/local.set mismatches** (~16 tests): Various call sites pass externref where f64 is expected.

Root cause: certain codegen paths (generator yield-star, nested destructuring defaults, closure bodies) don't insert externref→f64 unbox coercion before f64-typed operations.

## Acceptance criteria

- 106 compile errors from f64/externref type mismatch eliminated
- Arithmetic on externref values correctly unboxes first
