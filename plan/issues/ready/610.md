# #610 — Worker timeouts: 1,034 tests killed after 30s

## Status: review

1,034 tests fail with "timeout: worker hung > 30s". These are NOT compiler hangs — they're tests that take longer than the 30s timeout to compile + run.

On the full 53k suite, many tests are complex class/generator/async patterns that genuinely take >30s to compile.

## Approach

1. Increase default timeout to 60s
2. For tests that still timeout, retry once with 120s
3. Track which tests consistently timeout — those are compiler performance issues, not bugs
4. Consider per-category timeout overrides (class tests need more time)

## Complexity: S

## Implementation Summary

### What was done
1. **Increased default worker timeout** from 30s to 60s (`DEFAULT_WORKER_TIMEOUT_MS = 60_000`)
2. **Made timeout configurable** via:
   - CLI flag: `--timeout=N` (seconds), e.g. `--timeout=90`
   - Environment variable: `TEST262_TIMEOUT=N`, e.g. `TEST262_TIMEOUT=90`
   - CLI flag takes precedence over env var
3. **Added retry mechanism** for timed-out tests: after all categories finish, tests that timed out are retried once with a doubled timeout (e.g. 120s if default is 60s)
4. **Updated error messages** to reflect the actual configured timeout dynamically
5. **Updated usage comments** in the script header

### Files changed
- `scripts/run-test262.ts` — all changes in this single file

### What worked
- Simple, non-breaking change — the `WORKER_TIMEOUT_MS` variable is used by a single `runBatch` function
- Retry logic reuses existing `runBatch` infrastructure with a temporarily doubled timeout
- JSONL dedup (last entry wins) means retry results naturally override initial timeout failures

### What didn't apply
- Per-category timeout overrides were not implemented (complexity not justified yet)
- The retry is single-pass only (no escalating retries) to keep runtime bounded
