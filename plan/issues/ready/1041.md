# #1041 — Multi-file module graph compilation

## Goal

The compiler currently has no module resolver. `src/import-resolver.ts:23` (`preprocessImports`) rewrites every `import` statement into a `declare const X: any` stub based on usage analysis. This is fine for single-file test262 inputs but blocks every npm-package stress test (#1031 lodash, #1032 axios, #1033 react, #1034 prettier), because none of them are single-file.

This issue is scoped intentionally narrowly: **do not build a real module graph compiler.** Instead, accept pre-bundled single-file input from `esbuild`/`rollup` and make sure the existing single-file pipeline handles the resulting bundle cleanly. A real graph compiler is a multi-month effort that can wait until the stress tests surface concrete pressure for it.

## Why

Cross-cutting gap identified in `plan/architecture/npm-stress-compiler-gaps.md` (section "Cross-cutting compiler gaps", item 1). All four stress tests name this as their biggest blocker. The pre-bundle workaround eliminates ~95% of the problem without committing to the real implementation.

## Approach

1. Add a helper script `scripts/bundle-for-wasm.ts` that wraps `esbuild --bundle --format=esm --platform=browser --target=es2020` with the right flags for our compiler (no top-level `await`, no dynamic import, no eval, strip `process.env.NODE_ENV` branches via `--define`).
2. Update `scripts/{lodash,axios,react,prettier}-stress.ts` harnesses to bundle first, then compile.
3. Audit `preprocessImports` to make sure bundled output (which only has `import 'node:...'` and similar host-only imports left) is handled correctly — no "declare const" leakage for user code.
4. Verify that deduping / hoisting performed by esbuild doesn't produce patterns the compiler currently chokes on (IIFE wrappers, `Object.defineProperty` for ESM exports, `__esModule` markers). Expect 1-2 follow-up issues for specific patterns esbuild emits.

## Acceptance criteria

- [ ] `scripts/bundle-for-wasm.ts` exists and produces a single-file ESM bundle from a package entry point
- [ ] Running the bundle through the compiler does not produce "declare const" stubs for symbols that the bundle actually defines
- [ ] Lodash `clamp` bundle compiles and runs (Tier 1 smoke test)
- [ ] Any compiler-side patterns surfaced by esbuild output are filed as follow-up issues

## Non-goals

- Real multi-file module graph compilation (explicitly deferred)
- Tree shaking beyond what esbuild already does
- Source map mapping across the bundle boundary
