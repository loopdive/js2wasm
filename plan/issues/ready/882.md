# #882 -- Test262 runner: 8 forks with inline compiler

## Problem

Current test262 runner uses 1 vitest fork with 4 compiler worker threads. The single fork grows from ~1GB to ~7GB over 48K tests because compiled Wasm modules accumulate in V8 and can't be freed without restarting the process. Runs take ~22 min (1300s) with cache disabled.

## Current architecture

```
vitest parent (370MB)
  └── fork 1 (peaks 7GB over 48K tests)
        ├── compiler worker thread 1 (warm TS Program ~200MB)
        ├── compiler worker thread 2
        ├── compiler worker thread 3
        └── compiler worker thread 4
```

- 1 fork, 4 compiler threads = 4 parallel compilations
- Fork runs all 48K tests sequentially, Wasm modules accumulate
- Total: ~7.5GB peak

## Proposed: Option A — 2 forks × 4 compiler threads

```
vitest parent (370MB)
  ├── fork 1 (24K tests, peaks ~4GB)
  │     └── 4 compiler worker threads
  └── fork 2 (24K tests, peaks ~4GB)
        └── 4 compiler worker threads
```

- 8 parallel compilations (same as current)
- Each fork: 24K tests → ~4GB peak each = ~8GB total
- Compile and execute can pipeline within each fork
- **Fastest option** — pipelining means the compiler thread compiles the next test while the current one executes

Changes:
- `vitest.config.ts`: `maxForks: 2`
- `--max-old-space-size`: increase to 4096 (each fork needs more heap)

## Proposed: Option B — 8 forks × 1 inline compiler

```
vitest parent (370MB)
  ├── fork 1 (6K tests, peaks ~1GB)
  │     └── inline compiler (no threads)
  ├── fork 2 ...
  └── fork 8 (6K tests, peaks ~1GB)
```

- 8 parallel compilations
- Each fork: 6K tests → ~1GB peak each = ~8GB total
- No IPC, no thread overhead — simplest architecture
- No pipelining — compile + execute are sequential per test
- **Smallest per-fork footprint** — less Wasm accumulation

Changes:
- `vitest.config.ts`: `maxForks: 8`, remove `--max-old-space-size` (default is fine for small forks)
- `test262-vitest.test.ts`: replace `CompilerPool` with direct `compile()` import
- Remove `scripts/compiler-pool.ts` dependency from test runner

## Recommendation

Try Option A first (simpler change — just config). If forks still bloat too much at 24K tests, try Option B.

## Acceptance criteria

- Full test262 run completes without OOM
- Peak RSS per fork stays under 4GB
- Total run time < 10 min (down from ~22 min)
- Pass count matches baseline (23,832)
- Memory monitor log shows improvement

## Test

Run with memory monitor: `pnpm run test:262`
Compare: total time, peak RSS, pass count vs current baseline.
