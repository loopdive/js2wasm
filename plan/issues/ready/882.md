# #882 -- Test262 runner: sharded parallel forks

## Problem

Current test262 runner uses 1 vitest fork with 4 compiler worker threads. The single fork grows from ~1GB to ~7GB over 48K tests because compiled Wasm modules accumulate in V8 and can't be freed without restarting the process. Runs take ~22 min with cache disabled. Dev agents can't run during test262 because the fork consumes all RAM.

## Current architecture

```
vitest parent (370MB)
  └── fork 1 (peaks 7GB over 48K tests, ~22 min)
        └── 4 compiler worker threads
```

## Proposed: sharded sequential batches with max parallel forks

```bash
# 10 cores available, run in 8 sequential shards
for shard in 1..8; do
  TEST262_WORKERS=10 npx vitest run --shard=$shard/8 tests/test262-vitest.test.ts
done
```

Each shard:
- **10 forks** (1 per core) running in parallel
- Each fork handles ~600 tests (48K / 8 shards / 10 forks)
- Each fork peaks at ~300MB (600 tests = minimal Wasm accumulation)
- **All forks die between shards** — memory fully reset

Per shard: 10 × 300MB + 400MB parent = ~3.4GB
Leaves ~10GB free for dev agents + system

Between shards: 0MB vitest — full memory available

**Total: ~3 min** (10x parallelism, 8 sequential shards)

## Implementation

### 1. Update `scripts/run-test262-vitest.sh`

Replace single vitest invocation with shard loop:

```bash
CORES=$(nproc)
SHARDS=8

for i in $(seq 1 $SHARDS); do
  echo "=== Shard $i/$SHARDS ==="
  TEST262_WORKERS=$CORES npx vitest run \
    --shard=$i/$SHARDS \
    tests/test262-vitest.test.ts \
    --reporter=verbose 2>&1 | tee -a /tmp/test262-vitest-run.log
done
```

### 2. Update `vitest.config.ts`

Remove fixed `maxForks`, use env var:
```ts
maxForks: parseInt(process.env.TEST262_WORKERS || String(require('os').cpus().length), 10),
execArgv: ['--max-old-space-size=512'],  // small heap per fork
```

### 3. Update `test262-vitest.test.ts`

Replace `CompilerPool` (4 worker threads) with direct `compile()` import:
- Each fork imports the compiler directly
- No worker threads, no IPC
- Simpler, less memory per fork

### 4. Merge shard results

Each shard writes its own JSONL. After all shards:
- Concatenate JSONL files
- Generate final report.json from combined results
- Update runs/index.json

## Key constraints

- Container: 16GB RAM
- Dev agents: ~700MB each, may be running during test262
- Must leave ~8GB free for 2-3 dev agents
- Therefore: max ~8GB for test262 at any point

## Acceptance criteria

- Full test262 run completes without OOM
- Peak RSS stays under ~4GB total for vitest
- Total run time < 5 min (down from ~22 min)
- Pass count matches baseline (23,832)
- Dev agents can run simultaneously without OOM
