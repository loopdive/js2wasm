# #873 -- Design dev branch protocol for agent merge workflow

## Problem

Current workflow: agents work on isolated worktree branches, tech lead cherry-picks/merges to main after completion. This causes:

1. **Merge conflicts** — 17 commits across 7 worktrees, multiple touching the same files. Cherry-picks fail, patches fail, manual resolution needed.
2. **Delayed integration** — work sits in worktrees for hours before merging, diverging from main
3. **Test file deletions** — patches accidentally delete files from other branches
4. **Stale bases** — agents work against old main, their fixes may not be compatible with other agents' changes

## Proposed: shared `dev` branch

Instead of isolated branches merging to main later:

1. Agents merge to `dev` branch frequently (after each commit, not after task completion)
2. `dev` branch is the integration branch — conflicts caught early when they're small
3. Tech lead periodically merges `dev` → `main` after running tests
4. Agents rebase from `dev` (not main) to stay current

## Decision: Option C — Agent rebases, ff-only merge (2026-03-29)

Upgraded from Option A after sprint-29 revealed that cherry-pick/restore workflows silently revert fixes (e.g. @vite-ignore lost in runtime.ts). Root cause: TTL restoring files with `git checkout HEAD --` after cherry-pick can revert unrelated changes.

## Protocol

### 1. Tech lead merges immediately after task completion
- When a dev agent signals completion, tech lead merges their branch to `main` within minutes
- Use `git merge --no-ff <branch>` (merge commit preserves history)
- Run equivalence tests after each merge: `npm test -- tests/equivalence.test.ts`
- If tests fail, revert the merge and notify the agent

### 2. Broadcast rebase after each merge
- After merging to `main`, tech lead broadcasts to all active agents: *"Main updated with #NNN, rebase before next commit: `git rebase main`"*
- Agents must `git rebase main` in their worktree before their next commit (not immediately — before next commit)

### 3. Conflict resolution: originating agent owns it
- If `git merge` has conflicts, tech lead does NOT resolve them
- Instead: abort the merge, notify the originating agent with the conflict details
- Agent rebases onto current `main`, resolves conflicts, re-signals completion
- This ensures the person who understands the code resolves the conflict

### 4. One merge at a time
- Tech lead processes completions sequentially (no parallel merges)
- First-signaled, first-merged — simple queue order
- If two agents finish simultaneously, merge one, broadcast rebase, then merge the other

### 5. No batching
- Never accumulate completed branches for a "merge session"
- Every completed task gets merged before the next task is dispatched to that agent
- This is the key behavioral change from previous workflow

## Path to Option C (future)

Once agents demonstrate reliable rebase/conflict resolution:
1. Agents rebase onto `main` themselves before signaling completion
2. Tech lead does fast-forward merge only (`git merge --ff-only`)
3. If ff-only fails, agent rebases again — tech lead never resolves conflicts
4. This removes tech lead as merge bottleneck entirely
