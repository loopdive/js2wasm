# #873 -- Design dev branch protocol for agent merge workflow

## Problem

Current workflow: agents work on isolated worktree branches, tech lead cherry-picks/merges to main after completion. This causes:

1. **Merge conflicts** — 17 commits across 7 worktrees, multiple touching the same files. Cherry-picks fail, patches fail, manual resolution needed.
2. **Delayed integration** — work sits in worktrees for hours before merging, diverging from main
3. **Test file deletions** — patches accidentally delete files from other branches
4. **Stale bases** — agents work against old main, their fixes may not be compatible with other agents' changes

## Proposed: shared `dev` branch

Instead of isolated branches merging to main later:

1. Agents merge to `dev` branch frequently (after each commit, not after task completion)
2. `dev` branch is the integration branch — conflicts caught early when they're small
3. Tech lead periodically merges `dev` → `main` after running tests
4. Agents rebase from `dev` (not main) to stay current

## Questions to resolve (PO + tech lead discussion)

- Should agents merge to `dev` themselves or should tech lead do it?
- How often should `dev` → `main` happen? After each task? After each wave?
- Should we use merge commits or fast-forward?
- How to handle broken `dev` — revert or fix forward?
- Should we run CI on `dev` automatically?
- Is there a simpler approach that doesn't need a separate branch?

## Acceptance criteria

- Documented protocol in CLAUDE.md and developer.md
- No more multi-hour merge sessions
- Conflicts caught within minutes, not hours
