# #1071 — for-of requires an array expression (non-array iterables in bundled JS)

## Problem

Compiling `node_modules/prettier/doc.mjs` emits two diagnostics:

```
for-of requires an array expression
for-of requires an array expression
```

These sites iterate over either a `Map` / `Set` / `Map.entries()` /
`.keys()` return value, or a hand-rolled iterator object (prettier's
doc-traverser uses a generator-like stack in places). Our codegen's
for-of path currently narrows to "the expression must be typed as an
array" and bails otherwise.

## Context

This gap is the broadest on the list: any real codebase that iterates over
a `Map`, `Set`, generator, or custom iterable hits it. Prettier is the
first stress test where we hit it twice in 1500 lines.

Our runtime already supports iteration via `Symbol.iterator` in host mode
(see `src/codegen/iteration.ts`), but the codegen check at the call site
is over-restrictive — it only whitelists `T[]` and `ReadonlyArray<T>`.

## ECMAScript spec reference

- [§14.7.5.6 ForIn/OfHeadEvaluation](https://tc39.es/ecma262/#sec-runtime-semantics-forinofheadevaluation) — step 6: calls GetIterator(exprValue) which works on any iterable, not just arrays
- [§7.4.3 GetIterator](https://tc39.es/ecma262/#sec-getiterator) — dispatches to @@iterator method on the value


## Acceptance criteria

- [ ] `for (const x of map.entries())` compiles in a test fixture
- [ ] `for (const x of someSet)` compiles in a test fixture
- [ ] `for (const x of generatorFn())` compiles in a test fixture

      expression with an iterable type, not just array types
