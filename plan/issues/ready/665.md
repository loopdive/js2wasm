# Issue #665: Native Wasm Date implementation

## Problem
~486 Date tests fail because there is no Date support at all. `new Date(...)` and Date methods like `.getTime()`, `.getFullYear()`, etc. are not compiled.

## Solution
Implement Date as a WasmGC struct with an i64 timestamp field (milliseconds since Unix epoch).

### Constructor forms:
- `new Date()` - timestamp 0 (no clock access in pure Wasm)
- `new Date(ms)` - explicit millisecond timestamp
- `new Date(y, m, d, h, min, s, ms)` - compute from components using Hinnant's days_from_civil

### Instance methods:
- `.getTime()` / `.valueOf()` - return timestamp as f64
- `.getFullYear()` / `.getUTCFullYear()` - decompose using civil_from_days
- `.getMonth()` / `.getUTCMonth()` - 0-indexed month
- `.getDate()` / `.getUTCDate()` - day of month
- `.getHours()` / `.getUTCHours()` - modular arithmetic on timestamp
- `.getMinutes()` / `.getUTCMinutes()`
- `.getSeconds()` / `.getUTCSeconds()`
- `.getMilliseconds()` / `.getUTCMilliseconds()`
- `.getDay()` / `.getUTCDay()` - day of week
- `.getTimezoneOffset()` - returns 0 (UTC)
- `.setTime(ms)` - update timestamp
- `.toISOString()` / `.toJSON()` - stub returning epoch string

### Static methods:
- `Date.now()` - returns 0 (no clock in pure Wasm)
- `Date.UTC(y, m, d, h, min, s, ms)` - compute UTC timestamp
- `Date.parse()` - stub returning NaN

## Implementation Summary

### What was done
- Added `__Date` struct type with single `(field $timestamp (mut i64))` field
- Implemented `ensureDateStruct`, `ensureDateCivilHelper` (Hinnant's civil_from_days), and `ensureDateDaysFromCivilHelper` (inverse) as pure Wasm helper functions
- Added `compileDateMethodCall` dispatcher for 20+ Date instance methods
- Added `new Date(...)` handling in `compileNewExpression` for 0, 1, and multi-arg forms
- Added `Date.now()`, `Date.UTC()`, `Date.parse()` static method handling
- Added Date to `ERROR_TYPES_SKIP` to prevent extern class registration
- Added Date type resolution in `resolveWasmType` to return struct ref instead of externref
- JS year 0-99 -> 1900-1999 quirk handled in multi-arg constructor
- All UTC getters are aliases for non-UTC getters (since we operate in UTC)

### What worked
- Hinnant's algorithm translates cleanly to i64 Wasm arithmetic
- The existing patterns for struct types and helper functions were straightforward to follow
- 20 test cases all pass, including round-trip validation against JS Date.UTC

### What didn't work initially
- Date methods were being intercepted by `isExternalDeclaredClass` -> `compileExternMethodCall` before reaching our handler; fixed by moving Date check before extern class dispatch
- Helper function locals were using ValType format instead of LocalDef format (need `{ name, type }`)
- Helper function used `exportName: undefined` instead of `exported: false`

### Files changed
- `src/codegen/expressions.ts` - Date constructor, method calls, helper functions
- `src/codegen/index.ts` - `resolveWasmType` Date handling, `ensureDateStructForCtx`, skip Date in extern class collection
- `tests/date-native.test.ts` - 20 test cases

### Tests now passing
- 20 new Date-specific tests covering constructors, getters, setters, static methods
