# #624 — Wasm struct type errors: struct.new/struct.get mismatches (1,092 CE)

## Status: review

1,092 tests fail with Wasm validation errors related to struct operations:
- `struct.new` with wrong number of fields
- `struct.get` on a reference that isn't the expected struct type
- Type mismatch between struct field type and value on stack

### Root cause
Class inheritance and struct type registration don't always produce compatible types. When a child class adds fields, the parent struct type doesn't match.

### Fix
Ensure struct type compatibility in class hierarchies. May need struct subtyping or ref.cast before struct operations.

## Complexity: M

## Implementation Summary

### What was done
Fixed the root cause of ~953 CE (struct.new "not enough arguments" errors) by preventing dynamic field addition to class struct types during property access compilation.

### Root cause analysis
Three locations in `src/codegen/expressions.ts` dynamically add fields to struct types when a property access encounters an unknown field. This is correct for anonymous/interface struct types, but catastrophic for class struct types because:

1. The constructor's `struct.new` only pushes default values for fields known at collection time
2. Dynamic field addition (e.g., from `C.prototype['x']` access) adds `prototype` or other TS-internal properties to the class struct after collection
3. The struct type definition now has more fields than the constructor supplies values for, causing "not enough arguments on the stack for struct.new"

### The fix
Added `!ctx.classSet.has(typeName)` guards to all three dynamic field addition sites in `expressions.ts`:
- Line ~6780: `compilePropertyLogicalAssignmentExternref` fallback
- Line ~7665: `compileCompoundAssignmentProperty` fallback
- Line ~15183: Property access fallback for unresolved fields

When the struct is a class type, the code now falls through to the externref-based fallback instead of corrupting the struct type.

### Files changed
- `src/codegen/expressions.ts` — three guard additions
- `tests/struct-type-errors.test.ts` — new test file with 6 test cases

### What worked
- The fix is minimal and surgical — only 3 lines changed
- Class struct types are now protected from dynamic mutation
- All 92 existing tests pass with no regressions

### What didn't
- The remaining ~139 CE from `local.set type mismatch` errors (e.g., var redeclaration with different object shapes) require a separate fix for struct type unification
- The `struct.get on externref` errors (74 CE) need ref.null → struct coercion
