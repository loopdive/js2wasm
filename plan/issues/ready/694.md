# Migrate test262 to vitest with per-test disk cache

## Problem
The existing test262 runner uses a custom standalone script. We want a vitest-native
test file that runs test262 tests with a per-test disk cache for compiled Wasm binaries,
so repeated runs skip recompilation for unchanged tests.

## Plan
1. Create `.test262-cache/` directory (add to .gitignore)
2. Create `tests/test262-vitest.test.ts` with disk-cached compilation
3. Hash compiler source files for cache invalidation
4. Start with first 3 categories as proof of concept (20 tests each)

## Implementation Summary

### What was done
- Created `tests/test262-vitest.test.ts` with per-test disk cache
- Cache stored in `.test262-cache/` (added to `.gitignore`)
- Cache key: MD5 of (wrapped source + compiler source hash)
- Compiler hash covers all codegen/*.ts files and runtime.ts
- Proof of concept: first 3 categories (arguments-object, asi, block-scope), 20 tests each = 60 tests
- All tests pass in vitest (conformance tracking, not gating -- errors caught and returned cleanly)
- Cache hit performance: ~35s cold -> ~7s warm (5x speedup)

### Files changed
- `tests/test262-vitest.test.ts` (new) -- vitest-native test262 runner with disk cache
- `.gitignore` -- added `.test262-cache/`
- `plan/issues/ready/694.md` (new) -- issue tracking

### Design decisions
- Used `buildImports` from `src/runtime.ts` (not the test helper) for correct import resolution
- Cached both `.wasm` binary and `.json` metadata (stringPool, imports) per test
- All errors (compile, instantiate, runtime) are caught -- no vitest failures from conformance issues
- `setExports` wired up for callback support
