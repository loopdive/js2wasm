# #694 — Evaluate vitest as test262 orchestration layer

## Status: open

### Question
Can we use vitest's parallel worker infrastructure instead of our custom worker pool?

### What vitest gives for free
- Memory-aware worker pool (vitest manages worker_threads)
- Per-test timeout with configurable retry
- Watch mode for dev iteration
- Custom reporters (could write JSONL reporter)
- Snapshot testing (could snapshot WAT output)
- Coverage integration
- Built-in concurrency control

### What we still need custom
- wrapTest() source transformation
- compile() → instantiate() → run() pipeline
- Negative test handling (phase detection)
- String pool / import resolution per test
- Source map enrichment

### Hybrid approach
Use vitest as orchestration, our code as the test body:
```typescript
// Each test262 file becomes a vitest test case
// vitest handles parallelism, timeouts, retries
// Our runTest262File does the actual work
```

### Better approach: vitest plugin with compile cache

Integrate compilation as a vitest plugin pre-step:

```typescript
// vitest.config.ts plugin
async buildStart() {
  // Pre-compile all test262 files, cache .wasm binaries to disk
  // Keyed by hash(source + compilerHash) — only recompile changed
  for (const file of findAllTest262Files()) {
    const hash = hashFile(file, compilerHash);
    if (existsSync(`.test262-cache/${hash}.wasm`)) continue; // cached
    const wrapped = wrapTest(readFileSync(file));
    const result = compile(wrapped.source, { skipWat: true });
    writeFileSync(`.test262-cache/${hash}.wasm`, result.binary);
  }
}
```

Each vitest test just loads the cached binary — no compilation:
```typescript
it(testFile, async () => {
  const binary = readFileSync(`.test262-cache/${hash}.wasm`);
  const { instance } = await WebAssembly.instantiate(binary, imports);
  const ret = instance.exports.test();
  expect(ret).toBe(1);
}, 90_000);
```

Benefits:
- **Re-runs are instant** (<1ms per test, just load+instantiate)
- Vitest manages workers, memory, crash recovery, retries
- Native vitest HTML/JSON/JUnit reporters
- Watch mode works (recompile only changed tests)
- `vitest --filter "Array"` for category filtering
- Snapshot WAT output for regression detection
- No custom worker pool code needed

### Recommendation
This is the right long-term architecture. Replaces #689-#692 with vitest's mature infrastructure. Implement after #693 (skipWat + cached TS host).
