# #1064 — DataView bridge: propagate RangeError / TypeError from native getX/setX calls

## Problem

Follow-up from #1056 (PR #90). The new DataView bridge in `src/runtime.ts` materializes a real
`DataView` over the `i32_byte` vec struct and calls the native `getUint16` / `setFloat32` / etc.
methods. When the native method throws (e.g. `RangeError` for out-of-range offset, `TypeError`
for detached buffer), the throw is either swallowed or returned as a value, so the test's
`assert.throws(RangeError, ...)` assertion fails.

## Evidence

PR #90 branch CI (`issue-1056-dataview-setters`) against post-#1056-merge baseline:

- **+76 improvements** (net-positive fix landed)
- **16 regressions**, 15 of which are the same bug class:

```
  11× DataView/prototype/get{Int8,Int16,Int32,Uint8,Uint16,Uint32,Float16,Float32,Float64,BigInt64,BigUint64}/index-is-out-of-range.js
   4× DataView/prototype/{getInt8,getUint8,setInt8,setUint8}/detached-buffer.js
   1× ArrayBuffer/prototype/slice/species.js  (unrelated, already-failing orthogonal test)
```

Sample failure message:

```
returned 6 — assert #5 at L48: assert.throws(RangeError, function() {
  sample.getUint16(1);
}, "1 + 2 > 2 (offset)");
```

The test calls `assert.throws(RangeError, fn)`. The runtime bridge executes `fn()` without a
try/catch around the native method invocation, so either:

- The throw is swallowed (bridge returns default), or
- The throw surfaces as a non-`RangeError` externref, failing `assert.throws`'s constructor check.

## Fix

In `src/runtime.ts` — `__extern_method_call` DataView branch (the one that matches
`/^(get|set)(Uint|Int|Float|BigInt|BigUint)(8|16|32|64)$/`):

1. Wrap the native `(get|set)Xxx()` call in `try { ... } catch (e) { throw e; }` so the throw
   propagates back through the externref boundary as a real error.
2. For `set*` methods, write bytes back via `__dv_byte_set` ONLY on success — do not write
   partial bytes if the native throws.
3. Verify the caller path converts the thrown externref back to a catchable Wasm exception
   (should already work via the standard Wasm exception bridge — confirm).

## Expected impact

~15 FAIL → PASS in `test/built-ins/DataView/prototype/*/index-is-out-of-range.js` and
`detached-buffer.js` buckets.
