# #629 — Destructuring in generator/class methods fails silently (2,444 FAIL)

## Status: review

2,444 tests fail with "returned 0" — these are destructuring patterns inside generator methods, async generator methods, and class method parameters. The destructuring compiles but produces wrong values at runtime.

### Common patterns
- `gen-meth-ary-ptrn-*` — generator method array destructuring
- `gen-meth-obj-ptrn-*` — generator method object destructuring
- `async-gen-meth-dflt-*` — async generator method default destructuring
- `cls-decl-gen-meth-*` — class declaration generator method patterns

### Root cause (actual)
Two independent bugs:

1. **Tuple struct not handled in `destructureParamArray`**: When untyped array destructuring parameters like `[x, y, z]` are used, TypeScript infers a tuple type `[any, any, any]`, which compiles to a Wasm struct with positional fields (`_0`, `_1`, `_2`). The existing `destructureParamArray` only handled vec array structs (with length + data fields), not tuple structs. When `getArrTypeIdxFromVec` returned -1 for tuple structs, the fallback just allocated locals with default values (null/0), never extracting actual array element values.

2. **Missing destructuring calls in standalone function declarations**: The function declaration compilation path in `statements.ts` called `emitDefaultParamInit` for parameter defaults but never called `destructureParamArray`/`destructureParamObject` for binding patterns. This affected both the no-captures path and the captures path.

### Fix
- `src/codegen/index.ts`: `destructureParamArray` now detects tuple structs (fields named `_0`, `_1`, ...) and extracts fields positionally via `struct.get` with proper null guards.
- `src/codegen/statements.ts`: Added `destructureParamArray`/`destructureParamObject` calls for standalone function declarations in both the no-captures and captures paths.
- `src/codegen/expressions.ts`: Added tuple struct branch in the arrow/function expression array destructuring path.
