# Issue #542: Negative test skip blocks 892 tests

892 test262 tests with `negative` metadata are being skipped unnecessarily.
These tests expect a parse/early/runtime error. Many should be attempted:
- If the compiler rejects the code (compile error), that's a pass
- If execution throws, that's a pass
- If execution succeeds without error, that's a fail

## Problem

Runtime negative tests (phase: runtime) go through `shouldSkip` which filters
them out for reasons like "uses eval", "uses with statement", etc. But since
these tests *expect* errors, we should try to compile and run them regardless.

Parse/early/resolution negative tests were already handled before `shouldSkip`,
but the `run-test262.ts` pre-filter was also incorrectly applying `shouldSkip`
to ALL negative tests.

## Implementation Summary

### What was done
Two changes to bypass `shouldSkip` for negative tests:

1. **`tests/test262-runner.ts` (`runTest262File`)**: Moved `shouldSkip` call after
   `isRuntimeNegative` is computed. Runtime negative tests now skip `shouldSkip`
   entirely and proceed to compilation/execution. If compilation fails, the
   existing handler at lines 2098-2103 reports `compile_error`. If execution
   throws/traps, it counts as a pass (expected behavior).

2. **`scripts/run-test262.ts` (pre-filter)**: Wrapped the `shouldSkip` call in a
   `!meta.negative` guard so all negative tests (parse/early/resolution/runtime)
   are forwarded to the worker for proper handling.

### What worked
- Clean separation: the `shouldSkip` bypass is minimal and targeted
- Existing runtime negative handling in `runTest262File` already correctly
  interprets compile errors and runtime traps -- no changes needed there

### Files changed
- `tests/test262-runner.ts` -- bypass shouldSkip for runtime negative tests
