# #617 — Wasm validation: not enough arguments for drop (109 CE)

## Status: review

109 tests fail with "not enough arguments on the stack for drop (need 1, got 0)". The compiler emits a `drop` instruction when the stack is empty.

This happens when an expression is expected to produce a value (for dropping) but the codegen path produces void instead. Likely in async function returns or conditional expressions where one branch is void.

## Complexity: S

## Implementation Summary

### Root cause
When calling async functions returning `Promise<void>`, the TS type checker reports `Promise<void>` as the return type (not `void`). The existing `isVoidType()` check returned `false` for `Promise<void>`, causing the codegen to believe the call pushed a value onto the stack. It then returned a non-null type, and callers emitted `drop` on an empty Wasm stack.

### What was done
1. Added `isEffectivelyVoidReturn()` helper that unwraps `Promise<T>` for functions in `ctx.asyncFunctions` before checking `isVoidType`.
2. Added `wasmFuncReturnsVoid()` helper that checks the actual Wasm function signature (ground truth) to determine if a `call` instruction produces a value.
3. Applied both checks at all 18+ call-return-type resolution points in `compileCallExpression` and related functions.
4. Added a safety net in `compileExpression` wrapper: if the last instruction is a `call` to a void Wasm function but the inner result claims a value was produced, correct the result to null.

### Files changed
- `src/codegen/expressions.ts` — added helpers, applied at all call-return-type sites
- `tests/drop-validation.test.ts` — 8 tests covering async void patterns
