# #1047 — Private class elements leak onto prototype (hasOwnProperty check fails)

## Problem

Private class elements defined with `#name` (or non-private methods placed into an internal slot) are appearing on the class prototype chain, so `Object.prototype.hasOwnProperty.call(C.prototype, "foo")` returns `true` when the spec requires `false`. The tests expect `foo` to be installed on instances (or on a private slot table) and not on `C.prototype`.

## Evidence from harvest

- **Test count:** 246 tests currently failing with this pattern
- **Top path buckets:**
  - `123 test/language/expressions/class/elements/*`
  - `123 test/language/statements/class/elements/*`
- **Top error messages:**
  - 12× `returned 4 — assert #3 at L111: assert(!Object.prototype.hasOwnProperty.call(C.prototype, "foo"))`
- **Sample test files:**
  - `test/language/expressions/class/elements/after-same-line-gen-literal-names.js`
  - `test/language/statements/class/elements/multiple-definitions-grammar-privatename-identifier-semantics-stringvalue.js`
  - `test/language/expressions/class/elements/after-same-line-static-gen-literal-names.js`

## Root cause hypothesis

Class element installation in `src/codegen/expressions.ts`/`src/codegen/statements.ts` attaches every class element to the prototype struct/field table uniformly, without distinguishing private-name storage from prototype-visible public properties. As a result, `hasOwnProperty.call(C.prototype, "foo")` observes the private field via our property table.

## Fix

Route private class elements through a private-name slot table (per-instance `Map`/struct field) rather than the public prototype descriptor set. Public methods/accessors stay on the prototype; private fields and private methods must be invisible to `hasOwnProperty` and `Object.getOwnPropertyNames` on the prototype.

## Expected impact

~246 FAIL, plus likely downstream unblocking for other `class/elements` private-name tests.

- src/codegen/statements.ts (class body walk)
