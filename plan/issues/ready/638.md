# #638 — Add reverse typeIdxToStructName map (12 O(N) to O(1))

## Status: review

12 occurrences of `for (const [name, idx] of ctx.structMap)` doing reverse lookups (find struct name by type index). Each was O(N) where N = number of struct types.

### Fix
Added `typeIdxToStructName: Map<number, string>` to CodegenContext, populated alongside every structMap.set call. Replaced all 12 reverse scans with O(1) map lookups.

## Complexity: XS

## Implementation Summary

### What was done
- Added `typeIdxToStructName: Map<number, string>` to the `CodegenContext` interface
- Initialized the map in both context creation sites (`generate` and `generateMultiModule`)
- Added `ctx.typeIdxToStructName.set(idx, name)` alongside all 9 `ctx.structMap.set(name, idx)` calls
- Replaced 12 O(N) reverse-lookup loops across 4 files with O(1) `ctx.typeIdxToStructName.get(idx)` calls

### Files changed
- `src/codegen/index.ts` — interface + init + 1 reverse lookup replaced + 8 set calls added
- `src/codegen/expressions.ts` — 4 reverse lookups replaced + 1 set call added
- `src/codegen/statements.ts` — 5 reverse lookups replaced
- `src/codegen/type-coercion.ts` — 2 reverse lookups replaced (complex blocks with toPrimitive/valueOf dispatch)
- `tests/equivalence/reverse-struct-map.test.ts` — new test file

### Tests
- All existing equivalence tests pass (basic-destructuring, for-of-array-destructuring, struct-field-index, array-of-structs, unary-plus-coercion)
- New test file with 3 cases covering object destructuring, nested destructuring, and property access
