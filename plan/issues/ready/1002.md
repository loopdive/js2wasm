# #1002 — RegExp js-host mode: finish Symbol protocol and remaining host-wrapper semantics

## Status: open

The js-host RegExp path is **partially done already**:

- construction works via host objects
- `RegExp.prototype.test()` works
- [#763](../done/763.md) completed major runtime-wrapper work for
  `exec`, `match`, `replace`, `split`, and `search`

But the host-mode story is still not “complete” enough to treat `#682` as a
single dual-mode issue.

## Remaining host-side gaps

1. `Symbol.match` / `Symbol.replace` / `Symbol.search` / `Symbol.split`
   protocol compliance
2. remaining `RegExp.prototype.exec()` edge cases
3. `lastIndex` and result-array behavior mismatches on tricky host-wrapper paths
4. cross-over behavior between native-string mode and host RegExp calls
5. any remaining `built-ins/RegExp` / string-with-RegExp failures that are
   specifically host-wrapper semantics, not missing standalone support

## Why this is separate from #682

Host-mode completion is a much smaller and more incremental problem:

- it leverages the existing JS engine for actual matching
- it mainly needs wrapper correctness and protocol compliance
- it should be solved independently of the much larger standalone-engine choice

Keeping it separate makes the roadmap clearer:

- `#1002`: finish js-host correctness
- [#682](./682.md): choose and integrate a standalone regex backend

## ECMAScript spec reference

- [§22.2.6.8 RegExp.prototype\[@@match\]](https://tc39.es/ecma262/#sec-regexp.prototype-@@match) — Symbol.match protocol dispatch
- [§22.2.6.11 RegExp.prototype\[@@replace\]](https://tc39.es/ecma262/#sec-regexp.prototype-@@replace) — Symbol.replace protocol dispatch
- [§22.2.6.12 RegExp.prototype\[@@search\]](https://tc39.es/ecma262/#sec-regexp.prototype-@@search) — Symbol.search protocol dispatch
- [§22.2.6.14 RegExp.prototype\[@@split\]](https://tc39.es/ecma262/#sec-regexp.prototype-@@split) — Symbol.split protocol dispatch


## Acceptance criteria

- host-mode RegExp support is documented as either complete or explicitly
  limited to named remaining semantics
- Symbol-protocol-based RegExp/string interactions are covered by tests
- the remaining host-wrapper failures are no longer mixed into the standalone
  engine planning
