# Property descriptor writable enforcement

## Problem
When `Object.defineProperty(obj, prop, { writable: false, value: v })` is compiled,
the property is set but subsequent assignments to that property silently succeed
instead of throwing a TypeError (as required by the spec in strict mode).

~1,000 test262 tests check property descriptor behavior; most check the writable bit.

## Approach
1. Add a `nonWritableProps` map to `CodegenContext` (`Map<string, Set<string>>`)
   keyed by struct name, values are non-writable field names.
2. In `compileObjectDefineProperty`, when descriptor contains `writable: false`,
   record the struct+field as non-writable.
3. At property assignment sites (struct.set), check the map and emit
   `throw new TypeError(...)` instead of allowing the write.

## Acceptance
- `Object.defineProperty(obj, "x", { value: 1, writable: false }); obj.x = 2;`
  throws TypeError at runtime
- Equivalence test covering writable enforcement passes
- No regression in existing tests
