# Issue #266: Unknown identifier -- scope resolution for multi-variable patterns

## Status: review

## Summary
~73 tests fail with "Unknown identifier: x; Unknown identifier: y; Unknown identifier: z" -- typically destructuring or multi-variable declarations where all variables fail to register in scope. This is often caused by unsupported destructuring patterns or variable declarations that the scope pass misses.

## Category
Sprint 4 / Group D

## Complexity: M

## Scope
- Audit variable scope registration for destructuring patterns (array, object, nested)
- Ensure all destructured variables are registered in the scope pass
- Handle rest elements in destructuring (`[a, ...rest] = arr`)
- Update scope analysis in `src/codegen/index.ts` or `src/codegen/statements.ts`

## Acceptance criteria
- Multi-variable destructuring patterns register all variables in scope
- At least 40 compile errors resolved

## Implementation Notes

### Root Cause
Two issues were found:

1. **For-loop initializer skipped destructuring**: `compileForStatement` in `statements.ts` only handled `ts.isIdentifier(decl.name)` for variable declarations in for-loop initializers. Array and object binding patterns (e.g., `for (const [x, y, z] = [1, 2, 3]; ...)`) were silently skipped, leaving the variables unregistered and uninitialized.

2. **Array destructuring didn't handle tuple structs**: `compileArrayDestructuring` only supported vec structs (array with length+data). Tuple literals like `[1, 2, 3]` produce tuple structs (fields `_0`, `_1`, `_2`), which were rejected with "vec data is not array" and fell through to `ensureBindingLocals` (allocating locals with zero defaults but no actual values).

### Changes
- **`src/codegen/statements.ts` - `compileForStatement`**: Added `compileObjectDestructuring` and `compileArrayDestructuring` branches for binding patterns in for-loop initializers, matching the existing logic in `compileVariableStatement`.

- **`src/codegen/statements.ts` - `compileArrayDestructuring`**: Added a tuple struct path that detects struct types with fields named `_0`, `_1`, etc. and reads fields directly via `struct.get` instead of going through the vec array path. Includes proper externref-to-f64 unboxing via `__unbox_number`.

- **`tests/equivalence.test.ts`**: Added two new tests:
  - "array destructuring in for-loop initializer" - verifies `for (const [x, y, z] = [1, 2, 3]; ...)` works correctly
  - "tuple array destructuring in variable declaration" - verifies `const [x, y, z] = [10, 20, 30]` works correctly

### Verification
- The original ~73 test262 failures from the report already compile successfully (many were fixed in previous sprints)
- The for-loop destructuring now properly initializes variables (previously defaulted to 0)
- No regressions in existing destructuring tests
