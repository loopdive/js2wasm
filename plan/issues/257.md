# Issue #257: Unsupported call expression -- method calls on returned values

## Status: review

## Summary
~303 tests fail with "Unsupported call expression" as the sole error. Many involve calling methods on return values of functions or on chained expressions (e.g., `foo().bar()`, `arr.map(...).filter(...)`). The codegen needs to handle call expressions where the callee is a member expression on a non-identifier base.

## Category
Sprint 4 / Group A

## Complexity: M

## Scope
- Identify the most common "Unsupported call expression" sub-patterns from test262 compile errors
- Add codegen support for method calls on function return values
- Handle chained method calls (e.g., `a.b().c()`)
- Update `compileCallExpression` in `src/codegen/expressions.ts`

## Acceptance criteria
- Method calls on function return values compile successfully
- Chained method calls compile successfully
- At least 50 compile errors resolved

## Implementation Notes

### Changes made in `src/codegen/expressions.ts`:

1. **Parenthesized/type-assertion unwrapping for call expressions**: Added a handler near the end of `compileCallExpression` that unwraps `(f)()`, `(x as T)()`, `(x!)()` patterns by stripping parentheses, `as` expressions, non-null assertions, and type assertions before recursively calling `compileCallExpression`.

2. **ElementAccessExpression to PropertyAccessExpression conversion**: Added handling at the top of the PropertyAccessExpression block to also match `obj["method"]()` patterns (ElementAccessExpression with string literal key). Creates a synthetic PropertyAccessExpression for uniform handling while preserving the original `expr` for type checker resolution via `getResolvedSignature`.

3. **String .concat() method fallback**: Modified the string method call handler to also check `ctx.funcMap.get(method)` as a fallback when `ctx.funcMap.get("string_" + method)` is not found. This handles methods like `concat` that are imported from `wasm:js-string` under their raw name rather than the `string_` prefix.

### Patterns fixed:
- `(f)()` - parenthesized function call
- `(x as T).method()` - method call on type assertion
- `obj["method"]()` - element access with string literal key as method call
- `"str".concat(other)` - string concat method (was looking for `string_concat` but import is `concat`)
- `foo().bar()` and `a.b().c()` - these already worked (type checker resolves receiver type from return type)

### Not fixed (out of scope):
- Method calls on `any`-typed objects (requires runtime dispatch)
- Dynamic property access calls `obj[variable]()` (requires runtime dispatch)
