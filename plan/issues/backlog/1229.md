# #1229 — `eval(literal)` / `new RegExp(literal)` should not re-compile per loop iteration

## Problem

Seven `compile_timeout` entries in the post-#1227 baseline are genuine
runtime infinite loops in our `eval` / `RegExp` shims, all sharing the
same shape: a `for` loop over the BMP codepoint range that compiles a
fresh regex on every iteration.

**Cluster 1 — `eval("/" + ... + "/")` over codepoints (5 tests):**

- `test/language/literals/regexp/S7.8.5_A1.1_T2.js`
- `test/language/literals/regexp/S7.8.5_A1.4_T2.js`
- `test/language/literals/regexp/S7.8.5_A2.1_T2.js`
- `test/language/literals/regexp/S7.8.5_A2.4_T2.js`
- `test/language/comments/S7.4_A6.js`

Representative pattern:

```js
for (var cu = 0; cu <= 0xffff; ++cu) {
  // (some filtering of cu values)
  var xx = "a\\" + String.fromCharCode(cu);
  try {
    var pattern = eval("/" + xx + "/");
    // assertion on pattern.source
  } catch (e) {
    // SyntaxError handling
  }
}
```

**Cluster 2 — `new RegExp(...)` over codepoints (2 tests):**

- `test/annexB/built-ins/RegExp/RegExp-leading-escape-BMP.js`
- `test/annexB/built-ins/RegExp/RegExp-trailing-escape-BMP.js`

Representative pattern: same shape but `new RegExp(...)` instead of
`eval("/.../")`.

In V8 each iteration is microseconds (parse + compile a tiny regex,
test, discard). On js2wasm each iteration appears to pay a much larger
fixed cost — likely IPC to the `__eval` host import (which spins up the
full TS+codegen+wasm-instantiate pipeline) per call. 65,536 iterations
× even ~50 ms each = an hour of wall-clock; in practice we hit the
30 s pool ceiling on the first few hundred iterations.

These tests aren't strictly compiler bugs — they're tests of the
*language-feature surface area*. They walk every BMP codepoint and
sanity-check that we accept / reject the same regex chars as the spec
mandates. They WILL run slowly even on a fast implementation. But the
order-of-magnitude penalty here is large enough that we cannot
reasonably converge on these tests today.

## Implementation plan

The fix has two halves; either alone makes the tests viable, both
together is ideal.

### Half 1 — host-side `eval(literal)` cache

`src/runtime.ts` (the `__eval` host import) currently re-compiles its
input every call. For a literal-only argument, the result is a pure
function of the input string. Add an LRU cache keyed on the source
text → compiled `(binary, exports)` tuple:

```ts
const evalCache = new LRU<string, { instance: WebAssembly.Instance; exports: any }>(
  /* maxEntries */ 256,
);
async function __eval_impl(src: string) {
  const hit = evalCache.get(src);
  if (hit) return hit.exports.run();
  const compiled = await compile(src, ...);
  const inst = await WebAssembly.instantiate(compiled.binary, ...);
  evalCache.set(src, { instance: inst, exports: inst.exports });
  return inst.exports.run();
}
```

For the 65k-iteration tests where each iteration has a *different*
input string, the cache hit rate is 0% and this alone won't help.

### Half 2 — host-side `new RegExp(source, flags)` cache

The `__new_regexp` host import (or wherever we route `new RegExp(...)`)
should cache the parsed regex AST or compiled IR, keyed on
`(source, flags)`. Re-parsing the same regex source millions of times
across an entire test262 run is wasted work.

For the 65k-iteration tests this also wouldn't help directly (each
iteration's regex is unique). But combined with…

### Half 3 — peephole: `eval("/" + str_literal_or_const + "/")` → `new RegExp(str_literal_or_const)`

When the AST for `eval(...)` is structurally `eval("/" + X + "/")` —
i.e. a fixed prefix `/` and suffix `/` with `X` either a literal or
local constant — the codegen could lower it directly to a `RegExp`
constructor call without going through the eval pipeline at all. This
collapses the per-iteration cost to one host call (regex parse+compile)
instead of two (eval pipeline + regex parse+compile).

`src/codegen/expressions.ts` `compileCallExpression` already specialises
known-name calls (e.g. `Math.sin`, `parseInt`); add an `eval` path that
detects the `"/" + X + "/"` literal-template and rewrites.

## Acceptance criteria

1. `eval(literal)` host import has an LRU cache, hit rate ≥80% on a
   representative `eval`-heavy benchmark (a short script that calls
   `eval("1+1")` in a loop). Half 1 unblocks programs that re-eval the
   same code many times even if the 7 target tests don't benefit.
2. `new RegExp(source, flags)` cached similarly. Half 2 should make
   `RegExp` constructor calls in normal code substantially cheaper.
3. The 7 target tests run to completion within the 30 s pool ceiling,
   producing their actual `pass`/`fail` outcome (likely `pass` for the
   regexp-literal tests; the `comments/S7.4_A6.js` test may surface a
   real correctness issue that becomes visible only once we can
   actually run it).
4. No regression in `tests/equivalence/` eval tests.
5. CI test262 `compile_timeout` count drops by 7 (clusters 1 + 2)
   without new regressions in `eval` / `RegExp` paths.

## Out of scope

- Compiler-side eager `eval` lowering for non-literal arguments — that
  needs the eval-as-eval-or-direct-invocation distinction and is
  considerably more work.
- General eval performance work beyond the cache (e.g. avoiding the
  full TS pipeline per call). Tracked separately if needed.
