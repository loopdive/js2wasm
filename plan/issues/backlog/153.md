# #153 — Iterator protocol for destructuring and for-of

## Status: open

## Problem
39 test262 compile errors: "Type 'unknown' must have a '[Symbol.iterator]()' method that returns an iterator." + 28 similar for `Type 'void'`. Destructuring and for-of on non-array types fails because the compiler requires the Symbol.iterator protocol.

## Fix
For known iterable types (arrays, strings, generators), ensure the iterator interface is recognized even in allowJs mode where TypeScript can't infer it. For generators, the return type already implements the iterator protocol.

## Tests blocked
~67 compile errors

## Complexity: M

## Root Cause Analysis (from agent investigation 2026-03-13)

The core bug is in `addUnionImports` index shifting. When `compileForOfIterator` (statements.ts:2283) compiles a for-of loop over a generator:

1. Compiles iterable expression into `fctx.body`, emitting `call <genFuncIdx>`
2. Swaps `fctx.body` to fresh `[]` for loop body (`savedBody = fctx.body; fctx.body = []`)
3. Loop body compilation may trigger `addUnionImports` (e.g. unboxing externref to f64)
4. `addUnionImports` (index.ts:5332) shifts function indices in `ctx.currentFunc.body` — but that's the **inner** loop body, not the **outer** `savedBody` containing the `call gen()` instruction
5. `savedBody` retains stale function indices → Wasm validation failure

### Fix Plan

1. Add `savedBodies: Instr[][]` to `CodegenContext` (index.ts ~line 233), initialize to `[]`
2. In `addUnionImports`, after shifting `ctx.currentFunc.body`, also shift all `ctx.savedBodies`
3. In every savedBody swap site (statements.ts for-of, expressions.ts logical ops, etc.), push/pop to `ctx.savedBodies`
4. This is a wide-reaching change — many savedBody sites across expressions.ts, statements.ts, index.ts
