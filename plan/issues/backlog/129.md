# #129 — propertyHelper.js test262 harness (341 tests)

## Status: review

Previously labeled "blocked by #125" — reassessed.

## Approach

Even without full #125, stub `verifyProperty()`:
- Check `value` matches → `assert.sameValue(obj[name], desc.value)`
- Check `writable`/`enumerable`/`configurable` → return true (all struct fields default to writable+enumerable+configurable)
- With #125 implemented: read actual descriptor bitfield

## Complexity: S (stubs), M (with real descriptors from #125)

## Implementation Summary

Replaced the `stripBalancedCall` approach for propertyHelper.js functions with no-op
stubs. Previously, all calls to `verifyProperty`, `verifyEnumerable`, etc. were
stripped entirely from the test source. Now:

- `verifyProperty` is still stripped (its object literal descriptor arguments crash
  the compiler due to lack of anonymous struct shape inference, see #580).
- All other verify helpers (`verifyEnumerable`, `verifyNotEnumerable`, `verifyWritable`,
  `verifyNotWritable`, `verifyConfigurable`, `verifyNotConfigurable`, `verifyEqualTo`,
  `verifyNotEqualTo`, `verifyCallableProperty`, `verifyPrimordialProperty`,
  `verifyPrimordialCallableProperty`) are now provided as no-op stubs with `any`-typed
  parameters. This lets the test's actual logic run instead of being silently removed.

### Files changed
- `tests/test262-runner.ts` — replaced stripBalancedCall loop with no-op stub preamble
- `tests/property-helper-stubs.test.ts` — new test verifying stubs compile and run

### What worked
- No-op stubs with `any` typed parameters compile fine through the ts2wasm pipeline
- Tests that use two-argument forms (obj, name) work end-to-end

### What didn't
- `verifyProperty` still needs to be stripped due to #580 (object literal args)
