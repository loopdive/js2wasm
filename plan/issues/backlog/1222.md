# #1222 — Premature i32 specialization for `let s = 0` accumulators silently saturates

## Problem

The compiler infers `let s = 0` as `i32` based on the integer-literal initializer
(see #595's `detectI32LoopVar` and similar shape inference for plain `let`).
When the body then performs `s = s + i` with both operands i32-typed, the
compiler routes the `+` through f64 (correct JS semantics: `number + number`
is f64) and stores the result back via `i32.trunc_sat_f64_s`.

The trunc_sat **saturates** rather than wrapping or upgrading the local to
f64. Once the running value exceeds 2³¹−1 the accumulator is pinned at
`2147483647` forever and every subsequent iteration is a no-op against the
saturation ceiling.

This is a soundness bug, not just a perf issue: the program returns the wrong
value with no diagnostic.

## Repro

```ts
// playground/examples/benchmarks/loop.ts (pre-fix)
export function bench_loop(): number {
  let s = 0;
  for (let i = 0; i < 1000000; i++) s = s + i;
  return s;
}
```

Expected: `499999500000` (matches Node/V8).
Actual:   `2147483647` (saturated i32.MAX).

WAT (current):

```wat
(func $bench_loop (result f64)
  (local $s i32)
  (local $i i32)
  ...
  (loop
    ...
    local.get 0
    f64.convert_i32_s
    local.get 1
    f64.convert_i32_s
    f64.add
    i32.trunc_sat_f64_s   ;; <-- silent saturation here
    local.set 0
    ...
  ))
```

The same pattern triggers any time a `let x = INT_LITERAL` accumulator is
written through an arithmetic expression that the compiler keeps in f64 but
whose target slot was shape-inferred to i32.

## Why this happens

- `#595` taught the compiler to allocate i32 locals for loop counters that
  match a narrow pattern.
- The accumulator path here goes through the same i32-local allocation
  because `let s = 0` looks integer-shaped at declaration time.
- The codegen for `s = (f64 expression)` falls back to
  `i32.trunc_sat_f64_s`, which is the wrong coercion when the produced
  value can exceed i32 range. Either the local should be re-typed to f64,
  or the store should detect potential overflow and widen.

## Acceptance criteria

1. The repro above returns `499999500000` (or matches V8 exactly) when
   compiled in default mode.
2. The fix must not regress #595 — bounded loop counters should stay i32.
3. Add a differential-test program (under `tests/differential/corpus/`)
   that sums `0..1_000_000` and asserts the V8 result.
4. Add a focused unit test that compiles `let s = 0; for (...) s = s + i;`
   and asserts either i32-with-overflow-check **or** a widened f64 local —
   pick one and document the chosen rule.
5. test262 net delta must be ≥ 0.

## Implementation sketch

Two viable routes — pick one:

**A. Conservative widening.** When shape inference assigns a local to i32
because of an integer-literal init but the only writes to it come from
arithmetic expressions that the compiler emits in f64, demote the local to
f64. The trunc_sat goes away entirely and the local matches the producing
expression.

**B. Sound i32 narrowing.** Keep the local i32 only when the compiler can
prove every assignment fits in i32 — i.e. the RHS is itself a chain of
i32-producing ops (`i32.add`, `i32.and`, `i32.or`, `| 0` / `& mask`
patterns, etc.). For mixed-mode RHS, fall back to (A).

Option (B) preserves the perf win from #595 for `(s + i) | 0` style code
(where the `| 0` already proves i32-soundness) and is the asm.js-tested
heuristic. Option (A) is simpler and probably easier to land first.

## Related

  left behind).
- #1166 — closed-world integer specialization (different scope: narrowing

  heuristics that option B would build on).

## Workaround in the meantime

Source-level: write `s = (s + i) | 0` to make the i32 wrap explicit. This
matches `ToInt32` semantics (modular wrap) and the compiler emits a single
`i32.add` per iteration. The playground `bench_loop` example was switched
to this form on 2026-05-01 so the displayed result is correct.
