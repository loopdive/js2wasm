# #1234 — `Array.prototype.{unshift,reverse,forEach,…}` should iterate over defined properties for non-Array receivers

## Problem

Two `compile_timeout` entries in the post-#1227 baseline are genuine
runtime infinite loops in our `Array.prototype` shims:

- `test/built-ins/Array/prototype/unshift/length-near-integer-limit.js`
- `test/built-ins/Array/prototype/reverse/length-exceeding-integer-limit-with-object.js`

Both build an array-like object with `length: 2 ** 53 - 2` (≈ 9 × 10¹⁵)
and a sparse set of defined indexed properties — typically a getter at
some index that throws to short-circuit the iteration:

```js
var arrayLike = {
  get "9007199254740986" () { throw new StopUnshift(); },
  "9007199254740987": "9007199254740987",
  // ... a handful of more indices, mostly holes
  length: 2 ** 53 - 2
};
assert.throws(StopUnshift, function() {
  Array.prototype.unshift.call(arrayLike, null);
});
```

Per the spec (§22.1.3.34 ArrayPrototype.unshift), the algorithm walks
`for (k = len - 1; k >= 0; k--) { from = ToString(k); if (HasProperty(O, from)) ... }`.
**V8 short-circuits via the getter at index `9007199254740986` on the very
first iteration** (it iterates from `length - 1` downwards; the first
`HasProperty` check at that index trips the getter, which throws and
unwinds). Our shim, however, walks every integer index from
`length - 1 ≈ 9 × 10¹⁵` toward zero and never reaches the getter in any
finite time.

This same shape applies to a 5.3 s "honourable mention" outlier:
`test/built-ins/Array/prototype/forEach/15.4.4.18-7-c-ii-1.js` puts a
single value at index 999,999 of a 6-element array and calls `forEach` —
our shim visits all 1,000,000 indices instead of only the seven
defined ones. Same fix pattern, same shim family.

## Implementation plan

`src/runtime.ts` (or wherever the `Array.prototype` shims live — likely a
`__array_unshift_call`, `__array_reverse_call`, etc. host import) needs
to switch from index-range iteration to defined-property iteration when
the receiver is **not a real `Array`**. The spec actually mandates
`HasProperty` per index in either case, but the iteration order can be
short-circuited:

1. **Real-array receiver** (`receiver instanceof Array`): keep current
   `[0, length)` iteration — these always have densely-keyed integer
   indices and short-circuiting buys nothing.

2. **Non-array receiver** (any other `O`): collect the integer-keyed
   own (and prototype-chain, depending on op) properties of `O` into a
   sorted array, then iterate that — `O(defined-props × log)` rather
   than `O(length)`. For `unshift` / `reverse`, iterate in the spec's
   prescribed order (descending or paired-swap respectively); for
   `forEach`, ascending.

Concretely, replace the inner loop shape:

```ts
// before — pseudocode
for (let k = len - 1; k >= 0; k--) {
  const fromKey = String(k);
  if (HasProperty(O, fromKey)) {
    const fromValue = Get(O, fromKey);
    Set(O, String(k + argCount - 1), fromValue, true);
  } else {
    DeletePropertyOrThrow(O, String(k + argCount - 1));
  }
}
```

with:

```ts
// after — pseudocode
const indices = collectIntegerIndexedKeys(O); // sorted ascending
for (let i = indices.length - 1; i >= 0; i--) {
  const k = indices[i];
  if (k >= len) continue;
  const fromKey = String(k);
  // … same body, but only running on actually-defined indices
  // Holes still need DeletePropertyOrThrow, but that's also keyed by `k`.
}
```

`collectIntegerIndexedKeys(O)` should walk `O`'s own enumerable
property names (and getter-installed accessors), filter by
`/^(?:0|[1-9]\d*)$/` and `< 2^53 - 1`, parse to numbers, sort. For
sparse arrays this is `O(defined)` rather than `O(length)`.

The "hole" case (`!HasProperty`) in the spec algorithm requires
`DeletePropertyOrThrow`. With defined-property iteration this only
runs at indices that *are* defined, so the hole semantics for the
*destination* index (which may itself be a hole) need to be checked
explicitly. Test the existing forEach/filter/map test262 tests to
confirm the rewrite preserves hole-skipping semantics.
