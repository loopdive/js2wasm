# Issue #720: IIFE return type mismatch causes Wasm validation errors

## Problem

When an IIFE (Immediately Invoked Function Expression) returns a different type
than the enclosing function, the compiler emitted invalid Wasm. Specifically:

- An IIFE returning `boolean` (i32) inside a function returning `number` (f64)
  would cause: `local.set[0] expected type i32, found f64.convert_i32_s of type f64`

This happened because `compileReturnStatement` coerces the return value to
`fctx.returnType`, which was the **outer** function's return type (f64), not the
IIFE's own return type (i32). The coerced f64 value was then stored into the
IIFE's i32-typed result local, causing a Wasm validation failure.

A secondary issue: tail-call optimization (`return_call`) merged call+return
into a single instruction inside IIFE bodies, preventing `patchReturns` from
finding and replacing the `return` instruction.

## Root Cause

In `compileExpression` for IIFE handling, the IIFE body was compiled with the
outer function's `fctx.returnType` still active. Return statements inside the
IIFE would coerce to the outer type instead of the IIFE's own return type.

## Fix

1. **Save/restore `fctx.returnType`** around IIFE body compilation, temporarily
   setting it to the IIFE's own return type (`iifeWasmRetType`).

2. **Handle `return_call` / `return_call_ref`** in `patchReturns`: undo the
   tail-call optimization by converting back to `call` + `local.set` + `br`.

## Impact

- 39 test262 compile errors fixed (all `language/function-code/10.4.3-*` tests)
- Pattern: `local.set[0] expected type i32, found f64.convert_i32_s of type f64`

## Implementation Summary

### What was done
- Modified IIFE compilation in `src/codegen/expressions.ts` to save/restore
  `fctx.returnType` around IIFE body compilation
- Extended `patchReturns` to handle `return_call` and `return_call_ref`
  instructions (undo tail-call optimization inside IIFEs)
- Added 2 equivalence tests for the specific pattern

### Files changed
- `src/codegen/expressions.ts` — IIFE return type save/restore + patchReturns fix
- `tests/equivalence/iife-and-call-expressions.test.ts` — 2 new test cases

### Tests now passing
- 70/70 IIFE equivalence tests (was 68/68, added 2)
- 39 test262 compile errors eliminated
