# Issue #287: Generator return in nested contexts uses incorrect br depth

## Problem

When a generator function contains a `return` statement inside nested constructs
(for loops, while loops, if/else, switch, labeled blocks, try/catch), the emitted
Wasm `br` instruction used `fctx.blockDepth` as its depth. However, `blockDepth`
only tracked the generator body block nesting (depth 1) and was NOT adjusted when
loop/block constructs added additional nesting levels via the savedBody/swap pattern.

This caused `return` inside e.g. a for loop + if statement to emit `br 1` instead
of the correct `br 4`, resulting in the return only breaking out of the if block
instead of the entire generator body.

## Root Cause

The for loop compiles body instructions into a separate array (savedBody pattern),
then assembles them inside a `block { loop { block { BODY } } }` structure (3 levels).
The `breakStack` entries are adjusted by +3/-3 for this nesting, but `blockDepth`
(used by generator return) was not adjusted.

## Fix

Added `generatorReturnDepth?: number` field to `FunctionContext`. This field:

1. Is initialized to `0` before the generator body block is compiled
2. Is adjusted by the same deltas as `breakStack` entries in all 20 loop/block
   constructs in `statements.ts` (for, while, do-while, if, switch, labeled, try)
3. Is used by `compileReturnStatement` instead of `blockDepth` when inside a generator
4. Is reset to `undefined` after generator body compilation

## Implementation Summary

### What was done
- Added `generatorReturnDepth?: number` field to `FunctionContext` interface in `index.ts`
- Set `generatorReturnDepth = 0` before generator body compilation, `undefined` after
- Added 20 adjustment lines in `statements.ts` (one per breakStack/continueStack pair)
- Changed `compileReturnStatement` to use `generatorReturnDepth ?? blockDepth`

### Files changed
- `src/codegen/index.ts` (6 lines added)
- `src/codegen/statements.ts` (27 lines added, 2 modified)
- `tests/issue-287.test.ts` (new file, 5 tests)
