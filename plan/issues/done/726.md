# #726 -- TypeError regression: ref.cast guard returns ref.null for valid objects (1,948 tests)

## Status: done

## Problem

The combination of #695 (TypeError throws for null/undefined property access) and #706 (guard all ref.cast in coerceType) caused a regression of 1,948 tests that previously passed.

The root cause: #706 replaced unguarded `ref.cast` with guarded patterns that return `ref.null` when the cast fails. But many valid objects fail `ref.test` not because they are null, but because they are a different struct type than expected (e.g., TypedArray wrapper vs plain struct). The `ref.null` result then hits the #695 null guard, which either throws TypeError or returns a default value -- both wrong for a valid non-null object.

### Regression breakdown by test category

| Category | Regressed tests |
|----------|----------------|
| TypedArray | 557 |
| Temporal | 307 |
| TypedArrayConstructors | 265 |
| Object | 148 |
| Other | 671 |

### How the bug manifests

1. Code accesses `obj.prop` where `obj` is a valid object
2. Compiler emits `ref.cast $ExpectedStruct` to downcast
3. #706 guard: `ref.test $ExpectedStruct` returns false (wrong struct type, but object is valid)
4. Guard pushes `ref.null $ExpectedStruct`
5. #695 null guard sees null, throws TypeError or returns default
6. Test fails -- expected the property value, got TypeError or wrong default

### Fix approach

When `ref.cast` guard fails (object is non-null but wrong type), the code should NOT return `ref.null`. Instead, it should fall through to the `__extern_get(obj, propName)` host fallback path, which can dynamically resolve the property on any object type.

This requires:
1. **Pass `propName` through more call sites** in `emitNullGuardedStructGet` and `emitExternrefToStructGet` so the fallback path knows which property to look up
2. **In the guard's else branch**, instead of `ref.null`, emit `extern.convert_any` + `__extern_get(obj, propName)` to do dynamic property lookup
3. **Distinguish null from type-mismatch**: null should throw TypeError (#695), type-mismatch should use dynamic fallback

### Key code paths to fix

In `src/codegen/expressions.ts`:
- `emitNullGuardedStructGet`: the else branch after `ref.test` currently pushes `ref.null` -- needs to route to extern_get
- `emitExternrefToStructGet`: the type-mismatch branch currently returns default -- needs extern_get fallback
- All call sites that invoke these functions need to pass the property name string

## Implementation Summary

### What was done

The original issue description assumed `__extern_get` would fix the problem, but investigation revealed that `__extern_get` cannot read WasmGC struct fields because they are opaque to JS (the JS host sees an empty object). The real fix required **multi-struct dispatch**: when a property access encounters an object whose struct type does not match the expected type, try all other struct types that have a field with the same name.

Three code paths were modified in `src/codegen/expressions.ts`:

1. **`emitNullGuardedStructGet`** (called when compiled object is ref_null): Instead of the broken `extern.convert_any` + `emitExternrefToStructGet` path (which produced null externref for ref.null values), the function now widens the value to anyref and uses `ref.test` against the primary struct type, then cascades through all alternative struct types that have the same field name. Only returns a default value if no struct type matches.

2. **`emitExternrefToStructGet`** (called when compiled object is externref): The simple path (no propName or ref-type fields) now includes multi-struct dispatch in the else branch, trying alternative struct types before returning defaults.

3. **Externref property access fallback** (the `isExternObj` code path around line 18964): This was the primary cause of the test failures. When an `any`-typed variable holds a WasmGC struct (converted to externref via `extern.convert_any`), `__extern_get` returns undefined because JS cannot see struct fields. The fix adds a `findAlternateStructsForField` check before the `__extern_get` call: convert the externref to anyref, try `ref.test` against each candidate struct type, and use `struct.get` if a match is found. Only falls back to `__extern_get` if no struct matches (i.e., the object is a genuine JS externref).

A helper function `findAlternateStructsForField(ctx, propName, excludeTypeIdx)` was added to scan `ctx.structFields` for all struct types that have a field with the given name.

### What worked

- Multi-struct dispatch correctly handles the case where runtime struct types differ from compile-time expectations
- The cascading `ref.test` chain is efficient (short-circuits on first match)
- No regressions: equivalence tests went from 41 failures to 37 (4 tests fixed as bonus)

### What didn't work

- The original approach (using `extern.convert_any` to convert ref.null to externref and then `__extern_get`) does not work because `extern.convert_any` on `ref.null` produces null externref, losing the original object
- `__extern_get` on WasmGC structs returns undefined because JS cannot see GC struct fields

### Files changed

- `src/codegen/expressions.ts`: added `findAlternateStructsForField`, rewrote `emitNullGuardedStructGet` with multi-struct dispatch, rewrote `emitExternrefToStructGet` simple path with multi-struct dispatch, added struct dispatch before `__extern_get` in externref property access path
- `tests/equivalence/issue-726-refcast-regression.test.ts`: new regression test (3 cases)
