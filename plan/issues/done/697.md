# #697 -- Struct type errors for non-class structs (813 CE)

## Status: done

813 tests fail with struct.new/struct.get/struct.set type errors. #624 fixed class structs but non-class structs (anonymous, interface, widened) still have mismatches.

## Complexity: M

## Implementation Summary

### Root cause analysis
The actual errors fell into several categories:

1. **ref vs ref_null type mismatch in if/block types** (largest bucket): The `widenNonDefaultableTypes` pass in `compiler.ts` widens all function parameter types, return types, locals, and globals from `ref` to `ref_null`. However, it did NOT widen block types inside function bodies (`if`, `block`, `loop` block types). After widening, `local.get` of a parameter produces `ref_null` (because the param type was widened), but the `if` block still declares result type `ref` (non-null). V8 correctly rejects this as a type error.

2. **Any-typed operand coercion** (~27 CE in equivalence tests): The `compileAnyBinaryDispatch` function compiled operands of `any + any` expressions without specifying the expected `ref_null $AnyValue` type, so `any`-typed variables produced `externref` instead of the required `ref_null $AnyValue` for the `__any_add` helper call.

3. **Non-null return types on helpers with if blocks**: The `__any_add`, `__any_sub`, `__any_mul`, `__any_mod`, `__any_div`, `__any_neg` helpers declared return type `ref $AnyValue` (non-null) and used `if` blocks with non-null types, but V8 inferred the result as nullable due to the parameter type mismatch described in (1).

### What was done
1. **`src/compiler.ts`**: Added `widenBlockTypesInBody()` to `widenNonDefaultableTypes()` that recursively walks function bodies and widens `ref` to `ref_null` in all block types (`if`, `block`, `loop`, `try` catch bodies). This is the main fix -- it ensures consistency between widened param/return types and block type declarations.

2. **`src/codegen/index.ts`**: Changed `__any_add`, `__any_sub`, `__any_mul`, `__any_div`, `__any_mod`, `__any_neg` helpers to use `anyRefNull` (nullable) return types and `if` block types instead of `anyRef` (non-null). This is safer and matches the actual semantics.

3. **`src/codegen/expressions.ts`**: Fixed `compileAnyBinaryDispatch` to pass `ref_null $AnyValue` as the expected type when compiling operands, ensuring proper coercion from externref to AnyValue. Also updated the return type from `ref` to `ref_null` to match the helper signatures.

### Results
- Equivalence tests: 59 failed -> 30 failed (29 fewer failures, 29 more passing)
- All remaining failures are pre-existing stack balance issues or unrelated type coercion bugs

### Files changed
- `src/compiler.ts` -- added block type widening in `widenNonDefaultableTypes`
- `src/codegen/index.ts` -- changed any-helper return types to nullable
- `src/codegen/expressions.ts` -- fixed operand coercion in `compileAnyBinaryDispatch`, updated return type
