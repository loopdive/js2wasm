# #698 --- Call type mismatch residual (1,044 CE)

## Status: done

1,044 tests fail with call/call_ref type mismatches. Up from 609 --- more tests attempted with SKIP_DISABLED. #659 fixed closure args but other dispatch paths remain: method calls through externref, indirect dispatch, super calls.

## Complexity: M

## Implementation Summary

### What was done
Added receiver/self parameter coercion at multiple call dispatch paths in expressions.ts. When calling a method, the receiver (self/this) argument must match the callee's declared first parameter type. Previously, only an externref-to-struct conversion was handled. Now, arbitrary type mismatches are coerced using the existing `coerceType` infrastructure.

### Key changes

1. **Class method calls** (property access): Get `paramTypes` early via `getFuncParamTypes`, then coerce receiver to match `paramTypes[0]` after compilation. Skip `ref_null -> ref` coercion before the null-guard (the guard handles that via `ref.as_non_null` in the non-null branch).

2. **Struct method calls**: Same pattern as class method calls.

3. **Element access method calls** (`obj['method']()`): Pass `paramTypes[0]` as expectedType when compiling receiver, plus post-compilation coercion.

4. **`.call()` / `.apply()` paths**: Coerce thisArg to match callee's self parameter type.

5. **Super method calls** (`compileSuperMethodCall`, `compileSuperElementMethodCall`): Coerce `this` local to match parent method's self parameter type using `getLocalType` + `coerceType`.

6. **Super property accessors** (`compileSuperPropertyAccess`): Same pattern for getters.

### What worked
- Passing expectedType hints to `compileExpression` for receivers catches most mismatches at the expression compilation level.
- Post-compilation coercion with `coerceType` catches remaining cases.
- The `valTypesMatch` function correctly detects both kind and typeIdx mismatches.

### What was tricky
- Must NOT coerce `ref_null -> ref` before null-guards: the null-guard pattern saves the ref to a temp local, checks `ref.is_null`, and only calls `ref.as_non_null` in the non-null branch. Coercing before the guard inserts `ref.as_non_null` unconditionally, causing null-pointer traps.
- Optional chaining paths (`compileOptionalCallExpression`) were left unchanged since they operate inside a `pushBody` context and already handle the receiver correctly.

### Files changed
- `src/codegen/expressions.ts`: Added `coerceArgIfNeeded` helper, coercion at 7 call dispatch sites
