# Finally block compilation: compile once, clone for each path

## Problem

When both catch and finally blocks exist in a try statement, the catch clause
body was compiled from TypeScript source twice (once for `catch $tag` and once
for `catch_all`). While the runtime behavior was correct (each copy lived on a
mutually exclusive control flow path), recompiling from source could cause
duplicate compilation side effects (extra locals, imports) and produced a
larger binary than necessary.

The finally block was already compiled once and cloned (addressed earlier in
commit d8fe05a3). This issue extends the same pattern to the catch body.

## Solution

- Pre-compile the catch clause body once into an instruction array
- When a finally block exists, compile the catch body at +1 break/continue
  depth (since it will be placed inside an inner try wrapper)
- Clone the catch body instructions (via `JSON.parse(JSON.stringify(...))`)
  for the `catch_all` handler
- Use the original catch body instructions for the `catch $tag` handler
- Added comprehensive tests covering all finally execution paths

**Files changed:**

- `tests/finally-block.test.ts` - new test file

**What worked:**
- Compile-once-clone pattern eliminates duplicate compilation side effects
- All existing tests continue to pass (no regressions)
- Break/continue depth adjustment moved to catch body pre-compilation

**What didn't apply:**
- The originally reported symptom (counter incrementing 2-3x at runtime) was
  not reproducible -- the Wasm control flow was already correct since each
  inlined copy lives on a mutually exclusive path
