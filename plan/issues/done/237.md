# Issue #237: WebAssembly type mismatch -- BigInt i64 vs externref

## Status: done

## Summary

170 tests fail at Wasm instantiation time with errors like "call[0] expected type externref, found i64.sub of type i64". These are BigInt-related tests where i64 values are passed to functions expecting externref (e.g., string concatenation, assertion functions).

## Root Cause

BigInt values are compiled as i64 in Wasm. When these values flow into externref-expecting contexts (like `assert.sameValue`, string concatenation with `+`, or any function that takes externref parameters), the Wasm validator rejects the type mismatch. The codegen needs to box i64 values into externref when passing to externref-typed parameters.

## Scope

- `src/codegen/expressions.ts` -- `coerceType` function, BigInt handling
- Tests affected: ~170 compile errors

## Expected Impact

Fixes ~170 Wasm validation errors, enabling BigInt tests to at least run.

## Suggested Approach

1. In `coerceType`, add a path for i64 -> externref:
   - Convert i64 to f64 via `f64.convert_i64_s`
   - Then box via `__box_number` (existing import)
   - This loses BigInt semantics but allows the tests to run
2. Alternatively, add a `__box_bigint` host import that wraps i64 into a JS BigInt externref
3. For string concatenation with BigInt, convert i64 to string via a new `__bigint_to_string` helper

## Acceptance Criteria

- [ ] i64 values can flow into externref-typed parameters without Wasm validation error
- [ ] BigInt assertion tests (`assert.sameValue(0n, 0n)`) pass validation
- [ ] At least 100 compile errors resolved

## Complexity: M

## Implementation Summary

### What was done
Added i64 (BigInt) support to the AnyValue boxing/unboxing paths in `coerceType`:

1. **i64 -> AnyValue boxing**: When an i64 value needs to be boxed into a `ref $AnyValue` (e.g., passing a BigInt to a function expecting `any`), the value is first converted to f64 via `f64.convert_i64_s`, then boxed using the existing `__any_box_f64` helper. This loses BigInt semantics but prevents Wasm validation errors.

2. **AnyValue -> i64 unboxing**: When an AnyValue needs to be converted back to i64, it is unboxed as f64 via `__any_unbox_f64`, then truncated to i64 via `i64.trunc_sat_f64_s`.

Note: The i64 -> externref path (via `__box_number`) already existed in coerceType. The missing paths were specifically for the AnyValue struct type used for `any`-typed parameters.

### What worked
- The fix is minimal and leverages existing helpers (`__any_box_f64`, `__any_unbox_f64`)
- No new host imports needed

### What didn't work / limitations
- BigInt precision is lost when values exceed f64 safe integer range (2^53), since we convert through f64. This is an acceptable tradeoff per the issue description.

### Files changed
- `src/codegen/expressions.ts` -- added i64 boxing/unboxing in `coerceType` AnyValue paths

### Tests
- `tests/issue-237.test.ts` -- 9 tests covering BigInt literals, arithmetic, comparisons, negation, bitwise ops, equality, and i64-to-AnyValue boxing
