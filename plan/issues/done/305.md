# Issue #305: Runtime failures -- computed property names and types/reference

## Status: done

## Summary
Runtime failures when accessing properties on array elements (e.g., `arr[0].value`). The bounds-checked array get produced a Wasm type validation error: "expected (ref N), got (ref null N)".

## Category
Sprint 5 / Group B

## Complexity: XS

## Scope
- Fix Wasm type validation error in bounds-checked array element access
- Add tests for computed property names and types/reference patterns

## Acceptance criteria
- Both runtime failures resolved

## Implementation Summary

### What was done
Fixed a Wasm type validation error in `emitBoundsCheckedArrayGet` where accessing a property on an array element (e.g., `arr[0].value`) would fail with "type error in fallthru[0] (expected (ref N), got (ref null N))".

### Root cause
The bounds-checked `if` block used the element type directly as its block type. When the element is a non-null `ref` (e.g., a struct), the `else` branch (out-of-bounds default) emits `ref.null` which produces `ref_null` -- incompatible with the `ref` block type.

### Fix
In `emitBoundsCheckedArrayGet`:
1. When element type is `ref`, use `ref_null` as the `if` block type so both branches validate
2. After the `if`, emit `ref.as_non_null` to narrow back to `ref` for downstream operations
