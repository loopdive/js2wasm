# Issue #664: Use source maps for runtime error line numbers

Runtime errors (null pointer, illegal cast, OOB) show Wasm function name + byte offset
but no TS line number. Enable source maps in the test262 runner and extract useful
context from Wasm trap errors.

## Implementation Summary

### What was done
- Added `extractWasmFuncName()` to parse V8 Wasm error stacks and extract function names
- Added `lookupSourceMapOffset()` to decode VLQ source map mappings and find the original
  source line for a given Wasm byte offset
- Added `enrichErrorMessage()` that combines function name + source-mapped line into error strings
- Modified `runTest262File()` to compile with `sourceMap: true` and use `enrichErrorMessage`
  for all runtime error results (compile errors, traps, assertion failures)
- Added comprehensive test suite with unit tests and end-to-end Wasm trap test

### Files changed
- `tests/test262-runner.ts` — added helper functions, enabled source maps, enriched error messages
- `tests/wasm-error-enrichment.test.ts` — new test file (11 tests)

### What worked
- VLQ decoder is self-contained (no dependency on the emit/sourcemap module)
- V8 error stack format is consistent and easy to parse
- Source map lookup is fast (linear scan of sorted entries, early exit)

### What didn't work
- Source map byte offset extraction depends on V8's specific stack format; other engines
  may have different formats (not a problem for Node.js-based test runner)
