# Issue #657: Yield outside generator residual (411 CE)

## Problem

411 test262 tests fail with `compile_error: yield expression outside of generator function`. The root cause is the IIFE (Immediately Invoked Function Expression) inlining optimization in `compileCallExpression`. When a generator function expression is used as an IIFE -- e.g. `(function* () { yield 1; yield 2; })()` -- the compiler inlines the body directly into the enclosing (non-generator) function context, where `fctx.isGenerator` is false. The `yield` expressions then trigger the "yield expression outside of generator function" error.

Breakdown by test category:
- 366 for-await-of tests (all use generator IIFE pattern)
- 24 Iterator/prototype tests
- 6 async-generator tests
- 6 for-of tests
- 4 yield-related tests
- 5 other

## Implementation Summary

### What was done
Added guards in both IIFE inlining code paths to skip inlining when the callee is a generator function expression (`function*`). When a generator IIFE is detected, the code falls through to the normal closure compilation path which correctly sets `isGenerator` on the inner FunctionContext.

### Files changed
- `src/codegen/expressions.ts` -- Two IIFE inlining sites modified:
  1. Inline IIFE handler in `compileCallExpression` (~line 12102): Added `isGeneratorIIFE` check to skip inlining
  2. `compileIIFE` function (~line 13242): Added early return for generator function expressions

### Tests
- Added `tests/generator-iife.test.ts` with 3 test cases covering generator IIFE patterns

### What worked
Simple guard checks at the two IIFE inlining sites. The existing closure compilation path already handles generator function expressions correctly (sets `isGenerator`, creates `__gen_buffer`, etc.).

### What didn't apply
The initial hypothesis of arrow functions inheriting generator context or class method issues was incorrect. The actual root cause was IIFE inlining bypassing the generator function context entirely.
