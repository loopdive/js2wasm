# Issue 45: Error Reporting with Source Locations

## Status: open

## Summary
Include source file, line, and column in all compiler error messages. Currently errors say what went wrong but not where.

## Motivation
Error messages like "Unsupported expression kind: 123" with no source location make debugging compiler errors extremely difficult for any non-trivial program.

## Design

### Error Format
```
input.ts:42:10 - error: Unsupported binary operator
```

### Implementation
Add a helper to the codegen context:
```typescript
function reportError(ctx: CodegenContext, node: ts.Node, message: string): void {
  const sf = node.getSourceFile();
  const { line, character } = sf.getLineAndCharacterOfPosition(node.getStart());
  ctx.errors.push({ message, file: sf.fileName, line: line + 1, column: character + 1, severity: "error" });
}
```

Replace `throw new Error(...)` calls in codegen with `reportError`, allowing multiple error collection.

### CompileError Type
```typescript
interface CompileError {
  message: string;
  file?: string;
  line?: number;
  column?: number;
  severity: "error" | "warning";
}
```

## Scope
- `src/codegen/index.ts` — add `reportError` helper, error collection (~20 lines)
- `src/codegen/expressions.ts` — replace throws with `reportError` (~40 lines)
- `src/codegen/statements.ts` — replace throws with `reportError` (~20 lines)
- `src/compiler.ts` — propagate errors to `CompileResult` (~10 lines)
- `tests/error-reporting.test.ts` (~40 lines)

## Complexity: M

## Acceptance criteria
- Errors include file name, line number, and column number
- Multiple errors can be collected
- Error format: `file:line:col - error: message`
- All existing tests still pass
