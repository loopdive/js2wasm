# Issue #281: Object literal property patterns -- shorthand, spread, methods

## Status: review

## Summary
~303 tests fail in language/expressions/object with compile errors. Many involve shorthand properties (`{x, y}`), spread in objects (`{...a}`), method definitions with computed names, and getter/setter pairs that the codegen does not fully handle.

## Category
Sprint 4 / Group C

## Complexity: M

## Scope
- Support shorthand property definitions in object literals
- Handle spread elements in object literals
- Support method definitions with various parameter patterns
- Update object literal compilation in `src/codegen/expressions.ts`

## Acceptance criteria
- Shorthand properties compile
- Object spread compiles
- Method definitions in objects with complex params compile
- At least 40 compile errors resolved

## Implementation notes

### Changes made in `src/codegen/expressions.ts`

1. **Method declarations with non-identifier names**: Extended method declaration
   handling to support string literal names (`{"getVal"() {...}}`), numeric literal
   names, and computed property names (`{[key]() {...}}`) by using `resolveAccessorPropName`
   instead of requiring `ts.isIdentifier(prop.name)`.

2. **Spread property ordering (JS semantics)**: Rewrote the field value emission loop
   to respect source order of properties. Previously, explicit property assignments
   always took precedence over spread sources regardless of order. Now, the code
   iterates through properties in source order and tracks the LAST assignment to each
   field, so `{ x: 1, ...base }` correctly uses `base.x` when base has an `x` property.

### Tests added (6 new equivalence tests)
- Object literal shorthand properties (`{x, y}`)
- Mixed shorthand and regular properties (`{a, b: 10}`)
- Object literal spread (`{...base, z: 3}`)
- Method with string literal name (`{"getVal"() {...}}`)
- Method with computed name (`{[key]() {...}}`)
- Spread override ordering (`{x: 1, ...base}` uses `base.x`)
