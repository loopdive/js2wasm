# Issue 97: NaN/undefined/null truthiness in boolean contexts

## Summary

Wasm boolean coercion uses `f64.ne(0)` which treats `NaN` as truthy, but in JavaScript `NaN` is falsy. This causes infinite loops when `NaN` (or `undefined`/`null`) is used as a loop condition.

## Example

```ts
for (var i = 0; NaN;) { count++; }
// JS: loop never executes (NaN is falsy)
// Wasm: infinite loop (NaN != 0 is true in IEEE 754)
```

## Root Cause

The compiler converts boolean conditions via `f64.ne(0)` (or similar). For IEEE 754 floats, `NaN != 0` evaluates to `true`, but JavaScript's ToBoolean(NaN) returns `false`.

## Fix

In boolean coercion contexts (if/while/for conditions, logical operators), emit:
```wasm
;; truthy check for f64 that matches JS semantics
local.get $val
local.get $val
f64.eq          ;; NaN != NaN → 0, otherwise 1
local.get $val
f64.const 0
f64.ne          ;; 0 → 0, nonzero → 1
i32.and         ;; both must be true: not-NaN AND not-zero
```

This is equivalent to: `val === val && val !== 0` — which matches JS truthiness for numbers.

## Affected test262 tests

- `language/statements/for/12.6.3_2-3-a-ii-14.js` (NaN condition)
- Any test using NaN/undefined/null as loop conditions

## Complexity

S — single change in boolean coercion codegen

## Dependencies

None
