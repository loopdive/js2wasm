# Issue #286: Logical assignment compile errors -- nullish and short-circuit

## Status: review

## Summary
~34 tests fail in language/expressions/logical-assignment with compile errors. These involve `&&=`, `||=`, and `??=` operators with property access targets, element access targets, or computed property names that the codegen does not support as assignment targets.

## Category
Sprint 4 / Group A

## Complexity: S

## Scope
- Support logical assignment on property access targets (`obj.x ??= default`)
- Support logical assignment on element access targets (`arr[i] ||= default`)
- Ensure short-circuit semantics (do not evaluate RHS if condition met)
- Update logical assignment in `src/codegen/expressions.ts`

## Acceptance criteria
- Logical assignment on property/element access compiles
- Short-circuit semantics preserved
- At least 15 compile errors resolved

## Implementation Notes

### Changes made

1. **Extended `compileLogicalAssignment`** to dispatch to property access and element access handlers before falling through to identifier-only logic.

2. **Added `compilePropertyLogicalAssignment`** — handles `obj.prop &&= val`, `obj.prop ||= val`, `obj.prop ??= val`. Resolves the struct type, compiles the object expression once (cached in a local), then delegates to the shared `emitLogicalAssignmentPattern` helper.

3. **Added `compileElementLogicalAssignment`** — handles `obj["prop"] &&= val`, `obj["prop"] ||= val`, `obj["prop"] ??= val` on struct bracket notation and vec struct array elements. Same pattern as property access but resolves field names from string/numeric literals.

4. **Added `emitLogicalAssignmentPattern`** — shared helper that takes `emitGet`/`emitSet` closures and emits the correct short-circuit `if/else` Wasm block for each operator type (`??=`, `||=`, `&&=`).

5. **Added `emitCompoundArithmeticOp`** — shared helper that emits the arithmetic/bitwise operation for compound assignment, deduplicating the switch statements that were repeated in `compileCompoundAssignment`.

6. **Added `compilePropertyCompoundAssignment`** and `compileElementCompoundAssignment`** — handle `obj.prop += val` and `obj["prop"] += val` / `arr[i] += val` patterns. These were referenced but missing from main after a merge conflict.

### Key decisions
- `??=` on f64 struct fields is not supported (f64 can never be null in Wasm). This correctly matches the semantic: ??= only makes sense for nullable types.
- Short-circuit semantics are preserved using the `savedBody`/`fctx.body = []` pattern already used by the identifier logical assignment code.
- The `emitGet`/`emitSet` closure pattern keeps each logical assignment operator implementation DRY across property access, element access, and vec struct targets.

### Files changed
- `src/codegen/expressions.ts` — added 6 new functions, extended `compileLogicalAssignment` dispatch
- `tests/issue286.test.ts` — 15 tests covering all operators on both access patterns plus short-circuit semantics
