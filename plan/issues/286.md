# Issue #286: Logical assignment compile errors -- nullish and short-circuit

## Status: review

## Summary
~34 tests fail in language/expressions/logical-assignment with compile errors. These involve `&&=`, `||=`, and `??=` operators with property access targets, element access targets, or computed property names that the codegen does not support as assignment targets.

## Category
Sprint 4 / Group A

## Complexity: S

## Scope
- Support logical assignment on property access targets (`obj.x ??= default`)
- Support logical assignment on element access targets (`arr[i] ||= default`)
- Ensure short-circuit semantics (do not evaluate RHS if condition met)
- Update logical assignment in `src/codegen/expressions.ts`

## Acceptance criteria
- Logical assignment on property/element access compiles
- Short-circuit semantics preserved
- At least 15 compile errors resolved

## Implementation notes
- Refactored `compileLogicalAssignment` to extract shared core logic into `emitLogicalAssignmentCore`
- Added `compileLogicalAssignmentPropertyAccess` for `obj.prop ??= value` patterns (struct field get/set)
- Added `compileLogicalAssignmentElementAccess` for `arr[i] ||= value` patterns (array/vec get/set, struct string-key access)
- Fixed `??=` on f64/i32 variables: these are never null, so `??=` is a no-op (was incorrectly emitting `ref.is_null` on f64)
- Property access: compiles object once into temp local, uses struct.get/struct.set
- Element access: supports vec structs ({length, data}), plain arrays, and struct string-literal index
- Static property access: handled via global.get/global.set
- All emitSet closures store new value in temp local, do struct/array.set, then push value back on stack
- 9 new equivalence tests added covering all operator/target combinations
