# Developer slot — issues #1174 + #1175 (sprint 45)

**Status**: ✅ DONE — PR #58 merged 2026-04-27.

## What landed

Single PR #58 fixing two `--target wasi` codegen regressions surfaced by the
#1125 competitive benchmark verification.

### #1174 — `string_constants::a,b,c` leaked under WASI

`object-ops` benchmark could not instantiate under wasmtime:
> unknown import: `string_constants::a,b,c` has not been defined

Root cause: two unrelated codegen paths registered host imports even in
nativeStrings mode.

1. **`emitStructFieldNamesExport`** (`src/codegen/index.ts`) emitted a
   `__struct_field_names(externref) -> externref` export that did
   `global.get` of a comma-joined CSV. Pure JS-host introspection
   machinery — dead code under WASI. **Now skipped entirely in
   nativeStrings mode.**

2. **`typeErrorThrowInstrs`** (`src/codegen/property-access.ts`) emitted
   `global.get strIdx; throw __exn` for null-check throws. Now uses a
   new `stringConstantExternrefInstrs` helper in `native-strings.ts`
   that materializes the message inline as a FlatString struct +
   `extern.convert_any` in nativeStrings mode.

Supporting fix: `addStringConstantGlobal`
(`src/codegen/registry/imports.ts`) now short-circuits in nativeStrings
mode — registers the value with sentinel `-1` (matching
`collectStringLiterals`'s finalize convention) and skips the host
import + global-index shift cascade.

### #1175 — string `+=` failed wasm validation under WASI

`text += alphabet.charAt(a)` produced
`call $__str_flatten(i32)` and `call $concat(i32 length-result, externref)`
— both rejected by wasm-validator.

**Root cause** (not what the issue spec hypothesized): function-index
**aliasing**, not coercion-path misuse:

1. `compileStringCompoundAssignment` had only the legacy host-import
   branch and called `addStringImports(ctx)` LATE during codegen,
   appending 5 `wasm:js-string` imports.
2. `addImport` increments `numImportFuncs` but does **not** shift any
   already-emitted `call funcIdx=N` instructions.
3. Indices that pointed at native helpers (`__str_flatten` at
   `funcIdx=1`) silently re-targeted to the new host imports
   (`length` at `funcIdx=1`), and other call sites shifted
   correspondingly.

**Fix**: added `compileNativeStringCompoundAssignment` branch that uses
`__str_concat` (`ref $AnyString → ref $AnyString`, no externref
roundtrip), never calls `addStringImports` late, and coerces RHS
through ref-typed paths. Returns `ref $AnyString` so downstream type
inference handles the result correctly.

## Open follow-ups (not in scope of this slot)

These surfaced during the WASI sanity smoke but are pre-existing bugs
in the nativeStrings path, not regressions from #1174/#1175:

1. **Class with property access under `--target wasi`** — `__str_copy_tree`
   emits a `local.get` of type `i32` where externref is expected. Likely
   another instance of the function-index aliasing pattern fixed for
   string `+=`. Worth searching the codebase for other
   `addStringImports(ctx)` calls on hot paths in nativeStrings mode.

2. **Array methods (`.reduce`) under `--target wasi`** — `__str_copy_tree`
   `call[0]` expects `i32` but gets `(ref null 2)`. Probably the same
   pattern.

Both surface only in nativeStrings mode (auto-on for WASI). The legacy
JS-host-mode equivalence suite is unaffected.
