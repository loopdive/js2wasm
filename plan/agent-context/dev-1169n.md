## Key finding: IR-fallback false-positive `compile_error` classification

**TL;DR**: When the IR phase selector accepts a function shape but the lowerer (or the override-build) throws a clean fallback, the diagnostic was being emitted with severity `"error"`. The test262 runner classifies a test as `compile_error` whenever `result.errors.some((e) => e.severity === "error")` — even when `result.success === true` and the Wasm body is identical to a non-experimentalIR build. This produced 30-40 false-positive "regressions" per PR that widens the IR selector.

**Root locations** (both in `src/codegen/index.ts`):

1. Post-`compileIrPathFunctions` site (originally line 786-788): propagates `report.errors` from the IR lowering pipeline to `ctx.errors` via `reportErrorNoNode`. Fired for every function that the lowerer threw clean fallback on.

2. Override-build catch block (originally line 771-778): `resolvePositionType` throws for class-typed params that don't resolve against `classShapes`, etc. Fired for selector-claimed harness functions whose types couldn't be mapped.

Both now emit severity `"warning"` (PRs #127 follow-up commits + PR #132 commits `dc687bcaf` and `1ac5c837c`).

**Why this matters going forward**: every future slice that widens the selector will trigger this pattern unless the fix is in place. The two fix sites are the only IR-related error paths in `generateModule` — confirmed via `grep -rn "reportErrorNoNode\|reportError" src/ir/ src/codegen/index.ts` (other matches are unrelated codegen errors with severity "error", appropriately).

**Long-term cleanup** (recommended): introduce a separate `IrPathReport` channel on `CompileResult` distinct from `errors`. Bridge tests (`#1181`) currently filter `r.errors` by message-prefix; that stops working cleanly once the channels split, so plan for a small bridge-test refactor at the same time.
