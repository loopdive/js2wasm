### Implementation approach

1. Parse source via `ts.createSourceFile("<eval>.ts", src, Latest, setParentNodes=true, JS)`.
2. Bail if `(sf as any).parseDiagnostics` has entries (malformed source → let host handle it).
3. Run `allNodesInlineSupported` pre-scan.
4. Hoist with the existing `hoistVarDeclarations`, `hoistLetConstWithTdz`, `hoistFunctionDeclarations` — these accept `ts.NodeArray<ts.Statement> | ts.Statement[]` and work on foreign nodes because `resolveWasmType` falls back to externref when the checker returns error-type.
5. Compile each statement with existing `compileStatement`. Last ExpressionStatement's value (coerced to externref) becomes the call result; otherwise push undefined via `emitUndefined`.
6. Evaluate any extra args for side effects and drop.

### Why the IIFE case is bailed out

`eval("(function(){ return 42; })()")` is listed in the spec's examples but NOT in the strict acceptance criteria. Compiling a FunctionExpression from a foreign AST requires the checker to resolve its callable type — that fails silently (binary compiles but the IIFE is emitted as an unreachable stub). The pre-scan rejects it so it correctly falls through to the dynamic `__extern_eval` path, where it still passes.

## Notable lessons / gotchas

1. **`snapshot_delta` is NOT a merge gate** — `.github/workflows/ci-status-feed.yml` L90-93 explicitly says so. It's informational; the real gate is `net_per_test`. When both `net_per_test` and `regressions` are null (diff job didn't produce the regression txt), the precedent from PR #7 (Array.from poisoning) is to merge on `conclusion: success` + positive snapshot_delta. I initially over-indexed on comparing my PR's snapshot_delta (+491) vs the previous PR's (+578) as a "-87 regression" signal — that's exactly the kind of noisy comparison the workflow comment warns against.

2. **`ts.createSourceFile` with `setParentNodes=true` is the right knob** — without parent pointers, upward AST walks inside codegen break silently.

3. **`parseDiagnostics` is an internal property** — access via `(sf as unknown as { parseDiagnostics?: readonly ts.Diagnostic[] }).parseDiagnostics`.

4. **Foreign-AST codegen is a minefield**. Anything that queries `ctx.checker.getTypeAtLocation(node)` on a node from `ts.createSourceFile` returns error-type, which downstream usually maps to externref silently. For simple expressions (arithmetic, string literals, `new TypeError(...)`, identifiers) this is fine. For function/class expressions it produces unreachable code with no diagnostic. The conservative pre-scan is the cheapest way to avoid silent mis-compiles.
