## Key root-cause finding

js2wasm itself **never emits the exact-ref opcode directly** — there is no `exact` byte in `src/emit/binary.ts`'s type encoder. The exact-ref encoding is **injected by Binaryen ≥ 124** when `wasm-opt --all-features` enables the WasmGC custom-descriptors proposal:

- Binaryen rewrites `(ref $T)` → `(ref exact $T)` for any leaf or single-instance type
- Triggered by both explicit `(sub final $T)` declarations **and** by plain `(struct …)` / `(array …)` types with no `sub` clause at all
- wasmtime ≤ 44 rejects with "custom descriptors required for exact reference types"

The issue's hypothesis ("js2wasm appears to be emitting `(ref exact $T)` somewhere in the array codegen") was wrong about the source — but the recommended fix ("stop emitting exact, falling back to plain `(ref $T)`") was right about the *direction*. The real fix had to live in two places because the compiler-side change can only address half the surface (the explicit-final half).

Verified empirically with `~/.wasmtime/bin/wasmtime` (v44.0.0, installed during the session via `https://wasmtime.dev/install.sh`):
- raw js2wasm binary loads fine in wasmtime (only fails on missing imports)
- `wasm-opt --all-features -O4` output **fails** with the exact-ref error
- `wasm-opt --all-features --disable-custom-descriptors -O4` output **loads** in wasmtime

`wasm-dis` displays `(exact $T)` for refs to final types even when the binary doesn't carry the explicit exact byte — this is a Binaryen display inference, not a binary content check. So tests should grep the **WAT for `sub final`** (or for the absence of), not count raw 0x4F bytes (0x4F is overloaded with the `i32.ge_u` instruction opcode).
