### Task #4 / #16 — #1225: Nested destructuring null/undefined missing TypeError (PR #130, merged)

Three code paths fixed for the spec-required TypeError on nested
destructuring of null/undefined (ECMA-262 §13.15.5.2 / §13.15.5.5 /
§8.4.2 GetIterator + RequireObjectCoercible):

1. **`emitArrayDestructureFromLocal`** (`src/codegen/expressions/assignment.ts`)
   — `[[ _ ]] = [null]`. TS infers `[null]` as a tuple struct (`__tuple_0`),
   not a vec. Function silently returned with no destructuring. Added
   tuple-struct detection (`every f.name === \`_${idx}\``), `widenLocalToNullable`
   on the source local so `ref.is_null` is valid even when `srcType.kind === "ref"`,
   and per-field `struct.get` for tuple receivers.

2. **`compileForOfAssignDestructuring`** (`src/codegen/statements/loops.ts`)
   — `for ([{x}] of [[null]])`. Branch handling externref nested elements
   now emits `emitExternrefDestructureGuard` before processing the nested
   pattern when non-empty. Both object and array literal patterns covered.

3. **`destructureParamObjectExternref`** (`src/codegen/destructuring-params.ts`)
   — `*method({ w: { x, y, z } = ... } = {})`. Nested-binding branch now
   applies the initializer (only on undefined per spec) and emits the
   externref destructure guard before recursing. Was previously silently
   producing undefined for inner identifiers.

   tuple-struct fields that get loaded into locals — the compiler stores
   field types as `ref T` even when the actual value can be `ref.null T`,
   and `ref.is_null` requires a nullable local declared type.
