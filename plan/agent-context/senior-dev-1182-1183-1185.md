## Side findings filed as new issues

### #1186 — fix(legacy): re-resolve native-string helpers post-shift

Pre-existing legacy bug: `compile(source, { nativeStrings: true })` produces
invalid Wasm for `for (const c of s)`. Stale `__str_charAt` funcIdx after
late-import shifts. Reproducible on `main` without any IR changes. The IR
sidesteps this by re-resolving names via `ctx.mod.functions[i].name`; the
legacy `compileForOfString` should adopt the same pattern. Easy fix.

### #1187 — test-runtime: JS-string ↔ native-string coercion helper

In `nativeStrings: true` mode, JS callers can't pass a JS string into a
Wasm export with a `string` param (`(ref $AnyString)` doesn't auto-coerce
from `externref`). Blocks proper dual-run testing for any IR feature that
touches string-typed signatures. Once #1186 + #1187 land, the dual-run
cases I had to disable in `tests/issue-1183.test.ts` (5 native-mode cases
asserting against JS-computed values instead of legacy parity) can be
re-enabled.

## Architecture state at shutdown

Per the post-#1185 architecture (assuming PR #72 lands), the IR Phase 4
build path is now:

```
Phase 1 prep:   makeFromAstResolver(ctx) — eager, stable methods only
Phase 1:        AST→IR build, resolver threaded into LowerCtx
Phase 2a:       per-function hygiene (CF → DCE → simplifyCFG)
Phase 2b:       module-scope inlineSmall
Phase 2c:       monomorphize
Phase 2d:       taggedUnions (identity in V1)
Phase 3 prep:   makeResolver(ctx, ...) — full resolver with deferred registries
Phase 3:        IR→Wasm lowering
```

The `IrFromAstResolver` and full `IrLowerResolver` are structurally
related — the latter satisfies the former — but they're constructed at
different times because the deferred registries (object / closure /
class) need Phase-3 state.
