## Key findings / discoveries

1. **The CLI's `-o` flag is an output directory, not a filename.** The issue spec showed `-o hello-fs.wasm` but that breaks (treats it as a directory and writes inside). README documents the correct `-o <dir>` form.

2. **The compiler's WASI scaffolding for `writeFileSync` is solid and complete:**
   - `src/compiler.ts:detectNodeFsImports` scans the original source for named `node:fs`/`fs` imports BEFORE `preprocessImports` strips them, so the codegen knows what to wire up.
   - `src/codegen/index.ts:registerWasiImports` only adds `path_open`/`fd_close` imports when `writeFileSync` is in `wasiNodeFsFuncs` — `console.log`-only programs stay minimal.
   - `src/codegen/index.ts:emitWasiWriteFileSyncHelper` generates `__wasi_write_file_sync(pathPtr, pathLen, dataPtr, dataLen)` directly in Wasm: hardcoded `dirfd=3` (first preopen), `oflags=O_CREAT|O_TRUNC=9`, `rights_base=FD_WRITE=64`. iovec at `memory[0..7]`, nwritten at `memory[8]`, opened-fd output at `memory[12]`.
   - `src/codegen/expressions/calls.ts` rewrites `writeFileSync(path, data)` calls in WASI mode: string-literal args go through `wasiAllocStringData` (compile-time data segment, deduped by offset), `const x = '...'` references via `compileWasiStringArgToLinearMemory`. Fully dynamic strings still emit `unreachable` (TODO: runtime GC-string → linear-memory encoder).

3. **Local `wasmtime`/`wasmer` binaries are not installed in the dev container.** All end-to-end verification was done via Node's built-in `node:wasi`. The test in `tests/issue-1035.test.ts` uses `node:wasi` with a `/tmp/wasi-test-1035` preopen.
