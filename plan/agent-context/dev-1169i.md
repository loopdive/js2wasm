## What shipped in #1169i

Step A of the slice-10 staging plan: **RegExp through IR**.

- `RegularExpressionLiteral` (`/foo/g`) → `extern.regex` IR instr → host `RegExp_new(pattern, flags)`.
- `new RegExp(...)` (1 or 2 args) → `extern.new` IR instr → `RegExp_new`.
- `r.test(s)`, `r.exec(s)` → `extern.call` IR instr → `RegExp_test` / `RegExp_exec`.

Plus the **complete IR scaffolding** so steps B–E are pure pattern repetition.

## Key concepts the next agent needs

### `IrType.extern { className }`

Opaque externref carrying the class identity statically. Used as the
SSA result type for `extern.new` / `extern.regex` / extern method
calls returning ref-shaped values. Never actually exists at the Wasm
level — `lowerIrTypeToValType` maps it to `externref`.

```ts
| { readonly kind: "extern"; readonly className: string }
```

The className is the static dispatch key — when a receiver of
`IrType.extern { className: "RegExp" }` does `.test(s)`, the lowerer
resolves the method by looking up `getExternClassInfo("RegExp")`'s
`methods.get("test")` for the param + result types, then emits
`call $RegExp_test`.

### Five new IR instr kinds (all in `src/ir/nodes.ts`)

| Instr | Lowering | Result IrType |
|---|---|---|
| `extern.new` | `args; call $<class>_new` | `IrType.extern { className }` |
| `extern.call` | `recv; args; call $<class>_<method>` | `irVal(method.results[0])` or null |
| `extern.prop` | `recv; call $<class>_get_<prop>` | `irVal(prop.type)` |
| `extern.propSet` | `recv; value; call $<class>_set_<prop>` | void |
| `extern.regex` | `<pattern globals>; <flags globals>; call $RegExp_new` | `IrType.extern { className: "RegExp" }` |

Note the legacy underscore convention: `_get_<prop>` / `_set_<prop>`
(NOT `<prop>_get` like the implementation plan says — the plan was
inaccurate; verify against `src/codegen/index.ts:6087-6111`).

### `IrFromAstResolver.getExternClassInfo`

Wired in `makeFromAstResolver` (in `src/ir/integration.ts`) — pure
pass-through over `ctx.externClasses`. Returns `IrExternClassMeta`:

```ts
{
  className: string;
  constructorParams: readonly ValType[];
  methods: ReadonlyMap<string, { params: readonly ValType[]; results: readonly ValType[] }>;
  properties: ReadonlyMap<string, { type: ValType; readonly: boolean }>;
}
```

**Critical**: methods carry the LEGACY signature shape — `params[0]`
is the receiver `externref`. The from-ast layer slices off
`params[0]` when matching call args. See `lowerMethodCall` in
`src/ir/from-ast.ts`.

### `KNOWN_EXTERN_CLASSES` (in `src/ir/select.ts`)

28-class allow-list mirroring `registerBuiltinExternClasses`. The
selector only claims `new <Class>(...)` shapes when the class is in
this set. Already includes everything needed for steps B–E:
`Uint8Array`, `Int8Array`, `Uint16Array`, etc., `ArrayBuffer`,
`DataView`, `Map`, `Set`, `Date`, `Error`, `Promise`, …

### Argument padding

`emitDefaultExternArg(cx, expectedTy)` in `from-ast.ts` pads missing
optional args:
- externref → `ref.null.extern` (host shim treats as undefined)
- f64 → `f64.const 0`
- i32 → `i32.const 0`
- i64 → `i64.const 0n`

Mirrors `pushDefaultValue` in `src/codegen/type-coercion.ts`. Critical
for cases like `new RegExp(p)` where the legacy host import
`RegExp_new` takes 2 externref params — passing 1 arg without padding
produces `WebAssembly.instantiate(): not enough arguments on the stack`.

### `coerceToExpectedExtern(value, expected, cx, where)`

Checks/coerces an SSA value to match a host import's expected ValType.
For step A, only handles same-kind val matches and the `string →
externref` / `extern → externref` no-op cases. **Will need
extending** for steps B–E that pass non-string ref types as args
(e.g. `new Uint8Array(buf)` where `buf: ArrayBuffer` — externref).

## Known limitations / things to extend in steps B–E

### 1. ElementAccess on extern receivers (NOT implemented)

`arr[i]` and `arr[i] = v` for TypedArrays — step B needs this.
Currently `lowerElementAccess` (in `from-ast.ts`) only handles
string-literal keys on `IrType.object` receivers. Add an arm for
`IrType.extern` receivers that emits an `extern.call` to the
synthetic `<className>_at` getter (or whatever the legacy uses —
check `src/codegen/property-access.ts:941+`).

`isPhase1Expr`'s ElementAccess arm also currently rejects
non-string-literal keys — needs widening for numeric-keyed access on
extern receivers.

### 2. Static method calls (NOT implemented)

`Date.now()`, `Promise.resolve(x)`, `Promise.reject(x)`, `Math.foo`,
`Array.isArray` — all currently fall back to legacy because the
`isPhase1Expr` call-expression arm only handles
`PropertyAccessExpression(callee, args)` where `callee` is itself a
Phase-1 expression. A bare `Date` identifier isn't in scope (it's
a host global), so `Date.now` rejects.

Fix path: special-case `<KnownExternClass>.<method>(...)` in the
selector and add a corresponding `extern.staticCall` IR instr in
the lowerer.

### 3. void method calls as statements (NOT implemented)

Calls like `arr.set(other)` where `set` is void — currently the IR
layer throws "void method ... used in expression position" because
`lowerCall` always expects a return value. Fix by routing
expression-statement calls through a separate `lowerStatementCall`
that allows void results.

### 4. tryFoldNullCompare bug fix

While testing step A, I found that `tryFoldNullCompare` was
folding `result !== null` for ANY non-`boxed` operand to a constant
— including externref-shaped values, where the runtime value CAN
be null. Fixed by bailing for `IrType.extern` and `IrType.val`
where the val's kind is `externref` or `ref_null`.

If steps B–E surface other nullable types (e.g. `Map.get(k)` returns
`V | undefined` where `undefined` lowers to ref.null.extern), watch
for the same fold issue and extend the bail list.

### 5. Native-strings mode coercion gap

`coerceToExpectedExtern` currently treats `IrType.string` as a no-op
for externref params, BUT only when `cx.resolver?.nativeStrings?.()`
returns false. In native-strings mode, the value is a `(ref
$AnyString)` struct ref and needs `extern.convert_any` to become
externref. Currently we throw and fall back to legacy — works fine
for step A (RegExp tests in JS-host mode) but will need fixing for
steps B–E that someone runs with `--nativeStrings`.

## How to validate steps B–E quickly

1. Pick a class family (e.g. Uint8Array).
2. Add a single equivalence test that uses the simplest construction
   + access pattern.
3. Run `npm test -- tests/equivalence/ir-slice10-<class>.test.ts`
   to confirm IR and legacy both pass.
4. Run `npm test -- tests/equivalence/regexp-methods.test.ts
   tests/equivalence/ir-slice4-classes.test.ts` to confirm step A
   + slice 4 didn't regress.
5. Full `tests/equivalence/` regression check before pushing.

The pattern from step A: typically <100 LOC + a 5-test equivalence
file per class family. The IR scaffolding is already in place.
