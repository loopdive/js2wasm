A sprint should span one week of token budget. This means:

**Why:** Sprint planning should match the token-budget cadence (weekly), not be prematurely closed when issues run out. Idle capacity = wasted budget.

**How to apply:** When sprint N's planned issues are nearly done, immediately pull sprint N+1 work into the active sprint (move files + frontmatter) rather than letting devs go idle. Keep the team at full load for the full week. "We're at X% of the week's budget" is the right metric for sprint progress, not issue count.
